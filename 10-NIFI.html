🎯 מבוא ל-Apache NiFi

Apache NiFi

Apache NiFi הוא מערכת קלה לשימוש, חזקה ואמינה לעיבוד והפצת נתונים.

Apache NiFi תומך בגרפים מוכוונים חזקים וניתנים להרחבה של ניתוב נתונים, טרנספורמציה ולוגיקת תיווך מערכת.

Apache NiFi הוא מערכת dataflow המבוססת על מושגי תכנות מבוסס זרימה (flow-based programming).

❓ מהו Apache NiFi?

הגדרה

Apache NiFi הוא מערכת dataflow המאפשרת:

  • עיבוד והפצת נתונים בצורה אמינה
  • ניתוב נתונים מורכב
  • טרנספורמציה של נתונים
  • תיווך בין מערכות

מושגי יסוד

Flow-Based Programming

NiFi מבוסס על תכנות מבוסס זרימה, שבו:

  • נתונים זורמים דרך רכיבים (processors)
  • כל רכיב מבצע פעולה ספציפית
  • הרכיבים מחוברים ביניהם ליצירת dataflow

✨ תכונות עיקריות של Apache NiFi

תכונות למפתחים

  • Web-based user interface - ממשק משתמש מבוסס web
  • Seamless experience - חוויה חלקה בין עיצוב, בקרה, משוב וניטור
  • Data Provenance - מעקב אחר dataflow מההתחלה ועד הסוף
  • Highly configurable - ניתן להגדרה גבוהה
  • Dynamic prioritization - עדיפות דינמית
  • Flow can be modified at runtime - Flow ניתן לשינוי בזמן ריצה
  • Loss tolerant vs guaranteed delivery - סובלנות לאובדן מול מסירה מובטחת
  • Low latency vs high throughput - latency נמוך מול throughput גבוה
  • Designed for extension - מתוכנן להרחבה
  • Build your own processors and more - בניית processors משלך ועוד
  • Enables rapid development and effective testing - מאפשר פיתוח מהיר ובדיקות יעילות

🔧 מושגים בסיסיים ב-Apache NiFi

הבנת הרכיבים המרכזיים של המערכת

● DFM (Data Flow Manager)

DFM הוא המשתמש/מנהל שיוצר ומנהל את ה-dataflows ב-NiFi.

ה-DFM משתמש בממשק ה-Web כדי ליצור, לערוך ולנהל את ה-processors וה-connections.

● Processor

מהו Processor?

Processor הוא הרכיב ב-NiFi שמשמש ל:

  • האזנה לנתונים נכנסים
  • משיכת נתונים ממקורות חיצוניים
  • פרסום נתונים ליעדים חיצוניים
  • ניתוב, טרנספורמציה או חילוץ מידע מ-FlowFiles

דוגמאות ל-Processors נפוצים:

📂 GetFile

קריאת קבצים מתיקייה

💾 PutFile

כתיבת קבצים לתיקייה

🗄️ ExecuteSQL

ביצוע שאילתות SQL

🔄 UpdateAttribute

עדכון attributes של FlowFile

🔍 LookupRecord

חיפוש נתונים ב-lookup tables

📡 PublishKafka

פרסום ל-Kafka topic

● FlowFile

מהו FlowFile?

FlowFile מייצג נתון יחיד ב-NiFi.

FlowFile מורכב משני רכיבים:

📄 FlowFile Content

הנתונים עצמם שמיוצגים על ידי ה-FlowFile

זה יכול להיות JSON, CSV, XML, טקסט או כל פורמט אחר

🏷️ FlowFile Attributes

מאפיינים שמספקים מידע או הקשר על הנתונים

כמו filename, path, uuid, mime.type ועוד

דוגמה ל-FlowFile Attributes:

filename: data.csv path: /input/ uuid: 12345678-1234-1234-1234-123456789abc mime.type: text/csv file.size: 1024 entryDate: 2024-01-15T10:30:00Z

● Connection

מהו Connection?

חיבור של אחד או יותר processors יחד יוצר Connection.

מאפיינים של Connection:

  • כל connection מורכב מאחד או יותר Relationships
  • ה-DFM יכול לקבוע אילו Relationships ישמשו (למשל: success, failure, וכו')
  • כל connection מכיל FlowFile Queue
  • כאשר FlowFile מועבר ל-Relationship, הוא מתווסף ל-queue של ה-Connection המתאים

Relationships נפוצים:

✅ success

הפעולה הצליחה

❌ failure

הפעולה נכשלה

⏭️ original

הנתון המקורי

📋 no match

אין התאמה

● Queue

מהו Queue?

Queue הוא המקום שבו FlowFiles ממתינים לפני עיבוד על ידי processor הבא.

כל Connection מכיל FlowFile Queue.

תכונות של Queue:

  • ניתן להגדיר backpressure - הגבלת מספר FlowFiles
  • ניתן להגדיר prioritization - עדיפות לעיבוד
  • ניתן לנטר את גודל ה-queue בזמן אמת
  • ניתן לראות את ה-FlowFiles הממתינים

● Process Group

מהו Process Group?

Process Group מספק מנגנון לקבץ רכיבים יחד.

שימושים:

  • שמירה על dataflows שונים ב-process groups שונים
  • Process group יכול להיות מבוסס על פרויקטים שונים
  • ארגון וניהול טוב יותר של flows מורכבים
  • אפשרות ליצור היררכיה של groups

יתרונות:

📁 ארגון

ארגון לוגי של flows

🔒 בידוד

בידוד בין פרויקטים שונים

♻️ שימוש חוזר

אפשרות לשימוש חוזר ב-templates

👥 עבודה משותפת

קל יותר לעבוד בצוות

● Controller Service

מהו Controller Service?

Controller Service הוא שירות משותף שניתן להשתמש בו על ידי מספר processors.

Controller Services מאפשרים שיתוף משאבים ו-configurations בין processors.

דוגמאות ל-Controller Services:

🗄️ DBCPConnectionPool

חיבור משותף למסד נתונים

🔍 LookupService

שירות lookup משותף

🔐 SSLContextService

תצורת SSL משותפת

📊 SchemaRegistry

ניהול schemas משותף

יתרונות:

  • שיתוף משאבים - חיבור אחד למסד נתונים במקום רבים
  • ניהול מרכזי - עדכון configuration במקום אחד
  • ביצועים טובים יותר - שימוש חוזר בחיבורים
  • אבטחה - ניהול credentials במקום אחד

💪 תרגילים מעשיים

למד NiFi דרך תרגילים מעשיים

תרגיל 1 הבנת ממשק המשתמש של NiFi

מטרה:

להכיר את ממשק המשתמש של NiFi ואת הרכיבים הבסיסיים.

שלבים:

  1. התחבר ל-NiFi UI (בדרך כלל http://localhost:8080/nifi)
  2. הכר את ה-Canvas - אזור העבודה הראשי
  3. בדוק את ה-Toolbar - כלים ליצירת processors
  4. הכר את ה-Component Toolbar - רכיבים זמינים
  5. בדוק את ה-Status Bar - מידע על ה-cluster
  6. נסה ליצור Processor חדש על ידי גרירה מה-Toolbar

💡 טיפים:

  • לחץ ימני על Processor לראות את התפריט
  • השתמש ב-zoom in/out לניווט ב-Canvas
  • בדוק את ה-Configuration של כל Processor
תרגיל 2 GetFile → PutFile

מטרה:

יצירת flow בסיסי לקריאת קבצים מתיקייה וכתיבתם לתיקייה אחרת.

רכיבים:

  • Source: GetFile Processor
  • Target: PutFile Processor

שלבים:

  1. צור GetFile Processor
  2. הגדר את ה-Input Directory (תיקיית המקור)
  3. צור PutFile Processor
  4. הגדר את ה-Directory (תיקיית היעד)
  5. חבר את GetFile ל-PutFile באמצעות Connection
  6. הגדר את ה-Relationship ל-"success"
  7. הפעל את שני ה-Processors
  8. שים קובץ בתיקיית המקור וצפה בהעברה
GetFile Configuration: - Input Directory: /input - File Filter: .* - Keep Source File: false PutFile Configuration: - Directory: /output - Conflict Resolution Strategy: replace

✅ בדיקה:

ודא שהקובץ הועתק מתיקיית המקור לתיקיית היעד.

תרגיל 3 File → Database

מטרה:

קריאת נתונים מקובץ והכנסתם למסד נתונים.

רכיבים:

  • Source: GetFile Processor
  • Target: PutDatabaseRecord Processor

שלבים:

  1. צור GetFile Processor לקריאת קובץ CSV
  2. צור PutDatabaseRecord Processor
  3. הגדר Controller Service מסוג DBCPConnectionPool
  4. הגדר את ה-Database Connection URL, Username, Password
  5. הגדר את ה-Table Name
  6. הגדר את ה-Record Reader (CSVReader)
  7. חבר את GetFile ל-PutDatabaseRecord
  8. הפעל את ה-Flow
DBCPConnectionPool Configuration: - Database Connection URL: jdbc:mysql://localhost:3306/mydb - Database User: root - Database Password: password - Database Driver Class Name: com.mysql.jdbc.Driver PutDatabaseRecord Configuration: - Record Reader: CSVReader - Table Name: my_table - Statement Type: INSERT

⚠️ חשוב:

ודא שה-Database Driver מותקן ב-NiFi (lib directory).

תרגיל 4 File → UpdateAttribute → Database

מטרה:

הוספת attributes לנתונים לפני הכנסתם למסד נתונים.

רכיבים:

  • Source: GetFile Processor
  • Transform: UpdateAttribute Processor
  • Target: PutDatabaseRecord Processor

שלבים:

  1. צור GetFile Processor
  2. צור UpdateAttribute Processor
  3. הוסף attributes חדשים:
    • timestamp - תאריך ושעה נוכחיים
    • source_file - שם הקובץ המקור
    • processed_by - שם המשתמש/מערכת
  4. חבר GetFile → UpdateAttribute → PutDatabaseRecord
  5. הגדר את ה-Relationships כראוי
  6. הפעל את ה-Flow
UpdateAttribute Configuration: Add new properties: - timestamp: ${now():format('yyyy-MM-dd HH:mm:ss')} - source_file: ${filename} - processed_by: NiFi

💡 Expression Language:

NiFi תומך ב-Expression Language להגדרת ערכים דינמיים.

דוגמאות: ${filename}, ${now()}, ${UUID()}

תרגיל 5 Database → Add Attributes → Database

מטרה:

קריאת נתונים ממסד נתונים, הוספת attributes, וכתיבתם חזרה למסד נתונים.

רכיבים:

  • Source: ExecuteSQL Processor
  • Transform: UpdateAttribute Processor
  • Target: PutDatabaseRecord Processor

שלבים:

  1. צור ExecuteSQL Processor
  2. הגדר את ה-SQL Query (למשל: SELECT * FROM source_table)
  3. הגדר Controller Service לחיבור למסד הנתונים
  4. צור UpdateAttribute להוספת attributes
  5. צור PutDatabaseRecord לכתיבה ל-target table
  6. חבר את כל ה-Processors
  7. הפעל את ה-Flow
ExecuteSQL Configuration: - SQL select query: SELECT id, name, email FROM users WHERE status = 'active' UpdateAttribute Configuration: - processed_date: ${now():format('yyyy-MM-dd')} - batch_id: ${UUID()} PutDatabaseRecord Configuration: - Table Name: processed_users - Statement Type: INSERT

✅ תוצאה צפויה:

נתונים נקראים מ-source table, מעובדים עם attributes חדשים, ונכתבים ל-target table.

תרגיל 6 Database → Lookup → Database

מטרה:

שימוש ב-Lookup Service להעשרת נתונים לפני כתיבתם למסד נתונים.

רכיבים:

  • Source: ExecuteSQL Processor
  • Lookup: LookupRecord Processor
  • Target: PutDatabaseRecord Processor

שלבים:

  1. צור ExecuteSQL לקריאת נתונים
  2. צור LookupRecord Processor
  3. הגדר Lookup Service (למשל: DatabaseLookupService)
  4. הגדר את ה-Lookup Strategy (Simple Key-Value Lookup)
  5. הגדר את ה-Key Field ו-Result Field
  6. צור PutDatabaseRecord לכתיבה
  7. חבר את כל ה-Processors
  8. הפעל את ה-Flow
LookupRecord Configuration: - Lookup Service: DatabaseLookupService - Lookup Strategy: Simple Key-Value Lookup - Key Field: customer_id - Result Field: customer_name DatabaseLookupService Configuration: - SQL Query: SELECT id, name FROM customers WHERE id = ? - Key Column: id - Value Column: name

💡 Use Case:

Lookup שימושי להעשרת נתונים עם מידע נוסף מטבלאות אחרות, כמו הוספת שם לקוח לפי ID.

תרגיל 7 File → Kafka → File

מטרה:

שילוב NiFi עם Kafka - פרסום נתונים ל-Kafka וצריכתם בחזרה.

רכיבים:

  • Source: GetFile Processor
  • Kafka Producer: PublishKafka Processor
  • Kafka Consumer: ConsumeKafka Processor
  • Target: PutFile Processor

שלבים - Producer Side:

  1. צור GetFile Processor
  2. צור PublishKafka Processor
  3. הגדר את ה-Kafka Brokers (למשל: localhost:9092)
  4. הגדר את ה-Topic Name
  5. חבר GetFile → PublishKafka

שלבים - Consumer Side:

  1. צור ConsumeKafka Processor
  2. הגדר את ה-Kafka Brokers
  3. הגדר את ה-Topic Name (אותו topic)
  4. הגדר את ה-Consumer Group
  5. צור PutFile Processor
  6. חבר ConsumeKafka → PutFile
PublishKafka Configuration: - Kafka Brokers: localhost:9092 - Topic Name: nifi-topic - Delivery Guarantee: best effort ConsumeKafka Configuration: - Kafka Brokers: localhost:9092 - Topic Name: nifi-topic - Consumer Group: nifi-consumer-group - Auto Offset Reset: earliest

⚠️ דרישות מוקדמות:

  • Kafka צריך להיות מותקן ופועל
  • Topic צריך להיות קיים ב-Kafka
  • ודא שה-Kafka libraries מותקנות ב-NiFi

✅ בדיקה:

שים קובץ בתיקיית המקור, ודא שהוא נשלח ל-Kafka ואז נצרך ונרשם לתיקיית היעד.

● מהו Apache NiFi Registry?

הגדרה

Apache NiFi Registry הוא subproject של Apache NiFi.

זהו יישום משלים שמספק מיקום מרכזי לאחסון וניהול משאבים משותפים על פני instance אחד או יותר של NiFi.

מטרות עיקריות:

  • Flow Registry - אחסון וניהול flows בגרסאות
  • Integration with NiFi - אינטגרציה עם NiFi לאחסון, אחזור ושדרוג flows בגרסאות
  • Version Control - שליטה בגרסאות של flows
  • Collaboration - עבודה משותפת על flows

● תרגיל - הבנת ממשק המשתמש של NiFi Registry

מטרה:

להכיר את ממשק המשתמש של NiFi Registry ואת הפונקציונליות הבסיסית.

שלבים:

  1. התחברות ל-Registry:
    • התחבר ל-NiFi Registry UI (בדרך כלל http://localhost:18080/nifi-registry)
    • ודא שה-Registry פועל וזמין
  2. יצירת Bucket:
    • צור Bucket חדש לאחסון flows
    • תן שם משמעותי (למשל: "Production Flows")
    • הוסף תיאור אם נדרש
  3. הגדרת NiFi להתחברות ל-Registry:
    • ב-NiFi, עבור ל-Controller Settings
    • הוסף Registry Client חדש
    • הגדר את ה-URL של ה-Registry
    • ודא שהחיבור עובד
  4. שמירת Flow ל-Registry:
    • ב-NiFi, בחר Process Group
    • לחץ על "Version" → "Start Version Control"
    • בחר את ה-Registry וה-Bucket
    • תן שם ל-Flow Version
    • שמור את ה-Flow
  5. צפייה ב-Registry:
    • עבור ל-Registry UI
    • בדוק שה-Flow נשמר ב-Bucket
    • צפה בפרטי ה-Version
  6. שדרוג Flow:
    • ערוך את ה-Flow ב-NiFi
    • שמור גרסה חדשה ל-Registry
    • בדוק ב-Registry שיש מספר גרסאות
Registry Client Configuration in NiFi: - Registry URL: http://localhost:18080 - Registry Name: Local Registry - Description: Local NiFi Registry instance

💡 Best Practices:

  • השתמש ב-Buckets שונים לסביבות שונות (Dev, Test, Prod)
  • תן שמות משמעותיים ל-Flow Versions
  • הוסף הערות בעת שמירת גרסה חדשה
  • בדוק את ה-Version History לפני שדרוג

✅ תוצאה צפויה:

Flow נשמר ב-Registry עם version control מלא, וניתן לשדרג אותו בין סביבות שונות.
