Apache Kafka

Master Guide

    🚀 Intro & Concepts

    Apache Kafka is a distributed streaming platform based on the publish-subscribe model. It is designed to be fast, scalable, and fault-tolerant.

    ETL vs. Data Streaming

    Old World (ETL): Batch processing, slow, resource-intensive, runs on schedule (not real-time).

    New World (Streaming): Real-time processing, low latency, fault-tolerant, scalable, handles continuous event streams.

    Broker
    A Kafka server. A cluster is made of multiple brokers.
    Topic
    A logical channel for data. Producers write to topics, Consumers read from them.
    Partition
    Topics are split into partitions for scalability. Ordering is guaranteed only within a partition.
    Offset
    A unique incremental ID assigned to every message in a partition.
    Replication
    Copies of data across brokers for High Availability (HA).
    Zookeeper
    Manages brokers, leader election, and cluster membership. Mandatory for Kafka.

    🏗️ Architecture

    Understanding how the components fit together.

    Producer
    Writes data to topics. Can choose acknowledgment levels (acks=0, 1, or all).
    Consumer
    Reads data from topics/partitions.
    Consumer Group
    A group of consumers working together. Each partition is consumed by only one member of the group for load balancing.

    💻 CLI Commands

    Essential commands for managing your Kafka cluster. (Note: Paths may vary based on installation).

    1. Service Management

    Start Zookeeper
    Must be started before Kafka.
    bin/zookeeper-server-start.sh config/zookeeper.properties
    Start Kafka
    Starts the broker.
    bin/kafka-server-start.sh config/server.properties

    2. Topic Management

    Create Topic
    Create a topic with specific replication and partitions.
    bin/kafka-topics.sh --create --zookeeper localhost:2181 --replication-factor 1 --partitions 1 --topic kafka-tst-01
    List Topics
    Show all available topics in the cluster.
    bin/kafka-topics.sh --zookeeper localhost:2181 --list
    Describe Topic
    View details like partition count and leader.
    bin/kafka-topics.sh --zookeeper localhost:2181 --describe --topic kafka-tst-01

    3. Console Clients

    Console Producer
    Open an interactive shell to send messages.
    bin/kafka-console-producer.sh --broker-list localhost:9092 --topic kafka-tst-01
    Console Consumer
    Read messages from the beginning of the log.
    bin/kafka-console-consumer.sh --bootstrap-server localhost:9092 --topic kafka-tst-01 --from-beginning

    ⚙️ Broker Configuration

    The core configuration for the Kafka Server is typically found in config/server.properties. Here are the most critical settings to understand.

    1. Connectivity

    broker.id
    A unique integer identifier for each broker in the cluster.
    broker.id=0
    listeners
    The socket server listens on. Format: PROTOCOL://YOUR.HOST.NAME:PORT.
    listeners=PLAINTEXT://localhost:9092
    advertised.listeners
    The specific IP/Port clients should connect to. Crucial for Docker/Cloud setups.
    advertised.listeners=PLAINTEXT://public-ip:9092
    zookeeper.connect
    Zookeeper connection string. For clusters, list all ZK nodes comma-separated.
    zookeeper.connect=localhost:2181

    2. Logs & Retention

    log.dirs
    Directory where Kafka persists data. Can be multiple paths.
    log.dirs=/tmp/kafka-logs
    log.retention.hours
    How long to keep data before deleting it. Default is 168 hours (7 days).
    log.retention.hours=168
    auto.create.topics.enable
    Allow topics to be created automatically when a producer writes to them.
    auto.create.topics.enable=true

    3. Full Example (server.properties)

    Production Template
    Common configuration pattern for a broker.
    # Server Basics broker.id=1 # Network listeners=PLAINTEXT://0.0.0.0:9092 advertised.listeners=PLAINTEXT://kafka-broker-1:9092 # Data log.dirs=/var/lib/kafka/data num.partitions=3 default.replication.factor=2 # Retention (7 Days) log.retention.hours=168 log.segment.bytes=1073741824 # Zookeeper zookeeper.connect=zk1:2181,zk2:2181,zk3:2181 zookeeper.connection.timeout.ms=18000

    🔌 Kafka Connect

    A framework to stream data into and out of Kafka without writing custom code.

    Architecture

    • Connectors: The logic (JAR) to connect to systems (HDFS, JDBC, File).
    • Tasks: The actual execution units moving the data.
    • Workers: Processes running the connectors and tasks.

    Configuration Examples

    Source Config (File)
    Example properties for reading from a file.
    name=kafka-file-source-task connector.class=FileStreamSource tasks.max=1 file=/tmp/sourceFile.log topic=file-source-test
    Sink Config (HDFS)
    Example properties for writing to Hadoop.
    name=hdfs-sink-task connector.class=HdfsSinkConnector tasks.max=1 topics=file-source-test hdfs.url=hdfs://localhost:9000
    Run Standalone
    Execute the connector with config files.
    bin/connect-standalone.sh config/connect-standalone.properties config/file-source.properties

    🧩 Core APIs

    The 4 main APIs used for development.

    1. Producer API
    Allows applications to send data to Kafka topics.
    2. Consumer API
    Allows applications to read streams of data from topics.
    3. Streams API
    Allows transforming data (filter, map, aggregate) from input topics to output topics.
    4. Connect API
    Reusable source/sink connectors (e.g., DB to Kafka).
