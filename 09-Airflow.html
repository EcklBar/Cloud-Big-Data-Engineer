🎯 מבוא ל-Apache Airflow

מהו Apache Airflow?

Airflow הוא פלטפורמה לכתיבה, תזמון וניטור של workflows באופן פרוגרמטי.

השתמש ב-Airflow לכתוב workflows כ-Directed Acyclic Graphs (DAGs) של משימות.

מאפיינים מרכזיים

📝

Programmatic

כתיבת workflows כקוד Python

⏰

Scheduling

תזמון אוטומטי של משימות

📊

Monitoring

מעקב וניטור בזמן אמת

🔄

Dynamic

pipelines דינמיים ומתפתחים

🔌

Extensible

ניתן להרחבה עם plugins

🎨

Rich UI

ממשק משתמש אינטואיטיבי

יתרונות Airflow

✅ למה Airflow?

  • Workflows as Code - workflows מוגדרים כקוד Python
  • Maintainable - קל לתחזוקה, versioning ו-testing
  • Collaborative - מאפשר שיתוף פעולה בין צוותים
  • Rich CLI - פקודות שורת פקודה עוצמתיות
  • Scalable - מתאים לכל גודל ארגון

תהליך בסיסי ב-Airflow

כתיבת DAG בPython
↓
Scheduler מזהה ומפעיל משימות
↓
Executor מריץ את המשימות
↓
Workers מבצעים את העבודה

🏠 נוצר ב-Airbnb

Airbnb היא חברה שגדלה מהר ומבוססת נתונים (data-informed).

צוותי הנתונים וכמות הנתונים גדלים במהירות, ובהתאמה גם המורכבות של האתגרים שהם מתמודדים איתם.

הצורך בפתרון

כוח העבודה הגדל של data engineers, data scientists ו-analysts ב-Airbnb זקוק לפלטפורמה שתאפשר תנועה מהירה.

פיתוח Airflow

Airbnb בנתה את Airflow כדי לאפשר למהנדסים לכתוב (author), לנטר (monitor) ולשדרג (retrofit) data pipelines.

Open Source

Apache Incubator

Airflow הצטרף ל-Apache Software Foundation והפך לפרויקט Apache מלא.

היום

🆚 Airflow vs Cron Jobs

  • Cron: פשוט אבל מוגבל - קשה לנהל תלויות, אין UI, אין monitoring
  • Airflow: מתוחכם - מנהל תלויות, UI מלא, monitoring, retry logic ועוד

🆚 Airflow vs Luigi

  • Luigi: פותח על ידי Spotify, פחות פופולרי
  • Airflow: קהילה גדולה יותר, UI טוב יותר, יותר plugins

🔀 מהו DAG?

Directed Acyclic Graph

DAG הוא גרף מכוון ללא מעגלים (ציקליים).

זהו הלב של Airflow - ייצוג של workflow שלם.

מרכיבי DAG

📊 מבנה

  • Nodes - Tasks (פעולות)
  • Edges - תלויות בין tasks
  • Directed - כל edge יש כיוון (מ-Task A ל-Task B)
  • Acyclic - אין מעגלים (לא יכול Task A להוביל בחזרה ל-Task A)

דוגמת DAG פשוטה

# Import libraries from airflow import DAG from airflow.operators.bash import BashOperator from datetime import datetime, timedelta # Default arguments default_args = { 'owner': 'airflow', 'depends_on_past': False, 'start_date': datetime(2024, 1, 1), 'retries': 1, 'retry_delay': timedelta(minutes=5), } # Create DAG dag = DAG( 'my_first_dag', default_args=default_args, description='A simple tutorial DAG', schedule_interval=timedelta(days=1), ) # Define tasks t1 = BashOperator( task_id='print_date', bash_command='date', dag=dag, ) t2 = BashOperator( task_id='sleep', bash_command='sleep 5', dag=dag, ) t3 = BashOperator( task_id='print_done', bash_command='echo "Done!"', dag=dag, ) # Set dependencies t1 >> t2 >> t3 # t1 then t2 then t3

תכונות חשובות של DAG

📅 Schedule

מתי ה-DAG צריך לרוץ

Cron expression או timedelta

📍 Start Date

מתי ה-DAG מתחיל להיות פעיל

Backfill אפשרי לתאריכים קודמים

🔄 Retries

כמה פעמים לנסות שוב במקרה של failure

retry_delay קובע כמה זמן לחכות

👤 Owner

מי אחראי על ה-DAG

שימושי לorganization וalerts

⚠️ כללים חשובים:

  • DAG חייב להיות Acyclic - אין לולאות!
  • כל task חייב להיות idempotent - הרצה מספר פעמים תיתן תוצאה זהה
  • תלויות חייבות להיות מפורשות וברורות

📋 Tasks & Operators

מהו Task?

Task הוא יחידת עבודה בודדת ב-DAG - צעד אטומי בpipeline.

ב-Airflow, tasks נוצרים באמצעות Operators.

סוגי Operators נפוצים

🐚 BashOperator

מריץ פקודות bash

task = BashOperator( task_id='my_task', bash_command='echo "Hello"', dag=dag )

🐍 PythonOperator

מריץ פונקציית Python

task = PythonOperator( task_id='my_task', python_callable=my_function, dag=dag )

📧 EmailOperator

שולח אימייל

task = EmailOperator( task_id='send_email', to='user@example.com', subject='Alert', dag=dag )

🗄️ SqlOperator

מריץ SQL query

task = PostgresOperator( task_id='run_query', sql='SELECT * FROM table', dag=dag )

📁 HiveOperator

מריץ Hive query

עבור data warehousing

🐘 SparkSubmitOperator

שולח Spark job

עבור big data processing

☁️ S3Operator

פעולות על AWS S3

העלאה/הורדה מS3

🎯 SimpleHttpOperator

קריאות HTTP/API

אינטגרציה עם APIs

🔄 DummyOperator

לא עושה כלום

שימושי לארגון לוגיקה

הגדרת תלויות בין Tasks

# Method 1: Using >> operator task1 >> task2 >> task3 # Linear: task1 → task2 → task3 # Method 2: Using set_upstream/set_downstream task2.set_upstream(task1) task3.set_upstream(task2) # Method 3: Multiple dependencies task1 >> [task2, task3] >> task4 # Fork and join # Method 4: Chain helper from airflow.models.baseoperator import chain chain(task1, task2, task3, task4)

💡 Idempotency - עקרון חשוב

כל task צריך להיות idempotent - הרצה מספר פעמים תיתן אותה תוצאה.

  • אם task נכשל ורץ שוב - צריך להיות בטוח
  • אל תסמוך על state חיצוני שמשתנה
  • השתמש ב-execution_date לא ב-datetime.now()

🔌 Hooks - חיבורים למערכות חיצוניות

מהו Hook?

Hook הוא ממשק לחיבור עם מערכות חיצוניות ובסיסי נתונים.

Hooks מכילים את הלוגיקה של החיבור - authentication, connection management וכו'.

Hooks נפוצים

🗄️ PostgresHook

חיבור ל-PostgreSQL

  • run() - הרצת query
  • get_records() - קבלת תוצאות
  • insert_rows() - הכנסת נתונים

🐬 MySqlHook

חיבור ל-MySQL

  • get_conn() - connection object
  • get_pandas_df() - DataFrame

🐘 HiveHook

חיבור ל-Hive

  • get_results() - תוצאות query
  • load_file() - טעינת קובץ

☁️ S3Hook

חיבור ל-AWS S3

  • load_file() - העלאה
  • download_file() - הורדה
  • list_keys() - רשימת קבצים

🔌 HttpHook

חיבור ל-HTTP endpoints

  • run() - בקשת HTTP
  • check_response() - בדיקת תגובה

📁 HDFSHook

חיבור ל-HDFS

  • load_file() - העלאה ל-HDFS
  • get_pandas_df() - קריאה לDF

שימוש ב-Hook

from airflow.providers.postgres.hooks.postgres import PostgresHook def my_function(): # Create hook instance hook = PostgresHook(postgres_conn_id='my_postgres') # Run query sql = "SELECT * FROM users WHERE age > 18" records = hook.get_records(sql) # Process results for record in records: print(record)

🔑 Connections

Hooks משתמשים ב-Connections שמוגדרים ב-Airflow UI:

  • Admin → Connections
  • הגדר: host, schema, login, password, port
  • Connection ID משמש בקוד
  • מאפשר הפרדה בין קוד ל-credentials

🏊 Pools & Queues

Pools - ניהול משאבים

מהו Pool?

Pool משמש להגבלת מספר tasks שרצים במקביל.

שימושי כדי למנוע עומס יתר על משאבים מסוימים.

דוגמה: הגבלת גישה למסד נתונים

תרחיש: יש לנו 100 tasks שצריכים לגשת למסד נתונים, אבל המסד יכול לטפל רק ב-10 חיבורים במקביל.

פתרון עם Pool:

  1. יוצרים pool בשם "database_pool" עם 10 slots
  2. כל task מוגדר עם pool='database_pool'
  3. רק 10 tasks ירוצו במקביל
  4. השאר יחכו בתור
# Define task with pool task = PythonOperator( task_id='query_db', python_callable=my_function, pool='database_pool', # Assign to pool dag=dag )

Queues - עדיפויות ב-Celery

מהו Queue?

כאשר משתמשים ב-CeleryExecutor, ניתן לשלוח tasks לqueues שונים.

כל queue יכול להיות מטופל על ידי workers שונים.

תרחיש שימוש

🚀 High Priority Queue

Tasks קריטיים שצריכים להתבצע מהר

Workers מהירים, יותר משאבים

🐢 Low Priority Queue

Tasks שיכולים לחכות

Workers איטיים, פחות משאבים

💾 Heavy Processing Queue

Tasks שצורכים הרבה CPU/RAM

Workers חזקים במיוחד

🌐 API Queue

Tasks שעושים קריאות API

Workers עם גישה לאינטרנט

# Assign task to specific queue task = BashOperator( task_id='heavy_processing', bash_command='python heavy_script.py', queue='heavy_queue', # Send to specific queue dag=dag )

📡 XComs - תקשורת בין Tasks

מהו XCom?

XCom (Cross-Communication) מאפשר ל-tasks לשתף מידע ביניהם.

זהו מנגנון להעברת נתונים קטנים בין tasks בDAG.

איך זה עובד?

Push & Pull

  • xcom_push() - Task שולח (push) ערך
  • xcom_pull() - Task אחר מושך (pull) את הערך
  • הנתונים נשמרים במסד הנתונים של Airflow
  • כל XCom מזוהה לפי: dag_id, task_id, execution_date, key

דוגמה

from airflow import DAG from airflow.operators.python import PythonOperator from datetime import datetime def push_function(**context): # Calculate some value value = 42 # Push to XCom context['ti'].xcom_push(key='my_key', value=value) # Or simply return (will be pushed automatically) return value def pull_function(**context): # Pull from XCom value = context['ti'].xcom_pull( task_ids='push_task', key='return_value' # default key for return values ) print(f"Received value: {value}") # Create DAG dag = DAG('xcom_example', start_date=datetime(2024, 1, 1)) push_task = PythonOperator( task_id='push_task', python_callable=push_function, provide_context=True, dag=dag ) pull_task = PythonOperator( task_id='pull_task', python_callable=pull_function, provide_context=True, dag=dag ) push_task >> pull_task

⚠️ מגבלות XCom:

  • גודל מוגבל - מיועד לנתונים קטנים (metadata, IDs, paths)
  • לא לדאטה גדולה - אל תשתמש לDataFrames גדולים!
  • Performance - שמירה במסד נתונים = overhead
  • חלופה: שמור דאטה גדולה ב-S3/HDFS והעבר רק את הpath

🔄 SubDAGs - ארגון DAGs מורכבים

מהו SubDAG?

SubDAG הוא DAG בתוך DAG - דרך לארגן קוד ולעשות reuse.

שימושי כשיש קבוצת tasks שחוזרת במספר מקומות.

מתי להשתמש?

תרחישים נפוצים:

  • Pattern חוזר - אותה סדרת tasks במספר DAGs
  • ארגון לוגי - קיבוץ tasks קשורים
  • Readability - הפיכת DAG מורכב לפשוט יותר
  • Reusability - שימוש חוזר בקוד

דוגמה

from airflow import DAG from airflow.operators.bash import BashOperator from airflow.operators.subdag import SubDagOperator from datetime import datetime # Define SubDAG function def sub_dag(parent_dag_name, child_dag_name, args): dag_subdag = DAG( dag_id=f'{parent_dag_name}.{child_dag_name}', default_args=args, schedule_interval="@daily", ) for i in range(3): BashOperator( task_id=f'task_{i}', bash_command=f'echo "Subtask {i}"', dag=dag_subdag, ) return dag_subdag # Main DAG dag = DAG( 'parent_dag', start_date=datetime(2024, 1, 1), schedule_interval='@daily' ) # Use SubDAG section_1 = SubDagOperator( task_id='section_1', subdag=sub_dag('parent_dag', 'section_1', dag.default_args), dag=dag, )

⚠️ שים לב:

  • SubDAGs יכולים להיות מורכבים לdebug
  • יש להם overhead של resources
  • חלופה: TaskGroups (Airflow 2.0+) - פשוט יותר!

Sequential Executor הוא מצב הרצה הפשוט ביותר של Airflow.

מריץ task אחד בכל פעם - אין parallelism כלל.

יתרונות וחסרונות

✅ יתרונות:

  • הגדרה פשוטה - לא צריך broker
  • טוב לפרודקשן קטן-בינוני
  • פחות moving parts = פחות דברים שיכולים להישבר
  • עלות נמוכה - machine אחת

⚠️ חסרונות:

  • מוגבל למכונה אחת
  • אם המכונה נופלת - הכל נופל
  • לא ניתן לhorizontal scaling
  • מוגבל במספר workers (לפי CPU cores)

הגדרה

# airflow.cfg [core] executor = LocalExecutor parallelism = 32 # max tasks in parallel across all DAGs [database] sql_alchemy_conn = postgresql+psycopg2://user:pass@localhost/airflow

💡 מתי להשתמש?

  • פרודקשן קטן-בינוני (עד ~100 DAGs)
  • כשיש machine חזקה עם הרבה cores
  • כשרוצים פשטות וקלות תחזוקה

  • כשצוות ה-DevOps יכול לנהל את הinfrastructure

✨ Best Practices ל-Apache Airflow

🎯 תכנון DAGs

עקרונות בסיסיים:

  • Idempotency - כל task צריך להיות idempotent
  • Atomicity - כל task צריך לעשות דבר אחד טוב
  • Deterministic - אותה input = אותה output
  • Independent - tasks לא תלויים בstate חיצוני משתנה

📝 כתיבת קוד

✅ DO

  • השתמש ב-Variables לconfig
  • הגדר timeouts לtasks
  • הוסף retries לtasks שיכולים להיכשל
  • השתמש ב-execution_date
  • שמור credentials ב-Connections
  • תעד את הDAGs שלך

❌ DON'T

  • אל תשתמש ב-datetime.now()
  • אל תקשיח credentials בקוד
  • אל תשתמש ב-XCom לדאטה גדולה
  • אל תעשה top-level code שרץ בimport
  • אל תשתמש במשתנים גלובליים

🏗️ ארכיטקטורה

בחירת Executor:

  • Development: SequentialExecutor
  • Small Production: LocalExecutor
  • Large Production: CeleryExecutor או KubernetesExecutor

📊 Monitoring & Alerting

מה לנטר?

  • DAG duration - כמה זמן לוקח לDAG להסתיים
  • Task failures - אילו tasks נכשלים
  • Queue size - כמה tasks בתור
  • Resource usage - CPU, Memory, Disk
  • SLA misses - האם עומדים בזמנים

🔧 אופטימיזציה

⚡ Performance

  • השתמש ב-Pools נבונה
  • הגדר parallelism נכון
  • השתמש ב-catchup=False אם לא צריך
  • נקה DAGs ישנים

💰 Cost

  • הרץ tasks כבדים בזמנים זולים
  • השתמש ב-spot instances לworkers
  • אופטימז את הmemory usage
  • השתמש ב-KubernetesExecutor לscale

🔒 Security

⚠️ אבטחה:

  • אל תשתף connection IDs בקוד
  • הגבל גישה ל-UI רק למורשים
  • השתמש ב-secrets backend (AWS Secrets Manager, Vault)

🚀 התקנת Apache Airflow עם Docker Compose

למה Docker Compose?

Docker Compose מאפשר להקים את Airflow עם כל הרכיבים הנדרשים (scheduler, webserver, database, redis) בקלות ובמהירות.

זהו הדרך המומלצת להתקנה מקומית ולפיתוח.

שיטה 1: שימוש ב-CDE Docker Compose (מומלץ)

1הרצת Docker Compose

הרץ את הפקודה הבאה בתיקיית הפרויקט:

docker compose up -d

הערה: אם כבר הרצת את הפקודה בשיעורים קודמים, אין צורך להריץ שוב.

2בדיקה ש-Airflow רץ

פתח את Docker Desktop וחפש את הקונטיינרים הבאים:

  • airflow_scheduler - המתזמן שמריץ tasks
  • airflow_webserver - ממשק המשתמש

בנוסף, בדוק שנוצרו 2 תיקיות חדשות בפרויקט:

  • airflow-data - נתוני Airflow
  • dags - כאן תשים את קבצי ה-DAG שלך

3גישה לממשק המשתמש

פתח דפדפן וגש לכתובת:

http://localhost:8080

התחבר עם הפרטים הבאים:

  • שם משתמש: airflow
  • סיסמה: airflow

✅ סיימת! Airflow רץ במערכת שלך

אתה יכול עכשיו להתחיל ליצור DAGs בתיקייה ./dags

שיטה 2: התקנה ידנית מלאה

1הגדרת תיקיות וסביבה

צור את התיקיות הנדרשות והגדר את UID:

mkdir -p ./dags ./logs ./plugins ./config echo -e "AIRFLOW_UID=$(id -u)" > .env

2אתחול Airflow

הרץ את תהליך האתחול שיוצר את בסיס הנתונים ומשתמש Admin:

docker compose up airflow-init

המתן עד שתראה הודעה דומה ל:

airflow-init_1 | Upgrades done airflow-init_1 | Admin user airflow created airflow-init_1 | 2.9.2 start airflow-init_1 | exited with code 0

3הפעלת כל הServices

הפעל את כל רכיבי Airflow:

docker compose up -d

4אימות Containers רצים

בדוק שכל הקונטיינרים רצים:

docker ps

אמור לראות:

  • airflow-webserver - port 8080
  • airflow-scheduler
  • airflow-worker
  • postgres - port 5432
  • redis - port 6379

5גישה ל-UI

פתח דפדפן וגש לכתובת:

http://localhost:8080

התחבר עם: airflow / airflow

🎉 התקנה הושלמה בהצלחה!

כל הרכיבים של Airflow רצים במערכת שלך עם CeleryExecutor מלא.

🛠️ פקודות שימושיות

# עצור את Airflow docker compose down # עצור ומחק volumes (נתונים) docker compose down --volumes # הצג לוגים של service מסוים docker compose logs webserver docker compose logs scheduler # הרץ פקודת Airflow CLI docker compose run airflow-worker airflow dags list # כניסה לקונטיינר docker exec -it <container-id> bash

⚠️ שים לב:

  • ה-setup הזה משתמש ב-CeleryExecutor - מתאים לפרודקשן
  • דורש לפחות 4GB RAM פנויים
  • וודא ש-Docker Desktop רץ לפני ההתקנה
  • בפעם הראשונה הinit יכול לקחת מספר דקות

🔍 פתרון בעיות נפוצות

Port 8080 תפוס

אם ה-port תפוס, שנה ב-docker-compose.yaml:

ports: - "8081:8080" # שנה ל-8081 או כל port פנוי אחר

Containers לא עולים

בדוק לוגים:

docker compose logs

אין הרשאות לתיקיות

הרץ:

sudo chown -R $(id -u):$(id -g) ./dags ./logs ./plugins

📚 צעד הבא

עכשיו שAirflow מותקן, עבור לטאב תיאוריה כדי ללמוד איך ליצור DAGs!

🏗️ ארכיטקטורת Airflow

Airflow תומך ב-3 מצבי ארכיטקטורה שונים, כל אחד עם יתרונות וחסרונות

1️⃣ Sequential Executor

הארכיטקטורה הפשוטה ביותר - מעבד משימה אחת בכל פעם

מאפיינים:

  • מריץ task אחד בכל פעם (אין parallelism)
  • מצב ברירת מחדל
  • setup מינימלי - עובד עם SQLite
  • לא מומלץ לפרודקשן

⚠️ שימוש: רק לפיתוח ובדיקות - לא לפרודקשן!

2️⃣ Local Executor

מריץ tasks במקביל על אותה המכונה

מאפיינים:

  • Processes נוצרים על ידי scheduler
  • Scales vertically (הוספת CPU/RAM)
  • ברמת production
  • לא צריך broker (כמו RabbitMQ)
  • מתאים לפרודקשן כשאין הרבה DAGs

✅ שימוש: פרודקשן קטן-בינוני, machine אחת

3️⃣ Celery Executor

ארכיטקטורה מבוזרת עם workers מרובים

מאפיינים:

  • Vertical וHorizontal scalable
  • ניתן לניטור (דרך Flower)
  • ברמת production
  • תומך ב-Pools ו-Queues
  • דורש broker (RabbitMQ/Redis)
  • מאפשר להפריד סוגי workloads שונים

✅ שימוש: פרודקשן בקנה מידה גדול, clusters

השוואה מהירה

תכונה Sequential Local Celery
Parallelism ❌ לא ✅ כן ✅ כן
Scalability ❌ אין ⬆️ Vertical ⬆️➡️ Both
Production Ready ❌ לא ✅ כן ✅ כן
Setup 🟢 פשוט 🟡 בינוני 🔴 מורכב
Monitoring בסיסי בסיסי ✅ Flower

🖥️ ממשק המשתמש של Airflow

Airflow מספק ממשק משתמש עשיר ואינטואיטיבי לניהול וניטור workflows

📋 DAGs List View

רשימת כל ה-DAGs במערכת
כולל סטטוס, לוחות זמנים, בעלים ופעולות

מה רואים?

  • שמות DAGs וסטטוס (active/paused)
  • הרצות אחרונות (success/failed)
  • Schedule interval
  • בעלים (owner)
  • פעולות: trigger, refresh, delete

🌳 Graph View

תצוגה גרפית של המבנה של DAG
מציגה tasks ואת התלויות ביניהם

מה רואים?

  • כל ה-tasks כnodes
  • חצים מציגים תלויות
  • צבעים מציגים סטטוס (green=success, red=failed)
  • לחיצה על task מציגה פרטים

🌲 Tree View

תצוגת עץ של הרצות DAG לאורך זמן
מציגה היסטוריה של ביצועים

מה רואים?

  • כל task בשורה נפרדת
  • עמודות מייצגות הרצות שונות
  • צבעים מציגים סטטוס לאורך זמן
  • מאפשר לזהות patterns של failures

📊 Gantt Chart

תרשים Gantt של זמני ריצה
מציג overlap ו-bottlenecks

מה רואים?

  • timeline של כל task
  • משך זמן ביצוע (duration)
  • overlap בין tasks
  • bottlenecks במערכת
  • שימושי לאופטימיזציה

⏱️ Task Duration

ויזואליזציה של משך זמן tasks
לאורך מספר הרצות

מה רואים?

  • גרף של duration לכל task
  • מגמות לאורך זמן
  • זיהוי tasks שמאטים
  • השוואה בין הרצות

📝 Code View

הצגת הקוד Python של DAG
ישירות בממשק

מה רואים?

  • קוד Python המלא של DAG
  • syntax highlighting
  • שימושי לdebug מהיר
  • הבנת לוגיקה מבלי לצאת מהממשק

💡 טיפים לשימוש בUI:

  • Graph View - הכי טוב להבנת מבנה ה-DAG
  • Tree View - הכי טוב לזיהוי patterns של failures
  • Gantt - הכי טוב לאופטימיזציה של ביצועים
  • Task Duration - הכי טוב לזיהוי bottlenecks
  • לחץ על tasks כדי לראות logs, retry, mark success/failed
