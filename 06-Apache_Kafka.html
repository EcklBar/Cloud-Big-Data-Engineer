Apache Kafka Masterclass

From "Old World" ETL to Real-Time Event Streaming.

1. The Problem & The Solution

Apache Kafka is a distributed streaming platform that acts as the Central Nervous System for data.

It powers real-time usage for giants like Netflix (Recommendations), Uber (Matching), and PayPal (Fraud Detection).

2. Evolution: ETL vs. Streaming

⏳ Old World (ETL)

  • Batch: Data processed nightly/hourly.
  • Rigid: Point-to-point "spaghetti" connections.
  • High Latency: Decisions are made on old data.

⚡ New World (Streaming)

  • Real-Time: Events processed as they occur.
  • Decoupled: Producers don't know Consumers.
  • Scalable: Handles massive throughput.

3. High-Level Architecture

The flow is simple: Producers push data in, the Cluster holds it, and Consumers pull it out.

1. Producers

Source Systems (Web Apps, IoT, DBs) that create and push events.

2. Kafka Cluster

A set of Brokers that store streams of records in categories called Topics.

3. Consumers

Target Systems (DWH, Analytics) that subscribe and process events.

4. Core Concepts (The Engine)

4.1 Zookeeper & Brokers

Kafka is a distributed system consisting of:

  • Zookeeper: The "Brain". Manages cluster state, leader election, and config.
  • Brokers: The "Body". Servers that store data. Connect to one, and you connect to all.

4.2 Topics, Partitions & Offsets

Topic: A logical stream (e.g., "Transactions"). Topics are split into Partitions.

⚠️
Master's Note on Ordering:
Order is guaranteed only within a partition. If you need total ordering, you must use 1 partition (which limits scale).

4.3 Replication

Partitions are replicated for fault tolerance. Each partition has:

  • Leader: Handles all Reads/Writes.
  • Follower: Passive backup. Takes over if the Leader dies.

5. Advanced Configurations

Producer Acks (Reliability)

How safe do you want your data to be?

  • acks=0: Fire & Forget. (Fastest, risky).
  • acks=1: Leader received it. (Default).
  • acks=all: Leader AND all Replicas received it. (Slowest, safest).

Consumer Groups (Scaling)

A group of consumers working together to read a topic.

Rule: One partition can be consumed by only one consumer in a group at a time.

7. Installation (Linux/Mac)

1

Download & Unpack

$ tar -xzf kafka_2.13-3.0.0.tgz
$ cd kafka_2.13-3.0.0
2

Start the Engines

Run in two separate terminals.

Terminal 1 (Zookeeper)
$ bin/zookeeper-server-start.sh config/zookeeper.properties
Terminal 2 (Kafka)
$ bin/kafka-server-start.sh config/server.properties

8. CLI Implementation

Creating a Topic

We create a topic named kafka-test-01 with 1 partition.

$ bin/kafka-topics.sh --create \
  --bootstrap-server localhost:9092 \
  --replication-factor 1 \
  --partitions 1 \
  --topic kafka-test-01

Real-Time Demo

Producer (Write):

$ bin/kafka-console-producer.sh --bootstrap-server localhost:9092 --topic kafka-test-01
> Hello Kafka
> This is a stream

Consumer (Read):

$ bin/kafka-console-consumer.sh --bootstrap-server localhost:9092 --topic kafka-test-01 --from-beginning
Hello Kafka
This is a stream
