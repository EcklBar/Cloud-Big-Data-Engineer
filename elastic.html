🎯 מבוא ל-Apache Kafka

מהו Apache Kafka?

Apache Kafka הוא פלטפורמה מבוזרת לסטרימינג נתונים המבוססת על מודל pub-sub (publish-subscribe).

Kafka הוא מהיר להפליא, סקלבילי, עמיד לתקלות ויכול לשמש כפלטפורמת נתונים מרכזית.

למה Kafka?

בעידן המודרני, ארגונים צריכים לעבד נתונים בזמן אמת:

⚡ Real-Time

עיבוד נתונים ברגע שהם נוצרים, לא בבאצ'ים

📈 סקלביליות

התמודדות עם מיליוני events בשנייה

🔒 Fault Tolerance

המערכת ממשיכה לעבוד גם בתקלות

🌐 מבוזר

חלוקת עומס על מספר שרתים

הבעיה שKafka פותר

🔄 מערכות ישנות (ETL)

  • רצות בזמנים מוגדרים (לא real-time)
  • מורכבות בהתמודדות עם נתונים רבים
  • קשות לסקלביליות
  • עלויות תפעול גבוהות

✨ הפתרון: Data Streaming עם Kafka

  • מערכת עצבים מרכזית לכל הנתונים בארגון
  • Message bus סקלבילי בזמן אמת
  • Source of Truth - מקור אמת יחיד לכל הנתונים
  • מזין מערכות: Databases, Data Warehouses, Hadoop, NoSQL ועוד

⚡ ETL vs Data Streaming

ETL - העולם הישן

⚠️ מגבלות ETL מסורתי:

  • התפתח בשנות ה-90, בעיקר לניתוח נתונים
  • רץ בזמנים מוגדרים - לא בזמן אמת
  • מורכב ב-incremental capturing
  • רץ על databases יחידים (Operational / DWH)
  • מתאים בעיקר ל-schematic data
  • קשה מאוד לסקלביליות בעומסים גבוהים
  • עלות תפעולית גבוהה - איטי ודורש משאבים
  • תמיכה מוגבלת או אפסית ב-real-time

Data Streaming - העולם החדש

✅ דרישות של העידן המודרני:

  • Fault Tolerance - עמידות בפני תקלות
  • Parallel Processing - עיבוד מקבילי
  • Low Latency - זמן תגובה נמוך | Real-Time
  • תמיכה בנתונים schematic ו-non-schematic
  • קל לתפעול וניטור
  • פתרון reusable מלא ולא רק כלי בודד

Apache Kafka הוא מערכת הודעות מבוזרת המבוססת על עקרון pub-sub (publish-subscribe).

מאפיינים מרכזיים

⚡ מהיר להפליא

מסוגל לעבד מיליוני messages בשנייה עם latency של מילישניות

📈 Highly Scalable

ניתן להרחבה אופקית - הוספת brokers ללא הפסקת שירות

🔒 Fault-Tolerant

עמיד לתקלות באמצעות replication ו-distributed architecture

💾 יכול לשמש כאחסון

אחסון זרם של records לפרקי זמן מוגדרים

Kafka כפלטפורמת Streaming

🌊 יכולות מרכזיות:

  • הפלטפורמה הפופולרית ביותר לsטרימינג נתונים
  • מספקת messaging backbone סקלבילי לאינטגרציית נתונים
  • מאפשרת בניית data pipelines לסטרימינג
  • מספקת לוגיקה לעיבוד וטרנספורמציה של נתונים
  • Event/Message handler - מטפל בשינויי נתונים בטבלאות, לוגים ועוד

חיבורים למקורות נתונים

  • מספק APIs לחיבור למקורות נתונים מרובים (Apps, DBs)
  • מטפל ב-high-volume ו-high-diversity של נתונים
  • Event-Centric Thinking - עיבוד אירועים ברגע שהם מתרחשים

🔧 רכיבים עיקריים ב-Kafka

📤 Producer

אפליקציה ששולחת (מפרסמת) messages ל-Kafka topics

  • יכול לבחור ל-איזה partition לשלוח
  • תומך ב-batching
  • תומך ב-compression

📥 Consumer

אפליקציה שקוראת messages מ-Kafka topics

  • שייכת ל-consumer group
  • שומרת offset
  • יכולה לקרוא במקביל

🖥️ Broker

שרת Kafka המאחסן ומנהל topics

  • חלק מ-cluster
  • מנהל partitions
  • מטפל ב-replication

🐘 ZooKeeper

מנהל את מטא-דאטה של ה-cluster

  • בוחר leader לpartitions
  • שומר קונפיגורציה
  • (בגרסאות חדשות - Kraft במקום)

📂 Topic

קטגוריה/feed name שאליו נשלחים messages

  • מחולק ל-partitions
  • ניתן ל-replication
  • כמו "טבלה" במסד נתונים

🗂️ Partition

חלוקה פיזית של topic לחלקים

  • ordered, immutable
  • כל message מקבל offset
  • מאפשר parallelism

📂 Topics & Partitions

מהו Topic?

Topic הוא stream של records. זה category או feed name שאליו נשלחים messages.

כל topic מזוהה על ידי שם ייחודי.

מאפיינים:

  • Topics הם multi-subscriber - יכולים להיות מספר consumers
  • Messages נשמרים לפרק זמן מוגדר (retention)
  • ניתן להגדיר compression (gzip, snappy, lz4)

מהו Partition?

🗂️ Partition = חלוקה של Topic

כל topic מחולק ל-partitions (אחד או יותר).

כל partition הוא ordered, immutable sequence של records.

מבנה Topic עם 3 Partitions

Topic: user-events Partition 0: [msg0] → [msg3] → [msg6] → [msg9] ... Partition 1: [msg1] → [msg4] → [msg7] → [msg10] ... Partition 2: [msg2] → [msg5] → [msg8] → [msg11] ... כל message מקבל offset ייחודי בpartition שלו

למה Partitions?

📈 Scalability

חלוקת הנתונים על מספר servers

⚡ Parallelism

מספר consumers יכולים לקרוא במקביל

🔒 Ordering

סדר מובטח בתוך partition

💾 Distribution

פיזור העומס והאחסון

⚠️ חשוב לזכור:

  • Ordering מובטח רק בתוך partition - לא בין partitions
  • Producer בוחר ל-איזה partition לשלוח (לפי key או round-robin)
  • מספר partitions לא ניתן לשינוי בקלות - תכנן מראש!

🖥️ Brokers & Clusters

מהו Broker?

Kafka Broker הוא שרת Kafka. זה תהליך שרץ על מכונה ומנהל את אחסון ה-messages.

תפקידי ה-Broker:

  • מקבל messages מ-producers
  • מאחסן messages על הדיסק
  • משרת messages ל-consumers
  • מנהל replication של partitions
  • מטפל ב-leader election

מהו Cluster?

🌐 Kafka Cluster

אוסף של מספר Kafka Brokers שעובדים יחד כמערכת אחת.

כל broker מזוהה על ידי broker ID ייחודי (מספר שלם).

Cluster עם 3 Brokers

Broker 0
Leader: P0, P1
Follower: P2
Broker 1
Leader: P2
Follower: P0
Broker 2
Leader: P3
Follower: P1

כל partition יש לו leader ו-followers
אם broker נופל - אחר הופך ל-leader

Replication

🔄 העתקה לגיבוי

כל partition יכול להיות replicated על מספר brokers.

  • Replication Factor קובע כמה עותקים
  • יש leader partition אחד - מטפל בכל הקריאות והכתיבות
  • יש follower partitions - רק מעתיקים נתונים
  • אם ה-leader נופל, אחד מה-followers הופך ל-leader

⚙️ Replication Factor מומלץ:

לפחות 3 בפרודקשן (מאפשר אובדן של עד 2 brokers)

🌊 Data Streaming

מהו Data Streaming?

Data streaming applications הם אפליקציות ששולחות נתונים ברגע שהם מגיעים או נוצרים.

הנתונים יכולים להיות מעובדים כדי להפיק ערכים.

Producer vs Consumer vs Stream Processing

📤 Producers

יוצרים records ושולחים אותם ל-Kafka

📥 Consumers

קוראים records מ-Kafka

🔄 Stream Processing

מעבדים ומנתחים את הנתונים בזמן אמת

⚠️ חשוב להבין:

Producers ו-Consumers רק מעבירים נתונים - לא מנתחים!

לניתוח צריך stream applications כמו:

  • Kafka Streams - ספריית Java native
  • Apache Spark (Java, Scala, Python, PySpark)
  • Apache Flink

תרחישי שימוש

📈 מסחר במניות

כאשר מניות נסחרות בבורסה, מחירי המניות משתנים.

אפליקציית streaming יכולה לקחת את המחירים ולשלוח אותם בזמן אמת ללקוחות או לאפליקציות אחרות שמשתמשות בנתונים לעיבוד נוסף.

🌐 ביקורים באתר & מכירות

עיבוד streaming יכול לספק פרטים על ביקורי לקוחות ומכירות עם נתוני השוואה בזמן אמת:

  • כמה משתמשים באתר כרגע?
  • מהן המכירות בשעה האחרונה לעומת אתמול?
  • אילו מוצרים הכי פופולריים עכשיו?

🔄 Message Flow

  1. Producer שולח message
  2. Broker מקבל ושומר
  3. Message מקבל offset
  4. Replication ל-followers
  5. Consumer קורא message
  6. Consumer מעדכן offset

⚙️ Key Components

  • ZooKeeper/KRaft: Coordination
  • Brokers: Storage & serving
  • Topics: Logical channels
  • Partitions: Physical division
  • Offsets: Message position

🎯 Guarantees

  • Ordering: Within partition
  • Durability: Configurable
  • At-least-once: Default
  • Exactly-once: Available

⚙️ פקודות Kafka Essential

💡 לפני שמתחילים

כל הפקודות מניחות ש-$KAFKA_HOME מוגדר לתיקיית ההתקנה של Kafka.

📂 Topics

יצירת Topic

$ $KAFKA_HOME/bin/kafka-topics.sh \ --create \ --bootstrap-server localhost:9092 \ --topic my-topic \ --partitions 3 \ --replication-factor 2

רשימת כל ה-Topics

$ $KAFKA_HOME/bin/kafka-topics.sh \ --list \ --bootstrap-server localhost:9092

תיאור Topic

$ $KAFKA_HOME/bin/kafka-topics.sh \ --describe \ --bootstrap-server localhost:9092 \ --topic my-topic

מחיקת Topic

$ $KAFKA_HOME/bin/kafka-topics.sh \ --delete \ --bootstrap-server localhost:9092 \ --topic my-topic

📤 Producer

Console Producer

$ $KAFKA_HOME/bin/kafka-console-producer.sh \ --bootstrap-server localhost:9092 \ --topic my-topic # Type messages and press Enter > Hello Kafka > This is a message > ^C (Ctrl+C to exit)

Producer עם Key

$ $KAFKA_HOME/bin/kafka-console-producer.sh \ --bootstrap-server localhost:9092 \ --topic my-topic \ --property "parse.key=true" \ --property "key.separator=:" # Format: key:value > user1:Login event > user2:Logout event

📥 Consumer

Console Consumer

$ $KAFKA_HOME/bin/kafka-console-consumer.sh \ --bootstrap-server localhost:9092 \ --topic my-topic \ --from-beginning # Shows all messages from the beginning

Consumer עם Group

$ $KAFKA_HOME/bin/kafka-console-consumer.sh \ --bootstrap-server localhost:9092 \ --topic my-topic \ --group my-consumer-group # Part of a consumer group

👥 Consumer Groups

רשימת Consumer Groups

$ $KAFKA_HOME/bin/kafka-consumer-groups.sh \ --bootstrap-server localhost:9092 \ --list

מצב Consumer Group (Offset Status)

$ $KAFKA_HOME/bin/kafka-consumer-groups.sh \ --bootstrap-server localhost:9092 \ --group my-consumer-group \ --describe # Shows: TOPIC, PARTITION, CURRENT-OFFSET, LOG-END-OFFSET, LAG

איפוס Offsets להתחלה

$ $KAFKA_HOME/bin/kafka-consumer-groups.sh \ --bootstrap-server localhost:9092 \ --group my-consumer-group \ --reset-offsets \ --to-earliest \ --topic my-topic \ --execute

🔌 Kafka Connect (REST API)

רשימת Connectors פעילים

$ curl localhost:8083/connectors | python -m json.tool # Output: # [ # "jdbc-source-connector", # "hdfs-sink-connector" # ]

סטטוס Connector

$ curl localhost:8083/connectors/my-connector/status \ | python -m json.tool

Pause/Resume Connector

# Pause $ curl -X PUT localhost:8083/connectors/my-connector/pause # Resume $ curl -X PUT localhost:8083/connectors/my-connector/resume

Restart Connector

$ curl -X POST localhost:8083/connectors/my-connector/restart

מחיקת Connector

$ curl -X DELETE localhost:8083/connectors/my-connector

⚠️ טיפים חשובים

  • תמיד וודא ש-Kafka Cluster רץ לפני הרצת פקודות
  • השתמש ב---bootstrap-server במקום --zookeeper (deprecated)
  • Consumer Groups שומרים offset - שימושי לresume מאיפה שעצרת
  • תמיד בדוק LAG של consumer groups לניטור ביצועים
