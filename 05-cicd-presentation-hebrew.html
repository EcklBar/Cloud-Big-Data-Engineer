🎯 מבוא ל-CI/CD

מהו CI/CD?

CI/CD מייצג שני תהליכים מרכזיים בפיתוח תוכנה מודרני:

  • CI - Continuous Integration (אינטגרציה רציפה): תהליך של שילוב שינויי קוד באופן תכוף ואוטומטי
  • CD - Continuous Deployment (פריסה רציפה): תהליך של פריסת שינויים לסביבת הייצור באופן אוטומטי

מטרות CI/CD

⚡ מהירות

קיצור זמן הגעת features למשתמשים

בדיקות אוטומטיות לפני כל פריסה

🔄 אוטומציה

הפחתת טעויות אנוש

📊 שקיפות

מעקב אחר כל שינוי

📜 SDLC ההיסטורי

ניהול גרסאות מרכזי

פיתוח תוכנה נוהל באמצעות מערכות ניהול גרסאות מרכזיות (Centralized VCS)

מחזורי פיתוח ארוכים

כל release עבר מחזורים ארוכים: פיתוח → בדיקה → תיקון → חזרה

גרסאות מוגבלות

הפקת גרסה לא הייתה אירוע תכוף

הפצה פיזית

תוכנה שוחררה בחבילות גרסה: דיסקים, חבילות התקנה

עדכון ידני

לקוחות עדכנו באופן אקטיבי את גרסת התוכנה שלהם

⚠️ החסרונות של SDLC ההיסטורי

בעיות מרכזיות:

  • תיקוני באגים לוקחים זמן רב - מגיעים רק בגרסאות עתידיות
  • שיפורים מתרחשים בקפיצות - לא בצורה הדרגתית
  • הפקת release ידנית - מופעלת באופן ידני ולא אוטומטי
  • עלות גבוהה - של תיאום, בדיקות והפצה

💡 התוצאה: חוסר גמישות, תגובה איטית לבאגים, וחוויית משתמש לא אופטימלית

📈 התפתחות התוכנה

🔀 מורכבות התוכנה

  • מערכות הפכו מתוחכמות ומורכבות יותר
  • מעבר ל-Microservices - הרבה חלקים נעים חלקית-עצמאיים
  • קודי בסיס (codebase) הפכו מורכבים
  • ניהול גרסאות, תלויות ותאימות הפך קריטי

📊 Git והמהפכה המבוזרת

  • Git הציג דרך סקלבילית וזריזה לפתח תוכנה
  • מודל ה-branching המבוזר מאפשר פיתוח סימולטני
  • מיזוג קוד ל-mainline הפך לאירוע תכוף
  • האתגר: ניהול ותיאום החלקים
  • קוד ה-mainline חייב להישאר יציב - באג בו מחייב החזרה וא restart תהליך הבדיקה

  • Microservices הפכה לפופולרית - פירוק האפליקציה לחלקים עצמאיים רבים עם releases נפרדים אך מתואמים

☸️ Kubernetes (K8s)

מהו Kubernetes?

Kubernetes הוא מערכת לתזמור קונטיינרים (Container Orchestration)

  • שוחרר על ידי Google ב-2014 כפרויקט קוד פתוח
  • חולל מהפכה בסביבות ההרצה המודרניות

מאפיינים עיקריים

🎯 Orchestration

הרצת cluster של containers

📈 Scale

🔄 סקירת CI/CD

🔄 Continuous Integration

  • שילוב רציף של חלקי קוד ו-branches שונים
  • בנייה ובדיקה אוטומטית של הקוד
  • זיהוי מוקדם של בעיות
  • שמירה על mainline יציב

🚀 Continuous Deployment

  • גרסאות חדשות ארוזות ונבדקות אוטומטית
  • פרסום תיקונים ושיפורים ברגע שהם מוכנים
  • פריסה לסביבות שונות באוטומציה
  • מעקב וניטור אחר deployments

תהליך CI/CD טיפוסי:

  1. Code - המפתח כותב קוד
  2. Commit - העלאה ל-Git
  3. Build - בנייה אוטומטית
  4. Test - הרצת בדיקות
  5. Package - יצירת artifact (Docker image)
  6. Deploy - פריסה לסביבה
  7. Monitor - ניטור ומעקב

🎯 המטרה: מחזור קצר ככל האפשר מקוד לייצור, עם אוטומציה מקסימלית ואמינות גבוהה

🔧 Jenkins

מהו Jenkins?

  • פלטפורמת אוטומציה לתהליכי CI/CD
  • הוצג ב-2011
  • קוד פתוח (Open-Source)
  • ניתן להתאמה אישית והרחבה רחבה
  • שוק של plugins חינמיים

מאפיינים מרכזיים

⚙️ Job Runner

הגדרת jobs לטסקים שונים

🌳 היררכיה

מבנה jobs היררכי

🔐 הרשאות

ניהול משתמשים וקבוצות

🔄 Triggers

הפעלה אוטומטית (git push, merge)

ארכיטקטורה מבוזרת

  • Master-Slave Architecture
  • אחד או יותר Jenkins masters מנהלים את ה-jobs
  • Jobs רצים על slaves שונים (servers)
  • שיפור ביצועים ומהירות לפרויקטים גדולים
  • סקלביליות של מערכת ה-CI/CD

Jenkinsfile

pipeline { agent any stages { stage('Build') { steps { sh 'make build' } } stage('Test') { steps { sh 'make test' } } stage('Deploy') { steps { sh 'make deploy' } } } }

✅ יתרונות Jenkins

  • קוד פתוח וקהילתי
  • רב-תכליתי
  • ניתן להתאמה אישית רבה
  • ניתן להרחבה
  • חלוץ - לא היה משהו טוב יותר בזמנו

❌ חסרונות Jenkins

  • התקנה ותחזוקה מורכבות
  • מיושן (לא חלק כמו CI/CD מודרני)
  • שפת Groovy (שנאויה)
  • בעיות אבטחה ב-platform וב-plugins
  • over-customizable - כל ארגון ממציא את הגלגל

📝 Infrastructure as Code (IaC)

מהו IaC?

Infrastructure as Code הוא הקונספט של ניהול התשתית דרך קוד

  • מניפסטים מתארים את דרישות ה-provisioning
  • Stack של IaC שומר את המצב הנוכחי של משאבים
  • שינויים במניפסטים מובילים ל-delta של דרישות
  • משאבים מוחלפים אוטומטית כדי לשקף את הדרישות החדשות

טכנולוגיות IaC מובילות

🌍 Terraform

Stateful provisioning של תשתית (cross-)cloud

resource "aws_instance" "web" { ami = "ami-12345678" instance_type = "t2.micro" tags = { Name = "WebServer" } }

⎈ Helm

ניהול (define, install, upgrade) אפליקציות K8s

helm install my-app ./my-chart helm upgrade my-app ./my-chart helm rollback my-app

🔧 Cloud SDKs

AWS CDK, Go CDK - SDKs לענן

  • Templates לשלד הכללי
  • Values/Variables לפרטים ספציפיים

🎬 GitHub Actions

למה GitHub Actions?

  • המקום המוביל לאירוח פרויקטי קוד פתוח
  • מגוון פרויקטים מציע טווח רחב של דוגמאות CI/CD
  • פרויקטים ציבוריים נהנים מ-features מתקדמים בחינם
  • תמיכה נרחבת ב-actions ו-workflows של צד שלישי
  • framework יחסית פשוט וישיר

מושגים מרכזיים

📅 Events

אירועים שמפעילים workflows:

  • Push לענף
  • Pull Request
  • Timer (schedule)
  • ועוד...

📋 Workflows

תהליכים אוטומטיים:

  • מוגדרים בקבצי YAML
  • נמצאים ב-.github/workflows
  • מכילים job אחד או יותר

⚙️ Jobs & Steps

יחידות ביצוע:

  • Job רץ על container ייעודי
  • Steps מבצעים פעולות
  • רצים ברצף או במקביל

מבנה ארכיטקטורה

Workflow ├── Event (trigger) ├── Job 1 │ ├── runs-on: ubuntu-latest │ ├── Step 1: Checkout code │ ├── Step 2: Setup Node │ └── Step 3: Run tests └── Job 2 ├── needs: Job 1 ├── Step 1: Build └── Step 2: Deploy

📋 Workflows ב-GitHub Actions

הגדרת Workflow

  • כל workflow מוגדר בקובץ YAML
  • קבצים נמצאים תחת .github/workflows
  • Workflow חייב להגדיר:
    • איזה events מפעילים אותו
    • Job אחד או יותר

Jobs

  • כל job רץ על container ייעודי בתוך מכונה וירטואלית
  • Job חייב להגדיר:
    • את ה-container image לשימוש
    • Step אחד או יותר לביצוע

Steps

Script Steps

Steps שמריצים script חייבים להגדיר את ה-script

- name: Run tests run: npm test

Action Steps

Steps שמשתמשים ב-actions קיימים חייבים להגדיר:

  • נתיב מלא וגרסה
  • כל הארגומנטים הנדרשים
- uses: actions/checkout@v3 with: ref: main

💡 דוגמה מעשית

Workflow פשוט שרץ על כל push:

name: learn-github-actions on: [push] jobs: check-bats-version: runs-on: ubuntu-latest steps: - uses: actions/checkout@v3 - uses: actions/setup-node@v3 with: node-version: '14' - run: npm install -g bats - run: | echo "testing bats version:" bats -v

הסבר שלב אחר שלב

name: learn-github-actions

שם ה-workflow שיוצג ב-UI

on: [push]

הפעלה על כל push לכל ענף

jobs: check-bats-version

job בודד בשם check-bats-version

runs-on: ubuntu-latest

שימוש ב-container image של Ubuntu

actions/checkout@v3

git checkout של הקוד (ברירת מחדל: main)

actions/setup-node@v3

התקנת Node.js גרסה 14

run: npm install -g bats

הרצת script שמתקין את חבילת bats

run: | (multiline)

הדפסת הודעה והצגת גרסת bats

🔍 צפייה ב-Actions

  1. עבור לטאב Actions ב-repository שלך ב-GitHub
  2. לחץ על workflow בפאנל הצדדי השמאלי
  3. לחץ על הרצת workflow מהרשימה
  4. בחר job מהגרף
  5. בחר step
  6. ראה את ה-console output של ה-step הספציפי

דוגמאות נוספות

CI למהנדסי נתונים - Python

name: Python CI on: push: branches: [ main, develop ] pull_request: branches: [ main ] jobs: test: runs-on: ubuntu-latest steps: - uses: actions/checkout@v3 - name: Set up Python uses: actions/setup-python@v4 with: python-version: '3.9' - name: Install dependencies run: | pip install -r requirements.txt pip install pytest pytest-cov - name: Run tests run: pytest --cov=./ --cov-report=xml - name: Upload coverage uses: codecov/codecov-action@v3

Docker Build & Push

name: Docker Build on: push: branches: [ main ] jobs: build: runs-on: ubuntu-latest steps: - uses: actions/checkout@v3 - name: Login to DockerHub uses: docker/login-action@v2 with: username: ${{ secrets.DOCKERHUB_USERNAME }} password: ${{ secrets.DOCKERHUB_TOKEN }} - name: Build and push uses: docker/build-push-action@v4 with: context: . push: true tags: myusername/myapp:latest

🔥 דוגמאות Workflows נוספות

1. CI/CD למהנדסי נתונים - Spark Job

name: Spark Job CI/CD on: push: branches: [ main ] pull_request: branches: [ main ] jobs: test: runs-on: ubuntu-latest steps: - uses: actions/checkout@v3 - name: Set up Python uses: actions/setup-python@v4 with: python-version: '3.9' - name: Install PySpark run: | pip install pyspark pytest pip install -r requirements.txt - name: Run Unit Tests run: pytest tests/ build-and-push: needs: test runs-on: ubuntu-latest steps: - uses: actions/checkout@v3 - name: Build Docker Image run: docker build -t my-spark-job:${{ github.sha }} . - name: Push to Registry run: | echo "${{ secrets.DOCKER_PASSWORD }}" | docker login -u "${{ secrets.DOCKER_USERNAME }}" --password-stdin docker push my-spark-job:${{ github.sha }}

2. Airflow DAG Validation

name: Airflow DAG Validation on: pull_request: paths: - 'dags/**' jobs: validate-dags: runs-on: ubuntu-latest steps: - uses: actions/checkout@v3 - name: Set up Python uses: actions/setup-python@v4 with: python-version: '3.9' - name: Install Airflow run: | pip install apache-airflow==2.7.0 pip install -r requirements.txt - name: Validate DAG Syntax run: | python -m py_compile dags/*.py - name: Test DAG Imports run: | python -c "from airflow.models import DagBag; d = DagBag(dag_folder='dags/'); assert len(d.import_errors) == 0"

3. Data Quality Tests

name: Data Quality Tests on: schedule: - cron: '0 8 * * *' # Daily at 8 AM workflow_dispatch: jobs: data-quality: runs-on: ubuntu-latest steps: - uses: actions/checkout@v3 - name: Set up Python uses: actions/setup-python@v4 with: python-version: '3.9' - name: Install dependencies run: pip install great_expectations pandas - name: Run Data Quality Checks run: | python scripts/run_data_quality.py env: DB_HOST: ${{ secrets.DB_HOST }} DB_PASSWORD: ${{ secrets.DB_PASSWORD }} - name: Send Slack Notification on Failure if: failure() uses: slackapi/slack-github-action@v1 with: webhook-url: ${{ secrets.SLACK_WEBHOOK }} payload: | { "text": "❌ Data Quality checks failed!" }

4. Deploy to Kubernetes

name: Deploy to K8s on: push: branches: [ main ] jobs: deploy: runs-on: ubuntu-latest steps: - uses: actions/checkout@v3 - name: Configure kubectl uses: azure/k8s-set-context@v1 with: kubeconfig: ${{ secrets.KUBE_CONFIG }} - name: Deploy to Kubernetes run: | kubectl apply -f k8s/deployment.yaml kubectl rollout status deployment/my-app - name: Verify Deployment run: | kubectl get pods kubectl get services

5. Multi-Environment Deployment

name: Multi-Environment Deploy on: push: branches: [ main, staging, develop ] jobs: deploy: runs-on: ubuntu-latest steps: - uses: actions/checkout@v3 - name: Determine Environment id: env run: | if [[ ${{ github.ref }} == 'refs/heads/main' ]]; then echo "environment=production" >> $GITHUB_OUTPUT elif [[ ${{ github.ref }} == 'refs/heads/staging' ]]; then echo "environment=staging" >> $GITHUB_OUTPUT else echo "environment=development" >> $GITHUB_OUTPUT fi - name: Deploy to ${{ steps.env.outputs.environment }} run: | echo "Deploying to ${{ steps.env.outputs.environment }}" ./deploy.sh ${{ steps.env.outputs.environment }}

6. Scheduled Data Pipeline

name: Daily ETL Pipeline on: schedule: - cron: '0 2 * * *' # 2 AM daily workflow_dispatch: # Manual trigger jobs: etl: runs-on: ubuntu-latest steps: - uses: actions/checkout@v3 - name: Setup Python uses: actions/setup-python@v4 with: python-version: '3.9' - name: Run ETL run: python scripts/daily_etl.py env: SOURCE_DB: ${{ secrets.SOURCE_DB }} TARGET_DB: ${{ secrets.TARGET_DB }} - name: Notify on Success if: success() run: echo "ETL completed successfully" - name: Notify on Failure if: failure() run: echo "ETL failed - check logs"

💡 טיפים לWorkflows:

  • השתמש ב-workflow_dispatch להפעלה ידנית
  • הגדר needs לתלויות בין jobs
  • השתמש ב-if: success() או if: failure() לlogic תנאי
  • שמור secrets ב-GitHub Secrets (לא בקוד!)
  • הוסף caching ל-dependencies (מהיר יותר!)

📝 תרגילים מעשיים

תרגיל 1: Workflow ראשון - Python Testing

מטרה: צור workflow שמריץ tests על כל push

דרישות:

  • יווצר ב-.github/workflows/test.yml
  • ירוץ על push ל-main
  • יתקין Python 3.9
  • יתקין dependencies מ-requirements.txt
  • ירוץ pytest
name: Python Tests on: push: branches: [ main ] jobs: test: runs-on: ubuntu-latest steps: - name: Checkout code uses: actions/checkout@v3 - name: Set up Python 3.9 uses: actions/setup-python@v4 with: python-version: '3.9' - name: Install dependencies run: | pip install --upgrade pip pip install -r requirements.txt pip install pytest - name: Run tests run: pytest tests/ -v

תרגיל 2: Docker Build & Push

מטרה: בנה Docker image ושלח ל-DockerHub

דרישות:

  • Build image רק אחרי tests מצליחים
  • Tag עם commit SHA
  • Push ל-DockerHub
  • השתמש ב-GitHub Secrets לcredentials
name: Docker Build and Push on: push: branches: [ main ] jobs: test: runs-on: ubuntu-latest steps: - uses: actions/checkout@v3 - name: Run tests run: | pip install pytest pytest tests/ build: needs: test # רק אחרי שtests עברו runs-on: ubuntu-latest steps: - uses: actions/checkout@v3 - name: Log in to DockerHub uses: docker/login-action@v2 with: username: ${{ secrets.DOCKERHUB_USERNAME }} password: ${{ secrets.DOCKERHUB_TOKEN }} - name: Build and push uses: docker/build-push-action@v4 with: context: . push: true tags: | myusername/myapp:latest myusername/myapp:${{ github.sha }}

תרגיל 3: Scheduled Workflow - Daily Report

מטרה: צור workflow שרץ אוטומטית כל יום

דרישות:

  • ירוץ כל יום ב-7 בבוקר
  • ירוץ גם באופן ידני (workflow_dispatch)
  • יריץ סקריפט Python
  • ישלח notification על הצלחה/כשלון
name: Daily Report on: schedule: - cron: '0 7 * * *' # 7 AM daily workflow_dispatch: # Manual trigger jobs: generate-report: runs-on: ubuntu-latest steps: - uses: actions/checkout@v3 - name: Setup Python uses: actions/setup-python@v4 with: python-version: '3.9' - name: Install dependencies run: pip install pandas requests - name: Generate Report run: python scripts/daily_report.py - name: Success Notification if: success() run: echo "✅ Report generated successfully" - name: Failure Notification if: failure() run: echo "❌ Report generation failed"

🔧 פתרון בעיות ב-GitHub Actions

❌ Workflow לא רץ

סיבות אפשריות:

  • קובץ ה-YAML לא ב-.github/workflows/
  • שגיאת syntax ב-YAML
  • ה-trigger (on:) לא התקיים
  • Workflow disabled

✅ פתרונות:

# 1. בדוק מיקום הקובץ ls .github/workflows/ # 2. validate YAML syntax # השתמש ב-YAML validator online או: yamllint .github/workflows/my-workflow.yml # 3. בדוק ב-GitHub UI # Actions tab → All workflows → בחר workflow # תראה אם יש שגיאות

טיפ: בדוק את ה-"Actions" tab ב-repository - תראה לוג של כל ההרצות

⏱️ Workflow איטי מדי

בעיות נפוצות:

  • התקנת dependencies לוקחת זמן
  • Build כבד
  • Tests ארוכים

✅ אופטימיזציות:

1. Cache Dependencies:
- name: Cache pip packages uses: actions/cache@v3 with: path: ~/.cache/pip key: ${{ runner.os }}-pip-${{ hashFiles('**/requirements.txt') }} - name: Install dependencies run: pip install -r requirements.txt
2. Matrix Strategy (Parallel):
jobs: test: runs-on: ubuntu-latest strategy: matrix: python-version: ['3.8', '3.9', '3.10'] steps: - uses: actions/checkout@v3 - name: Set up Python ${{ matrix.python-version }} uses: actions/setup-python@v4 with: python-version: ${{ matrix.python-version }}
3. Job Dependencies:
# הרץ jobs קלים קודם, כבדים רק אם הקלים עברו jobs: lint: runs-on: ubuntu-latest steps: - run: flake8 . # Fast! test: needs: lint # רק אם lint עבר runs-on: ubuntu-latest steps: - run: pytest # Slower deploy: needs: [lint, test] # רק אם שניהם עברו runs-on: ubuntu-latest steps: - run: ./deploy.sh

🔐 Secrets לא עובדים

בעיות:

  • Secret לא מוגדר
  • שם Secret לא נכון
  • Secret לא זמין לworkflow

✅ פתרון:

  1. Settings → Secrets and variables → Actions
  2. New repository secret
  3. שם ב-UPPERCASE (convention)
  4. גישה ב-workflow: ${{ secrets.SECRET_NAME }}
- name: Use Secret run: echo "API Key is set" env: API_KEY: ${{ secrets.API_KEY }} # ⚠️ לעולם אל תדפיס secrets! # ❌ run: echo ${{ secrets.API_KEY }} # BAD! # ✅ run: echo "Using API key..." # GOOD!

  4. ✅ Secrets מוגדרים?

✨ CI/CD Best Practices

🎯 Workflow Design

✅ עשה:

  • קטן ומהיר: שמור workflows פשוטים וממוקדים
  • Fail Fast: הרץ tests מהירים קודם
  • Cache: השתמש ב-caching לdependencies
  • Parallel Jobs: הרץ במקביל כשאפשר
  • Descriptive Names: שמות ברורים ל-workflows, jobs, steps
  • Status Badges: הוסף badges ל-README

❌ אל תעשה:

  • ❌ Hardcode secrets בקוד
  • ❌ workflows ענקיים בקובץ אחד
  • ❌ הרצת tests איטיים על כל commit קטן
  • ❌ Deploy לproduction ללא tests
  • ❌ שכחת timeout (workflow יכול לרוץ לנצח!)

🔒 Security Best Practices

⚠️ אבטחה:

  • ☐ Secrets: כל credentials ב-GitHub Secrets
  • ☐ Pin Versions: השתמש ב-@v3 לא @main
  • ☐ Least Privilege: הרשאות מינימליות
  • ☐ Review Actions: בדוק third-party actions
  • ☐ Branch Protection: הגן על main branch
  • ☐ Required Reviews: approve לפני merge
  • ☐ Audit Logs: עקוב אחר פעילות

⚡ Performance Tips

מהירות:

  • Cache: dependencies, build artifacts
  • Artifacts: שמור בין jobs
  • Conditional Steps: if: להרצה סלקטיבית
  • Matrix: parallel testing
  • Self-Hosted Runners: לפרויקטים גדולים
# Cache example - name: Cache node modules uses: actions/cache@v3 with: path: ~/.npm key: ${{ runner.os }}-node-${{ hashFiles('**/package-lock.json') }} # Conditional step - name: Deploy if: github.ref == 'refs/heads/main' run: ./deploy.sh

🎯 Production Checklist

  • ☐ Tests: unit, integration, e2e
