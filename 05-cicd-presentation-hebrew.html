CI/CD למהנדסי נתונים

NAYA College

סקירת הקורס וסדר היום

ברוכים הבאים לקורס המקיף על שיטות CI/CD למהנדסי נתונים. נחקור את ההתפתחות של שיטות פיתוח תוכנה וטכניקות אוטומציה מודרניות.

מה נלמד בקורס:

  • היסטוריה של SDLC (Software Development Lifecycle)
  • ה-SDLC המודרני
  • מהו CI/CD
    • אינטגרציה רציפה (Continuous Integration)
    • פריסה רציפה (Continuous Deployment)
  • דוגמאות מעשיות וניסיון מעשי

יעדי הלמידה

בסוף הקורס, תבינו את המושגים הבסיסיים של CI/CD, תכירו כלים ופלטפורמות פופולריות, ותוכלו ליישם צינורות CI/CD עבור פרויקטים של הנדסת נתונים.

SDLC היסטורי

הבנת המקום ממנו באנו עוזרת לנו להעריך את השיטות המודרניות.

מאפייני פיתוח תוכנה מסורתי:

  • פיתוח תוכנה מנוהל באמצעות מערכות בקרת גרסאות מרכזיות
  • כל גרסה עוברת מחזורים ארוכים (פיתוח, בדיקה, תיקון, חזרה)
  • יצירת גרסה חדשה אינה אירוע תדיר
  • תוכנה משוחררת בחבילות ממוספרות: חבילות, דיסקים וכו'
  • לקוחות מעדכנים באופן אקטיבי את הגרסה של התוכנה שלהם

החסרונות:

  • תיקוני באגים לוקחים זמן רב להגיע (רק בגרסאות הבאות)
  • שיפורים מתרחשים בצעדים לא הדרגתיים
  • יצירת גרסה חדשה מופעלת באופן ידני

התפתחות התוכנה

התפתחויות משמעותיות בתוכנה בשנים האחרונות שינו את ה-SDLC.

גורמי התפתחות מרכזיים:

מורכבות התוכנה

  • מערכות הופכות מתוחכמות ומורכבות יותר
  • מעבר למיקרו-שירותים - הרבה חלקים נעים חלקית עצמאיים
  • בסיסי קוד הופכים מורכבים
  • ניהול גרסאות, תלויות ותאימות הופך קריטי

מורכבות קוד ו-Git

  • Git מציג דרך ניתנת להרחבה וזריזה לפתח תוכנה
  • מודל הסתעפות מבוזר מאפשר פיתוח בו-זמני
  • מיזוג קוד לענף הראשי הופך לאירוע תדיר
  • האתגר הופך לניהול ותיאום החלקים
  • קוד הענף הראשי חייב להיות יציב: באג בענף הראשי מאלץ להחזיר את הקוד אחורה ולהתחיל מחדש את תהליך הבדיקה

מהפכת ה-CI/CD המודרנית

ארגוני מו"פ מסתגלים

  • ארגוני מו"פ מתאימים את עצמם לשינויים במאפייני התוכנה
  • הגדלת מספר האנשים והצוותים המעורבים בתוכנה מסוימת
  • מתודולוגיות חדשות מונעות זריזות (כמו סקראם) צוברות פופולריות
    • התמקדות בערך וגמישות
    • הסתמכות על מחזורים קצרים מקצה לקצה (פיתוח, בדיקה, שחרור)

קונטיינרים ומיקרו-שירותים

  • קונטיינרים מספקים דרכים חדשות להפוך את ה-SDLC לחסין:
    • סביבות פיתוח ובדיקה מאוחדות
    • יצירת חבילות הופכת פשוטה
    • חיכוך הפריסה מופחת (פחות תלות בסביבת ההרצה)
  • מיקרו-שירותים הופכים לגישה פופולרית: פירוק האפליקציה להרבה חלקים עצמאיים עם שחרורים נפרדים אך מתואמים

עידן הענן

  • מהפכת הענן הציגה את מודל "כשירות"
    • IaaS, PaaS: הקצאה וניהול קלים של תשתית ופלטפורמה
  • SaaS
    • תוכנה כשירות מבטלת ניהול גרסאות ושחרורים
    • תוכנה מתעדכנת ומתוקנת באופן רציף
    • לקוחות מצפים לעדכונים פרואקטיביים ללא נקיטת פעולות

Jenkins - חלוץ האוטומציה

🎩

מהו Jenkins?

  • פלטפורמת אוטומציה
  • הוצג ב-2011
  • קוד פתוח
  • ניתן להתאמה אישית והרחבה גבוהה
    • שוק של תוספים חינמיים להרחבת הפונקציונליות
    • כמעט כל דבר בפלטפורמה ניתן לשינוי

Jenkins כמריץ משימות

  • הגדרת משימות שונות למשימות שונות
  • מבנה משימות בקבוצות היררכיות
  • הקצאת קבוצות משתמשים והרשאות למשימות
  • הפעלת משימות אוטומטית על אירועים (git push, merge)
  • מעקב ובדיקה של הרצות משימות

בנייה מבוזרת

  • ארכיטקטורת מאסטר-סלייב
    • מאסטר אחד או יותר של Jenkins מנהלים את המשימות
    • משימות רצות על סלייבים שונים (שרתים)
  • שיפור ביצועים ומהירות לפרויקטים/משימות גדולים
  • אפשור סקיילביליות של מערכת ה-CI/CD

✅ יתרונות

  • רב-תכליתי

פריסת ענן ותשתית כקוד

Kubernetes (K8s)

  • תזמור קונטיינרים: הרצת אשכול של קונטיינרים
  • שוחרר על ידי גוגל ב-2014 כפרויקט קוד פתוח
  • חולל מהפכה בסביבת ההרצה המודרנית:
    • כל סביבת הרצה היא אשכול K8s
    • בנוי לסקיילביליות ותמיכה במשאבים המוקצים דינמית
    • כל ספק ענן מספק פתרון K8s מנוהל מוכן לשימוש

תהליך פריסת ענן

  • פריסה לסביבת ענן מודרנית כוללת:
    • הקצאת המשאבים הדרושים (צמתים, רשת, אחסון וכו')
    • הכנת גרסאות מעודכנות של מניפסטים של K8s
    • עדכון האשכול להרצת הגרסאות החדשות
  • תשתית כקוד הוצגה כדי להתמודד עם מורכבות התשתית החדשה

תשתית כקוד (IaC)

  • תשתית כקוד (IaC) היא הרעיון של ניהול התשתית באמצעות קוד
  • מניפסטים מתארים את דרישות ההקצאה
  • מחסנית של IaC שומרת את המצב של משאבים שנפרסו כעת
  • פרסום שינויים למניפסטים של IaC גורם לדלתא של דרישות
  • משאבים מוחלפים אוטומטית כדי לשקף את הדרישות החדשות

GitHub Actions

למה GitHub Actions?

  • האירוח המוביל לפרויקטי קוד פתוח
  • מגוון הפרויקטים מציע טווח רחב של דוגמאות CI/CD
  • פרויקטים ציבוריים נהנים מתכונות ארגוניות מתקדמות בחינם
  • תמיכה נרחבת בשימוש בפעולות וזרימות עבודה של צד שלישי
  • מסגרת פשוטה וישירה יחסית

איך GitHub Actions עובד

  • הגדרת זרימת עבודה שתופעל כאשר אירוע מתרחש במאגר שלך
  • זרימת עבודה מכילה משימה אחת או יותר (מבוצעות ברצף או במקביל)
  • משימות מבצעות צעדים ברצף, כל צעד:
    • מריץ סקריפט
    • קורא לפעולה (הרחבה לשימוש חוזר)

אירועים שמפעילים זרימות עבודה

  • דחיפת קוד לענף
  • דחיפת תג חדש
  • פתיחה/עדכון/סגירה של pull request
  • טיימר (תזמון זרימות עבודה שגרתיות)

דוגמה לזרימת עבודה

name: learn-github-actions on: [push] jobs: check-bats-version: runs-on: ubuntu-latest steps: - uses: actions/checkout@v3 - uses: actions/setup-node@v3 with: node-version: '14' - run: npm install -g bats - run: | echo "testing bats version:" bats -v

רכיבי זרימת עבודה

  • כל זרימת עבודה מוגדרת באמצעות קובץ yaml
  • הניחו קבצי yaml של זרימות עבודה תחת תיקיית .github/workflows במאגר שלכם
  • זרימת עבודה חייבת להגדיר:
    • אילו אירועים מפעילים את זרימת העבודה
    • משימה אחת או יותר
  • כל משימה רצה על קונטיינר ייעודי בתוך מכונה וירטואלית
  • משימה חייבת להגדיר:
    • את תמונת הקונטיינר לשימוש
    • צעד אחד או יותר לביצוע
