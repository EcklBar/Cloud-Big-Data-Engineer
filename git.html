מבוא לניהול גרסאות (VCS)

Version Control System היא אפליקציה המנהלת ומאחסנת כל גרסה (Revision) של הקוד והקבצים שלך.

🛡️

למה זה קריטי?

  • שיתוף פעולה בצוות על אותו קוד מקור[cite: 1553].
  • ניהול שחרורי גרסאות (Releases).
    • [cite_start]
  • יכולת "חזרה בזמן" (Rollback) במקרה של באגים[cite: 1554].
🌐

המודל המבוזר (Distributed)

בניגוד לכלים ישנים, ב-Git לכל מפתח יש עותק מלא של המאגר ושל ההיסטוריה כולה. [cite_start]זה מאפשר עבודה מהירה, גם ללא חיבור לאינטרנט[cite: 1578].

[cite_start]

אחסון דלתא (Delta Storage) [cite: 1657-1662]

Git משתמש במנגנון חכם לחיסכון במקום:

📦 קבצים מקוריים: נשמרים כ-Object שלם בפעם הראשונה.

[cite_start]

📝 שינויים: נשמרים כ-Deltas (הפרשים) בלבד בין הגרסאות[cite: 1672].

[cite_start]

📸 Commit: הוא בעצם "תמונת מצב" (Snapshot) שמצביעה למצב הפרויקט ברגע נתון[cite: 1685].

[cite_start]

סוגי אובייקטים ב-Git [cite: 1705-1714]

Binary Large Object: מכיל את התוכן הגולמי של הקובץ (ללא שם הקובץ או הרשאות). [cite_start]רק הדאטה נטו[cite: 1706].
מייצג תיקייה. [cite_start]הוא מכיל רשימה של קבצים (שמות) והפניות ל-Blobs שלהם, או הפניות ל-Trees אחרים (תתי-תיקיות)[cite: 1707].
מייצג גרסה. [cite_start]מכיל מצביע ל-Tree הראשי של הפרויקט ברגע השמירה, בתוספת מטא-דאטה: מי כתב (Author), מתי (Date), והודעת הסבר [cite: 1711-1713].

פקודות בסיס (Basic Commands)

[cite_start]

ארגז הכלים הבסיסי שכל מהנדס נתונים חייב להכיר [cite: 1618-1623].

git init # יוצר מאגר Git חדש בתיקייה הנוכחית

git clone <repo_url> # מוריד עותק מלא של מאגר קיים מהשרת

git status # מציג את מצב הקבצים (מה שונה, מה מוכן לשמירה)

git add <file> # מעביר קובץ ל-Staging Area (הכנה לשמירה)

git commit -m "message" # שומר את השינויים במאגר המקומי כגרסה חדשה

💡 טיפ למקצוענים

השתמשו ב-git diff לפני ה-Add כדי לראות בדיוק אילו שורות השתנו בקוד, וב-git log כדי לראות את היסטוריית השינויים בפרויקט.

תהליך העבודה (The 3 Stages)

[cite_start]

הבנת המבנה של Git היא המפתח למניעת טעויות. [cite: 1631-1644]

Repository
היסטוריה שמורה
➜
Staging Area
האינדקס (הכנה)
➜
Working Directory
ארגז החול (עריכה)

1. Working Directory

כאן אתם יוצרים, עורכים ומוחקים קבצים. [cite_start]זהו אזור העבודה המיידי שלכם [cite: 1633-1635].

2. Staging Area

אזור ביניים. אתם "בוחרים" אילו קבצים ששיניתם ייכנסו לגרסה הבאה. [cite_start]פקודת git add מעבירה קבצים לכאן [cite: 1636-1639].

3. Repository

בסיס הנתונים של Git. [cite_start]פקודת git commit לוקחת את מה שב-Staging ושומרת אותו לנצח בהיסטוריה [cite: 1640-1643].

[cite_start]

מצבי קבצים (File States) [cite: 1764-1769]

  • Untracked: קובץ חדש ש-Git עדיין לא מכיר.
  • Unmodified: קובץ קיים שלא שונה מאז הגרסה האחרונה.
  • Modified: קובץ קיים שבוצעו בו שינויים.
  • Staged: קובץ שסומן ומוכן ל-Commit.

ענפים (Branching & Merging)

הכוח האמיתי של Git. [cite_start]עבודה על פיצ'רים במקביל מבלי להרוס את הקוד הראשי. [cite: 1820-1826]

# יצירת ענף חדש ומעבר אליו
git checkout -b feature_branch

# הצגת כל הענפים
git branch -a

# חזרה לענף הראשי (Main/Master)
git checkout main

# מיזוג השינויים מהענף הצדדי לראשי
git merge feature_branch

אסטרטגיות מיזוג

[cite_start]

Fast-Forward Merge [cite: 1863-1865]

קורה כשהענף הראשי לא השתנה מאז יצירת הענף הצדדי. Git פשוט "מזיז את המצביע" קדימה. אין היסטוריה מפוצלת.

Merge Commit

קורה כשהייתה היסטוריה מתפצלת (גם הראשי וגם הצדדי השתנו). Git יוצר Commit מיוחד שמאחד את שני הענפים.

מה קורה כששני אנשים משנים את אותה שורה ב-אותו קובץ? [cite_start]Git עוצר ומבקש עזרה. [cite: 1861-1862]

[cite_start]

זיהוי קונפליקט בקוד [cite: 1874-1885]

<<<<<<< HEAD # הקוד שנמצא כרגע ב-Main print("Hello World") ======= # הקוד החדש מהענף הצדדי (my_side_branch) print("Hello Git") >>>>>>> my_side_branch

איך פותרים?

  1. פותחים את הקובץ.
  2. בוחרים איזו גרסה להשאיר (או משלבים את שתיהן).
  3. מוחקים את הסימונים של Git (<<<<, ====, >>>>).
    4. [cite_start]
  4. מבצעים git add לקובץ המתוקן[cite: 1892].
    5. [cite_start]
  5. מבצעים git commit כדי לסיים את המיזוג[cite: 1894].

המלצה חמה: השתמשו ב-VSCode. [cite_start]הוא מציג כפתורים נוחים כמו "Accept Incoming Change" שהופכים את התהליך לקל ופשוט. [cite: 1897-1903]

מעבדת תרגול (Hands-on Labs)

אין דרך טובה יותר ללמוד מאשר לכתוב פקודות בעצמכם.

[cite_start]

📝 תרגיל 1: יצירת מאגר [cite: 1952-1956]

  1. צרו תיקייה חדשה ופתחו בה טרמינל.
  2. אתחלו מאגר חדש: git init.
  3. צרו קובץ בשם name.txt וכתבו בו את שמכם.
  4. העבירו את הקובץ ל-Staging: git add name.txt.
  5. בצעו Commit ראשון: git commit -m "Initial commit".
[cite_start]

🌿 תרגיל 2: עבודה עם ענפים [cite: 1959-1962]

  1. צרו ענף חדש: git checkout -b side_branch.
  2. שנו את התוכן של name.txt לשם של חבר.
  3. בצעו Stage ו-Commit לשינוי בענף החדש.
  4. השתמשו ב-git log ו-git status כדי לראות את השינוי.
[cite_start]

🔀 תרגיל 3: מיזוג וקונפליקטים [cite: 1970-1975]

  1. חזרו לענף הראשי: git checkout main.
  2. נסו למזג: git merge side_branch.
  3. שימו לב! אם שיניתם את הקובץ גם ב-Main, תקבלו קונפליקט.
  4. פתרו את הקונפליקט כך ששני השמות יופיעו בקובץ.
  5. סיימו את המיזוג עם Commit.
