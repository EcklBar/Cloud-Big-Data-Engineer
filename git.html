Introduction to Git

Version Control System (VCS) is software that manages and tracks every revision of your code and files.

Why is it Critical?

  • Collaborate on the same source code with a team.
  • Manage software releases.
  • Rollback capabilities: Revert to previous versions if bugs are discovered.

Distributed Model

Unlike older tools (like SVN), in Git, every developer has a full copy of the repository and its entire history. This allows for fast, offline work.

Core Concepts: Delta Storage & Objects

Git uses a smart mechanism to store changes efficiently:

📦 Delta Storage

Git doesn't save full copies of every file in every version. It saves the Deltas (Changes) between versions to save space and improve performance.

🧩 Git Objects

Blob (Binary Large Object): Contains the raw content of the file (binary). It has no metadata (no filename, no timestamps).
Tree: Represents a directory. It contains a list of filenames and pointers to Blobs or other Trees (sub-directories).
Commit: A snapshot of the project. It points to the main Tree and contains metadata: Author, Date, and a Message explaining the change.

The 3-Stage Workflow

Understanding these three stages is the key to mastering Git.

Working Directory Your Sandbox (Editing)
➜
Staging Area Index (Preparation)
➜
Repository Database (History)

1. Working Directory

This is where you create, edit, and delete files. These changes are local and untracked until you act on them.

2. Staging Area (Index)

A middle ground. You "select" which modified files will be part of the next version. The git add command moves files here.

3. Repository

The permanent database. The git commit command takes everything from Staging and saves a snapshot into history.

File States

  • Untracked: New file, Git doesn't know it yet.
  • Unmodified: Existing file, no changes since last commit.
  • Modified: Existing file with changes, not yet staged.
  • Staged: File marked to go into the next commit.

Commands Cheat Sheet

The essential toolkit for Data Engineers.

Setup & Configuration

git init # Initialize new repo locally git clone <url> # Download repo from server git config --global user.name "Your Name"

Daily Work

git status # Check file status git add <file> # Move to Staging git add . # Stage ALL changes git commit -m "message" # Save snapshot git diff # See changes (before add)

History & Navigation

git log # View commit history git log --oneline # Compact history git checkout <hash> # Travel back in time git blame <file> # See who changed what

Syncing

git fetch # Download changes (no merge) git pull # Fetch + Merge (Update local) git push # Upload commits to server

Branching Strategies

Branches allow you to work on features in parallel without affecting the main code.

# Create and switch to a new branch git checkout -b feature_branch
# List all branches git branch -a
# Switch back to main git checkout main
# Merge the feature into main git merge feature_branch

Merge Types

🚀 Fast-Forward Merge

Occurs when the main branch has not changed since you created your side branch. Git simply moves the pointer forward. The history remains linear.

🔀 Merge Commit (Recursive)

Occurs when there is a divergent history (both Main and Feature branches advanced). Git creates a special "Merge Commit" that ties the two histories together.

Solving Conflicts

What happens when two people change the same line in the same file? Git pauses and asks for help.

1. Identify the Conflict

Git injects markers into your file:

<<<<<<< HEAD
print("Hello World") // Code currently in Main
=======
print("Hello Git") // Code from feature_branch
>>>>>>> feature_branch

2. How to Resolve?

Manual Resolution

  1. Open the file.
  2. Delete the markers (<<<<, ====, >>>>).
  3. Decide which code to keep (or combine both).
  4. Save the file.
  5. Run git add <file>.
  6. Run git commit to finish.

Using VSCode (Recommended)

VSCode detects conflicts and provides clickable buttons above the code:

  • Accept Current Change: Keep what is in Main.
  • Accept Incoming Change: Take the code from the branch.
  • Accept Both Changes: Keep both lines.

Hands-on Labs

Open your terminal and follow these exercises.

📝 Part 1: Repository Basics

  1. Create a new folder and open terminal inside it.
  2. Initialize a repo: git init.
  3. Create a file named name.txt and write your name inside.
  4. Stage the file: git add name.txt.
  5. Commit the file: git commit -m "Initial commit".

🌿 Part 2: Branching

  1. Create a new branch: git checkout -b side_branch.
  2. Change the text in name.txt to a friend's name.
  3. Stage and Commit the change on the new branch.
  4. Check history using: git log and git status.

🔀 Part 3: Merging & Conflicts

  1. Switch back to main: git checkout main.
  2. Action: Change name.txt in Main as well (to create a conflict). Commit it.
  3. Attempt to merge: git merge side_branch.
  4. Conflict! Open the file and resolve it so both names appear.
  5. Finalize: git add name.txt then git commit.
  6. Verify with git log --graph --oneline --all.
