Introduction to Version Control (VCS)

Version Control System is an application that manages and stores every revision of your code and files.

🛡️

Why is it Critical?

  • Collaborate on the same source code within a team.
  • Manage software releases.
  • Rollback: Revert to previous versions if bugs are discovered.
🌐

Distributed Model

Unlike older tools (like SVN), in Git, every developer has a full copy of the repository and its entire history. This allows for fast, offline work.

Git Advantages & Challenges

✅

Advantages

  • Distributed development: Work on your local copy
  • High performance: Fast operations
  • Ease of use: Intuitive commands once learned
  • Strong branching support: Excellent history management
  • Extensive tools: Advanced operations made easy
⚠️

Challenges

  • New mental model: Different approach to version control
  • Learning curve: Takes time to master
  • Destructive operations: Some commands can lose data if misused
  • Binary file limitations: Not ideal for large binary files

Delta Storage Mechanism

Git uses a smart mechanism to save space:

📦 Original Files: Stored as a complete object the first time.

📝 Changes: Stored as Deltas (differences) between versions.

📸 Commit: A "snapshot" of the project state at a given moment.

Git Objects Types

Binary Large Object: Contains the raw content of the file (binary). It has no metadata (no filename, no permissions). Just pure data.
Represents a Directory. It contains a list of filenames and pointers to their Blobs, or pointers to other Trees (sub-directories).
Represents a Version. Contains a pointer to the main Tree of the project at that moment, plus metadata: Author, Date, and a Message.
A Label associated with a commit, for example, a version number (v1.0, v2.0). Tags are used to mark specific points in history as important.

Basic Commands

The essential toolkit every Data Engineer needs to know.

git init # Initialize a new repository in the current directory
git clone <repo_url> # Download a full copy of an existing remote repository
git status # Check file status (what changed? what is staged?)
git add <file> # Move file to Staging Area (prepare for commit)
git commit -m "message" # Save changes to the local repository history

💡 Pro Tip

Use git diff before adding files to see exactly what lines changed, and git log to view the full project history.

The Workflow (The 3 Stages)

Understanding Git's structure is key to preventing errors.

1. Working Directory

Where you create, edit, and delete files. This is your immediate workspace.

2. Staging Area

A middle ground. You "choose" which modified files go into the next version. git add moves files here.

3. Repository

The database. git commit takes everything from Staging and saves it forever in history.

File States

  • Untracked: New file Git doesn't know yet.
  • Unmodified: Existing file that hasn't changed.
  • Modified: Existing file with changes.
  • Staged: File marked ready for the next Commit.
  • Committed: Changes safely saved in the local database.

Branching & Merging

Git's superpower. Work on features in parallel without breaking the main code.

# Create and switch to a new branch
git checkout -b feature_branch
# List all branches
git branch -a
# Switch back to main branch
git checkout main
# Merge side branch into main
git merge feature_branch

Merge Strategies

Fast-Forward Merge

Happens when the main branch has not changed since you created the side branch. Git simply "moves the pointer" forward. No divergent history.

git merge --ff-only feature_branch # Only merge if fast-forward is possible

Merge Commit

Happens when there is divergent history (both main and side branches advanced). Git creates a special Merge Commit joining both branches.

git merge --no-ff feature_branch # Always create a merge commit

Squash Merge

Combines all commits from the feature branch into a single commit in the target branch. Great for clean history.

git merge --squash feature_branch
git commit -m "Complete feature"

Octopus Merge

Merge more than two branches simultaneously. Rarely used, requires no conflicts.

git merge branch1 branch2 branch3

Managing Conflicts

What happens when two people change the same line in the same file? Git stops and asks for help.

Identifying a Conflict

<<<<<<< HEAD
# Code currently in Main
print("Hello World")
=======
# Code from side branch
print("Hello Git")
>>>>>>> side_branch

How to Resolve?

  1. Open the file.
  2. Choose which version to keep (or combine manually).
  3. Delete Git markers (<<<<, ====, >>>>).
  4. Run git add on the fixed file.
  5. Run git commit or git merge --continue to finish the merge.

Recommendation: Use VSCode or another modern IDE. It shows helpful buttons like "Accept Incoming Change", "Accept Current Change", "Accept Both Changes" making the process easy.

Using VSCode Merge Editor

🔧 VSCode Merge Resolution Steps

  1. Open the file with the conflict
  2. Click "Resolve in Merge Editor"
  3. Compare changes and resolve conflicts
  4. Select "Accept Incoming" for changes from the side branch
  5. Select "Accept Current" for changes from the main branch
  6. Manually update the result if needed
  7. Click "Complete Merge" to stage the file
  8. Run git merge --continue or use VSCode to finalize

Comprehensive Command List

Repository Management

git init # Create new local repo
git clone <url> # Download copy of remote repo
git status # Display working directory status
git remote -v # View all remotes with URLs
git remote # View all remotes
git remote add origin <url> # Add new remote repo
git remote add new_name <path> # Add new name for remote repo

Staging & Committing

git add <file> # Stage specific file
git add . # Stage all changes in current dir
git restore --staged <file> # Unstage file (same as reset for file)
git restore --staged # Unstage all files
git reset # Unstage changes
git reset --hard # Reset working dir to last commit
git commit # Create new commit (changeset) in local repo
git commit -m "message" # Commit with one-line message

Branching & Merging

git branch # List all branches
git branch <name> # Create new branch
git branch -d <name> # Delete local branch (prevents if unmerged)
git branch -D <name> # Force delete local branch (even if unmerged)
git branch -m old_name new_name # Rename branch
git checkout <branch> # Switch to another branch
git checkout -b <name> # Create & switch to new branch (fails if exists)
git checkout -B <name> # Create or reset existing branch to current HEAD
git merge <branch> # Merge branch into current branch
git merge <branch> --ff-only # Only merge if fast-forward possible
git merge --no-ff <branch> # Always create merge commit, even if fast-forward possible
git merge --squash <branch> # Combine all commits into one without merge commit
git merge -s octopus <branch1> <branch2> ... # Merge multiple branches with single merge commit

Cherry-pick

git cherry-pick <commit-hash> # Apply commit from another branch
git cherry-pick <hash1> <hash2> # Apply multiple commits
git cherry-pick --continue # Continue after resolving conflicts
git cherry-pick --abort # Cancel cherry-pick process

History & Information

git log # Display commit history in local repo
git log --oneline # Compact commit history
git log --oneline --graph --all # Compact history with visual graph of all branches
git log --since="YYYY-MM-DD" --until="YYYY-MM-DD" # Show commit history in date range
git show <commit> # Display details of specific commit
git diff # Compare changes (local, staged, or between commits)
git diff --staged # Compare staged files to last commit
git blame <file> # Show who changed each line in file
git tag <tag-name> # Create local tag for current commit to mark important version

Undoing Changes

git checkout <file> # Restore file to last commit state
git revert <commit> # Create new commit that undoes changes from specific commit
git revert <commit> --no-edit # Revert without opening text editor
git revert <commit> -n # Revert without creating new commit
git reset HEAD <file> # Remove specific file from stage
git reset --soft HEAD~1 # Undo last commit but keep changes staged
git reset --mixed HEAD~1 # Undo last commit, keep changes in working dir but not staged
git reset --hard HEAD~1 # Undo last commit and discard changes

Advanced Commands

Power tools for advanced Git users.

Rebase

git rebase main # Rebase current branch onto main
git rebase -i master # Interactive rebase of current branch onto master
# Allows you to edit, combine, or remove commits
# Interactive rebase commands:
# p, pick = use commit as is
# r, reword = use commit, but edit commit message
# e, edit = use commit, but stop for amending
# s, squash = combine commit into previous (keep both messages)
# f, fixup = combine commit into previous (discard this message)
# x, exec = run shell command during rebase
# d, drop = remove commit completely

Bisect

git bisect # Start binary search to find bug
git bisect start # Start bisect session
git bisect bad # Mark current commit as bad
git bisect good <commit-hash> # Mark known good commit
git bisect run <script_name> # Run automated test with script
git bisect reset # End bisect and return to original state

Filter Repo

git install git-filter-repo # Install filter-repo tool
git filter-repo --path secret.txt --invert-paths # Remove specific file from history
git filter-repo --path config.ini --invert-paths --force # Force remove file from history

⚠️ Warning: This rewrites Git history. Use with extreme caution!

Reflog (Recovery)

git reflog # Display reference log
git reflog show # Show reflog details
git reflog expire # Clean old reflog entries
git reflog --since # Display reflog entries from specific time
git reflog --until # Display reflog entries until specific time

Stash

git stash # Save changes temporarily
git stash list # List all stashes
git stash pop # Apply & remove last stash
git stash apply # Apply last stash (keep it)
git stash drop # Delete last stash

Worktree

git worktree # Manage multiple working directories
git worktree add <path> <branch> # Create new worktree at path for branch
git worktree list # Display all linked worktrees
git worktree remove <path> # Remove worktree

Git Submodules

git submodule # Manage submodules
git submodule add <repository_link|dir> <path> # Add new submodule
git submodule list # Display submodules
git submodule status # Show status (commit + sync state) of submodules
git submodule update --init # Initialize and fetch submodules
git submodule update --init --recursive # Initialize and update all nested submodules
git submodule update --init <path_to_submodule> # Initialize and update specific submodule
git submodule deinit <submodule_name> # Unregister submodule
git rm <path> # Remove submodule from repository

Hands-on Labs

The best way to learn is by doing. Open your terminal.

📝 Lab 1: Local Repo

  1. Create a new folder and open terminal.
  2. Initialize repo: git init.
  3. Create file name.txt with your name.
  4. Stage it: git add name.txt.
  5. Commit: git commit -m "Initial commit".

🌿 Lab 2: Branching

  1. Create new branch: git checkout -b side_branch.
  2. Change content of name.txt to a friend's name.
  3. Stage and Commit changes on the new branch.
  4. Check history: git log and git status.

🔀 Lab 3: Merging & Conflicts

  1. Switch back to main: git checkout main.
  2. Change name.txt on main branch too (create a conflict).
  3. Stage and commit the change.
  4. Attempt merge: git merge side_branch.
  5. Notice: You get a conflict!
  6. Resolve conflict so both names appear.
  7. Stage resolved file and finish merge with git commit.

🍒 Lab 4: Cherry-pick

  1. Create a new branch: git checkout -b experimental.
  2. Make several commits with different changes.
  3. Switch to main: git checkout main.
  4. Use git log --oneline experimental to find a commit hash.
  5. Cherry-pick one specific commit: git cherry-pick <hash>.
