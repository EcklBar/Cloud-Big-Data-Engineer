Git

Data Engineer's Guide

Introduction

Version Control System (VCS) manages and stores revisions of your code.

Why is it Critical?

  • Collaboration on source code.
  • Release management.
  • Rollback: Revert to previous versions if bugs occur.

Distributed Model

Unlike SVN, in Git every developer has a full copy of the repository history. Allows offline work and speed.

Core Concepts

📦 Delta Storage

Git stores changes (deltas) rather than full file copies for every version to save space.

🧩 Git Objects

  • Blob: Raw file content (binary).
  • Tree: Represents a directory structure.
  • Commit: Snapshot with metadata (Author, Date).
  • Tag: Label for a specific commit (v1.0).

Basic Commands

Setup

git init # Initialize new repo
git clone <url> # Download repo
git remote -v # View remotes
git remote add origin <url>

Daily Work

git status # Check file status
git add <file> # Stage file
git commit -m "msg" # Save snapshot
git push # Upload to server

The 3-Stage Workflow

Working Directory
➜
Staging Area
➜
Repository

File States

  • Untracked: New files.
  • Modified: Changed but not staged.
  • Staged: Ready for commit.
  • Committed: Saved in history.

Actions

  • git add: Working -> Staging
  • git commit: Staging -> Repo
  • git checkout: Repo -> Working

Branching Strategies

Explore different merge types interactively.

git checkout -b feature # Create & Switch
git branch # List branches
git checkout main # Switch back
git merge feature # Merge feature into main

Advanced Command Toolkit

Powerful tools for history rewriting and debugging.

Rebase (Interactive)

Rewrite commit history for a clean log.

git rebase -i master

# Options inside editor:
p, pick = use commit
r, reword = edit message
s, squash = melt into previous
d, drop = remove commit

Bisect (Bug Hunting)

Binary search to find which commit introduced a bug.

git bisect start
git bisect bad # Current is broken
git bisect good <commit> # Last known good
git bisect reset # Finish

Filter Repo

Remove sensitive data (passwords) from history.

git filter-repo --path secrets.txt --invert-paths

Reflog

A safety net. Shows ALL head movements, even deleted commits.

git reflog
git reset --hard HEAD@{1}

Worktree

Manage multiple working trees attached to the same repo.

git worktree add ../new-folder branch
git worktree list

Submodules

Repositories inside repositories.

git submodule add <url>
git submodule update --init --recursive

Solving Conflicts

When two people change the same line, Git stops. Here is how to fix it.

Identifying Conflict

<<<<<<< HEAD
print("Code in Main")
=======
print("Code in Feature")
>>>>>>> feature_branch

Manual Resolution

  1. Open the file.
  2. Delete markers (<<<<, ====, >>>>).
  3. Keep the correct code.
  4. git add <file>
  5. git commit to finish.

✨ Pro Tip: Use VSCode

VSCode provides clickable buttons: Accept Current | Accept Incoming. This is safer and faster than manual editing.

Hands-on Exercises

📝 Lab 1: Local Repo

  1. Initialize new repo: git init
  2. Create name.txt with your name.
  3. Stage it: git add name.txt
  4. Commit: git commit -m "Add name"

🌿 Lab 2: Branching

  1. Create branch: git checkout -b side_branch
  2. Change name in name.txt.
  3. Stage & Commit.
  4. Check history: git log --graph

🔀 Lab 3: Merging & Conflicts

  1. Switch to main: git checkout main
  2. Merge: git merge side_branch
  3. Challenge: If you changed main too, resolve the conflict so both names appear.
  4. Finalize with git add and git commit.
