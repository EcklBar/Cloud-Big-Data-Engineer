מבוא לניהול גרסאות (VCS)

Version Control System היא מערכת המנהלת ומאחסנת כל גרסה (Revision) של הקוד והקבצים שלך.

🛡️ למה זה קריטי?

  • שיתוף פעולה בצוות על אותו קוד מקור.
  • ניהול שחרורי גרסאות (Releases).
  • יכולת "חזרה בזמן" (Rollback) במקרה של באגים.

🌐 המודל המבוזר (Distributed)

בניגוד לכלים ישנים (כמו SVN), ב-Git לכל מפתח יש עותק מלא של המאגר ושל ההיסטוריה כולה. זה מאפשר עבודה מהירה, גם ללא חיבור לאינטרנט.

מושגי יסוד: Delta Storage ו-Objects

Git משתמש במנגנון חכם לשמירת שינויים ולניהול ההיסטוריה:

📦 אחסון דלתא (Delta Storage)

במקום לשמור עותקים מלאים של כל קובץ בכל גרסה, Git שומר את ה-שינויים (Deltas) בין הגרסאות. זה חוסך מקום ומייעל את הביצועים.

🧩 אובייקטים ב-Git

  • Blob: התוכן הגולמי של הקובץ (בינארי, ללא שם הקובץ).
  • Tree: מייצג תיקייה. מכיל שמות קבצים והפניות ל-Blobs או ל-Trees אחרים.
  • Commit: "תמונת מצב" (Snapshot) של הפרויקט. מכיל הפניה ל-Tree הראשי, שם המחבר, תאריך והודעת הסבר.

פקודות בסיס (Basic Commands)

ארגז הכלים הבסיסי להתחלת עבודה עם Git.

git init # אתחול מאגר (Repository) חדש בתיקייה הנוכחית
git clone <repository_url> # הורדת עותק של מאגר קיים מהשרת
git status # בדיקת מצב הקבצים (מה שונה? מה מוכן לשמירה?)
git add <file> # העברת קובץ ל-Staging Area (הכנה לשמירה)
git commit -m "message" # יצירת גרסה חדשה (Commit) במאגר המקומי
git log # צפייה בהיסטוריית הגרסאות

תיקיית העבודה (הקבצים שלך)

עריכה / יצירה

מצבי קבצים (File States)

  • Untracked: קובץ חדש ש-Git עדיין לא מכיר.
  • Unmodified: קובץ קיים שלא שונה מאז הגרסה האחרונה.
  • Modified: קובץ קיים שבוצעו בו שינויים אך טרם הוכנו לשמירה.
  • Staged: קובץ שסומן ומוכן להיכנס ל-Commit הבא.

ענפים (Branching & Merging)

הכוח האמיתי של Git. עבודה על פיצ'רים במקביל מבלי להרוס את הקוד הראשי.

למה לעבוד עם ענפים?

דמיינו את הקוד כעץ. כל ענף הוא קו התפתחות נפרד. אתם יכולים לעבוד על "ענף צדדי" לפיתוח פיצ'ר חדש, ורק כשהוא מוכן - למזג אותו חזרה לענף הראשי (Main/Master).

פקודות ענפים

git branch <name> // יצירת ענף git checkout <name> // מעבר לענף git merge <name> // מיזוג ענף לנוכחי

סוגי מיזוגים (Merge Types)

🚀 Fast-Forward Merge

מתרחש כאשר הענף הראשי לא השתנה מאז שיצרתם את הענף הצדדי. Git פשוט "מזיז את המצביע" קדימה. ההיסטוריה נשארת ליניארית (קו ישר).

🔀 Merge Commit (Recursive)

מתרחש כאשר הייתה היסטוריה מתפצלת (גם הראשי וגם הצדדי התקדמו). Git יוצר Commit מיוחד שמאחד את שני הענפים ומשמר את ההיסטוריה המפוצלת.

ניהול התנגשויות (Merge Conflicts)

מה קורה כששני אנשים משנים את אותה שורה ב-אותו קובץ? Git עוצר את המיזוג ומבקש עזרה.

זיהוי קונפליקט בקוד

Git יסמן את האזור הבעייתי בקובץ בצורה כזו:

<<<<<<< HEAD # הקוד שנמצא כרגע בענף הראשי (Main) print("Hello World") ======= # הקוד החדש מהענף הצדדי (my_side_branch) print("Hello Git") >>>>>>> my_side_branch

איך פותרים?

  1. פותחים את הקובץ (רצוי ב-VSCode).
  2. מחליטים איזו גרסה נכונה (או משלבים את שתיהן ידנית).
  3. מוחקים את הסימונים של Git (<<<<, ====, >>>>).
  4. מבצעים git add לקובץ המתוקן.
  5. מבצעים git commit כדי לסיים את המיזוג.

טיפ: ב-VSCode יופיעו כפתורים נוחים מעל הקונפליקט: "Accept Incoming Change" (קבל את החדש) או "Accept Current Change" (השאר את הקיים).

מעבדת תרגול (Hands-on Labs)

בצעו את התרגילים הבאים בטרמינל שלכם כדי להטמיע את הידע.

חלק 1: יצירת מאגר (Local Exercise Part #1)

  1. צרו תיקייה חדשה ופתחו בה טרמינל.
  2. אתחלו מאגר חדש: git init.
  3. צרו קובץ בשם name.txt וכתבו בו את שמכם.
  4. העבירו את הקובץ ל-Staging: git add name.txt.
  5. בצעו Commit ראשון: git commit -m "Initial commit".

חלק 2: עבודה עם ענפים (Local Exercise Part #2)

  1. צרו ענף חדש בשם side_branch ועברו אליו:
    git checkout -b side_branch.
  2. שנו את התוכן של name.txt לשם של האדם שלידכם.
  3. בצעו Stage ו-Commit לשינוי בענף החדש.
  4. השתמשו ב-git log, git status ו-git show כדי לראות את השינוי.

חלק 3: מיזוג וקונפליקטים (Local Exercise Part #3)

  1. חזרו לענף הראשי: git checkout main (או master).
  2. נסו למזג את הענף הצדדי: git merge side_branch.
  3. אם נוצר קונפליקט: פתחו את הקובץ ופתרו אותו כך ששני השמות יופיעו.
  4. לאחר הפתרון, בצעו git add name.txt וסיימו עם git commit.
  5. צפו בהיסטוריה המלאה באמצעות git log.
