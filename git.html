Introduction to Version Control (VCS)

Version Control System is an application that manages and stores every revision of your code and files.

🛡️

Why is it Critical?

  • Collaborate on the same source code within a team.
  • Manage software releases.
  • Rollback: Revert to previous versions if bugs are discovered.
🌐

Distributed Model

Unlike older tools (like SVN), in Git, every developer has a full copy of the repository and its entire history. This allows for fast, offline work.

Delta Storage Mechanism

Git uses a smart mechanism to save space:

📦 Original Files: Stored as a complete object the first time.

📝 Changes: Stored as Deltas (differences) between versions.

📸 Commit: A "snapshot" of the project state at a given moment.

Git Objects Types

Binary Large Object: Contains the raw content of the file (binary). It has no metadata (no filename, no permissions). Just pure data.
Represents a Directory. It contains a list of filenames and pointers to their Blobs, or pointers to other Trees (sub-directories).
Represents a Version. Contains a pointer to the main Tree of the project at that moment, plus metadata: Author, Date, and a Message.

Basic Commands

The essential toolkit every Data Engineer needs to know.

git init # Initialize a new repository in the current directory

git clone <repo_url> # Download a full copy of an existing remote repository

git status # Check file status (what changed? what is staged?)

git add <file> # Move file to Staging Area (prepare for commit)

git commit -m "message" # Save changes to the local repository history

💡 Pro Tip

Use git diff before adding files to see exactly what lines changed, and git log to view the full project history.

The Workflow (The 3 Stages)

Understanding Git's structure is key to preventing errors.

1. Working Directory

Where you create, edit, and delete files. This is your immediate workspace.

2. Staging Area

A middle ground. You "choose" which modified files go into the next version. git add moves files here.

3. Repository

The database. git commit takes everything from Staging and saves it forever in history.

File States

  • Untracked: New file Git doesn't know yet.
  • Unmodified: Existing file that hasn't changed.
  • Modified: Existing file with changes.
  • Staged: File marked ready for the next Commit.

Branching & Merging

Git's superpower. Work on features in parallel without breaking the main code.

[Image of git branching strategy diagram]
# Create and switch to a new branch
git checkout -b feature_branch

# List all branches
git branch -a

# Switch back to main branch
git checkout main

# Merge side branch into main
git merge feature_branch

Merge Strategies

Fast-Forward Merge

Happens when the main branch has not changed since you created the side branch. Git simply "moves the pointer" forward. No divergent history.

Merge Commit

Happens when there is divergent history (both main and side branches advanced). Git creates a special Merge Commit joining both branches.

Managing Conflicts

What happens when two people change the same line in the same file? Git stops and asks for help.

Identifying a Conflict

<<<<<<< HEAD # Code currently in Main print("Hello World") ======= # Code from side branch print("Hello Git") >>>>>>> side_branch

How to Resolve?

  1. Open the file.
  2. Choose which version to keep (or combine manually).
  3. Delete Git markers (<<<<, ====, >>>>).
  4. Run git add on the fixed file.
  5. Run git commit to finish the merge.

Recommendation: Use VSCode. It shows helpful buttons like "Accept Incoming Change" making the process easy.

Comprehensive Command List

Repository Setup

git init # Create new repo
git clone <url> # Clone existing repo
git remote -v # View remotes
git remote add origin <url> # Add remote

Staging & Committing

git status # Check status
git add . # Stage all changes
git restore --staged <file> # Unstage file
git commit -m "msg" # Commit changes
git reset --hard # Discard all changes

Branching & Merging

git branch # List branches
git checkout -b <name> # Create & switch
git branch -D <name> # Force delete branch
git merge <branch> # Merge into current
git cherry-pick <commit> # Apply specific commit

History & Inspection

git log --oneline # Compact history
git log --graph # Visual history
git diff # View changes
git blame <file> # See who changed lines
git tag v1.0 # Tag a version

Advanced / Recovery

git revert <commit> # Undo commit (new commit)
git reflog # View reference logs (recovery)
git stash # Save changes temporarily
git rebase -i # Interactive rebase

Hands-on Labs

The best way to learn is by doing. Open your terminal.

📝 Lab 1: Local Repo

  1. Create a new folder and open terminal.
  2. Initialize repo: git init.
  3. Create file name.txt with your name.
  4. Stage it: git add name.txt.
  5. Commit: git commit -m "Initial commit".

🌿 Lab 2: Branching

  1. Create new branch: git checkout -b side_branch.
  2. Change content of name.txt to a friend's name.
  3. Stage and Commit changes on the new branch.
  4. Check history: git log and git status.

🔀 Lab 3: Merging & Conflicts

  1. Switch back to main: git checkout main.
  2. Attempt merge: git merge side_branch.
  3. Notice: If you changed the file in Main too, you get a conflict.
  4. Resolve conflict so both names appear.
  5. Finish merge with Commit.
