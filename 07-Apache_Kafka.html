🎯 מבוא ל-Apache Kafka

מהו Apache Kafka?

Apache Kafka הוא פלטפורמה מבוזרת לסטרימינג נתונים המבוססת על מודל pub-sub (publish-subscribe).

Kafka הוא מהיר להפליא, סקלבילי, עמיד לתקלות ויכול לשמש כפלטפורמת נתונים מרכזית.

למה Kafka?

בעידן המודרני, ארגונים צריכים לעבד נתונים בזמן אמת:

⚡ Real-Time

עיבוד נתונים ברגע שהם נוצרים, לא בבאצ'ים

📈 סקלביליות

התמודדות עם מיליוני events בשנייה

🔒 Fault Tolerance

המערכת ממשיכה לעבוד גם בתקלות

🌐 מבוזר

חלוקת עומס על מספר שרתים

הבעיה שKafka פותר

🔄 מערכות ישנות (ETL)

  • רצות בזמנים מוגדרים (לא real-time)
  • מורכבות בהתמודדות עם נתונים רבים
  • קשות לסקלביליות
  • עלויות תפעול גבוהות

✨ הפתרון: Data Streaming עם Kafka

  • מערכת עצבים מרכזית לכל הנתונים בארגון
  • Message bus סקלבילי בזמן אמת
  • Source of Truth - מקור אמת יחיד לכל הנתונים
  • מזין מערכות: Databases, Data Warehouses, Hadoop, NoSQL ועוד

⚡ ETL vs Data Streaming

ETL - העולם הישן

⚠️ מגבלות ETL מסורתי:

  • התפתח בשנות ה-90, בעיקר לניתוח נתונים
  • רץ בזמנים מוגדרים - לא בזמן אמת
  • מורכב ב-incremental capturing
  • רץ על databases יחידים (Operational / DWH)
  • מתאים בעיקר ל-schematic data
  • קשה מאוד לסקלביליות בעומסים גבוהים
  • עלות תפעולית גבוהה - איטי ודורש משאבים
  • תמיכה מוגבלת או אפסית ב-real-time

Data Streaming - העולם החדש

✅ דרישות של העידן המודרני:

  • Fault Tolerance - עמידות בפני תקלות
  • Parallel Processing - עיבוד מקבילי
  • Low Latency - זמן תגובה נמוך | Real-Time
  • תמיכה בנתונים schematic ו-non-schematic
  • קל לתפעול וניטור
  • פתרון reusable מלא ולא רק כלי בודד

🔧 רכיבים עיקריים ב-Kafka

📤 Producer

אפליקציה ששולחת (מפרסמת) messages ל-Kafka topics

  • יכול לבחור ל-איזה partition לשלוח
  • תומך ב-batching
  • תומך ב-compression

📥 Consumer

אפליקציה שקוראת messages מ-Kafka topics

  • שייכת ל-consumer group
  • שומרת offset
  • יכולה לקרוא במקביל

🖥️ Broker

שרת Kafka המאחסן ומנהל topics

  • חלק מ-cluster
  • מנהל partitions
  • מטפל ב-replication

🐘 ZooKeeper

מנהל את מטא-דאטה של ה-cluster

  • בוחר leader לpartitions
  • שומר קונפיגורציה
  • (בגרסאות חדשות - Kraft במקום)

📂 Topic

קטגוריה/feed name שאליו נשלחים messages

  • מחולק ל-partitions
  • ניתן ל-replication
  • כמו "טבלה" במסד נתונים

🗂️ Partition

חלוקה פיזית של topic לחלקים

  • ordered, immutable
  • כל message מקבל offset
  • מאפשר parallelism

📂 Topics & Partitions

מהו Topic?

Topic הוא stream של records. זה category או feed name שאליו נשלחים messages.

כל topic מזוהה על ידי שם ייחודי.

מאפיינים:

  • Topics הם multi-subscriber - יכולים להיות מספר consumers
  • Messages נשמרים לפרק זמן מוגדר (retention)
  • ניתן להגדיר compression (gzip, snappy, lz4)

מהו Partition?

🗂️ Partition = חלוקה של Topic

כל topic מחולק ל-partitions (אחד או יותר).

כל partition הוא ordered, immutable sequence של records.

מבנה Topic עם 3 Partitions

Topic: user-events 📦 Partition 0: [msg0] → [msg3] → [msg6] → [msg9] ... 📦 Partition 1: [msg1] → [msg4] → [msg7] → [msg10] ... 📦 Partition 2: [msg2] → [msg5] → [msg8] → [msg11] ... 💡 כל message מקבל offset ייחודי ב-partition שלו

למה Partitions?

📈 Scalability

חלוקת הנתונים על מספר servers

⚡ Parallelism

מספר consumers יכולים לקרוא במקביל

🔒 Ordering

סדר מובטח בתוך partition

💾 Distribution

פיזור העומס והאחסון

⚠️ חשוב לזכור:

  • Ordering מובטח רק בתוך partition - לא בין partitions
  • Producer בוחר ל-איזה partition לשלוח (לפי key או round-robin)
  • מספר partitions לא ניתן לשינוי בקלות - תכנן מראש!

🖥️ Brokers & Clusters

מהו Broker?

Kafka Broker הוא שרת Kafka. זה תהליך שרץ על מכונה ומנהל את אחסון ה-messages.

תפקידי ה-Broker:

  • מקבל messages מ-producers
  • מאחסן messages על הדיסק
  • משרת messages ל-consumers
  • מנהל replication של partitions
  • מטפל ב-leader election

מהו Cluster?

🌐 Kafka Cluster

אוסף של מספר Kafka Brokers שעובדים יחד כמערכת אחת.

כל broker מזוהה על ידי broker ID ייחודי (מספר שלם).

Cluster עם 3 Brokers

Broker 0
Leader: P0, P1
Follower: P2
Broker 1
Leader: P2
Follower: P0
Broker 2
Leader: P3
Follower: P1

כל partition יש לו leader ו-followers
אם broker נופל - אחר הופך ל-leader

Replication

🔄 העתקה לגיבוי

כל partition יכול להיות replicated על מספר brokers.

  • Replication Factor קובע כמה עותקים
  • יש leader partition אחד - מטפל בכל הקריאות והכתיבות
  • יש follower partitions - רק מעתיקים נתונים
  • אם ה-leader נופל, אחד מה-followers הופך ל-leader

⚙️ Replication Factor מומלץ:

לפחות 3 בפרודקשן (מאפשר אובדן של עד 2 brokers)

הנתונים יכולים להיות מעובדים כדי להפיק ערכים.

📤 Kafka Producers

מהו Producer?

Producer הוא אפליקציה שיוצרת ושולחת (מפרסמת) records ל-Kafka topics.

איך Producer עובד?

  1. Producer יוצר record (key, value, timestamp)
  2. הSerializer ממיר את הkey והvalue ל-bytes
  3. הPartitioner בוחר לאיזה partition לשלוח
  4. Record מתווסף ל-batch
  5. Batch נשלח לBroker
  6. Broker מחזיר אישור (ack)

Partitioning Strategy

🔑 עם Key

אם record כולל key:

  • Hash של הkey קובע partition
  • אותו key → תמיד אותו partition
  • מבטיח ordering לפי key

🎲 בלי Key

אם אין key:

  • Round-robin בין partitions
  • חלוקה שווה של העומס
  • אין guarantee על ordering

🎯 Custom Partitioner

ניתן ליישם לוגיקה משלך:

  • חלוקה לפי ערך בmessage
  • חלוקה לפי timestamp
  • חלוקה לפי גיאוגרפיה

Acknowledgment Modes

⚙️ הגדרת acks

  • acks=0 - אין המתנה לאישור (מהיר, לא בטוח)
  • acks=1 - המתן לאישור מה-leader (ברירת מחדל)
  • acks=all - המתן לאישור מכל ה-replicas (הכי בטוח, הכי איטי)

אופטימיזציה של Producer

✨ Best Practices:

  • Batching - שליחת messages בקבוצות (batch.size, linger.ms)
  • Compression - דחיסה (gzip, snappy, lz4, zstd)
  • Idempotence - מניעת duplicates (enable.idempotence=true)
  • Max In-Flight - כמות בקשות במקביל (max.in.flight.requests.per.connection)

📥 Kafka Consumers

מהו Consumer?

Consumer הוא אפליקציה שקוראת records מ-Kafka topics.

Consumer Groups

👥 מהו Consumer Group?

קבוצה של consumers שעובדים יחד כדי לצרוך topic.

  • כל partition נצרך על ידי consumer אחד בלבד בgroup
  • אם יש יותר consumers מאשר partitions - חלקם יישארו idle
  • אם consumer נופל - ה-partitions שלו יועברו לאחרים (rebalancing)

דוגמה: Topic עם 4 Partitions

Consumer Group: analytics-group 📊 Topic: user-events (4 partitions) 👤 Consumer 1 → 📦 Partition 0, 📦 Partition 1 👤 Consumer 2 → 📦 Partition 2, 📦 Partition 3 ✓ כל partition נצרך פעם אחת ✓ עומס מחולק בין 2 consumers

Offsets

🔢 מהו Offset?

Offset הוא מספר ייחודי שמזהה כל message ב-partition.

  • Consumer שומר את ה-offset האחרון שקרא
  • מאפשר לresume מהמקום האחרון
  • שמור ב-topic מיוחד: __consumer_offsets

Rebalancing

⚠️ מהו Rebalancing?

תהליך שבו partitions מחולקים מחדש בין consumers בgroup.

קורה כאשר:

  • Consumer מצטרף לgroup
  • Consumer יוצא מהgroup (crash או shutdown)
  • נוספו partitions ל-topic

במהלך rebalancing: הgroup לא יכול לקרוא messages!

✨ Kafka Best Practices

🏗️ תכנון ועיצוב

📂 Topics & Partitions

  • מספר partitions: התחל עם ~10-30 partitions לפי throughput צפוי
  • אל תפריז: יותר מדי partitions = overhead על הcluster
  • תכנן מראש: קשה לשנות מספר partitions אחר כך
  • Key design: בחר key שמבטיח חלוקה שווה בין partitions
  • Retention: הגדר retention policy הגיוני (זמן או גודל)

🖥️ Cluster Sizing

  • מינימום 3 brokers בפרודקשן (for replication)
  • Replication factor = 3 (מאפשר אובדן של 2 brokers)
  • בדוק throughput צפוי: כמה MB/sec אתה צריך?
  • דיסק מהיר: SSD מומלץ לlatency נמוך

📤 Producer Best Practices

אמינות (Durability)

  • acks=all - לדאטה קריטי
  • enable.idempotence=true - למניעת duplicates
  • max.in.flight.requests=1 - אם ordering קריטי
  • retries > 0 - נסה שוב במקרה של failure

ביצועים (Performance)

  • batching: הגדל batch.size ו-linger.ms
  • compression: השתמש ב-lz4 או snappy
  • buffer.memory: הגדל אם throughput גבוה
  • partitioning: השתמש ב-key לbalancing טוב

📥 Consumer Best Practices

אמינות

  • Manual commit: commit רק אחרי עיבוד מוצלח
  • isolation.level=read_committed - לexactly-once
  • Handle rebalancing: commit לפני rebalance
  • Error handling: מה עושים עם message שנכשל?

ביצועים

  • מספר consumers = מספר partitions (למקסימום parallelism)
  • fetch.min.bytes: הגדל לבאצ'ים גדולים יותר
  • max.poll.records: כמה records לפעם
  • session.timeout.ms: לא קצר מדי (יגרום לrebalances מיותרים)

🔒 אבטחה (Security)

⚠️ אבטחה בפרודקשן

  • SSL/TLS: הצפנת תעבורה
  • SASL: authentication (Kerberos, SCRAM, OAuth)
  • ACLs: authorization - מי יכול לעשות מה
  • Encryption at rest: הצפנת דיסק
  • אל תחשוף את Kafka לאינטרנט ישירות!

📊 ניטור ותחזוקה

✅ מה לנטר?

  • Consumer lag - האם consumers מספיקים?
  • Under-replicated partitions - בעיות replication
  • Disk usage - מתקרבים למכסה?
  • CPU & Memory - bottlenecks?
  • Error rates - הרבה שגיאות?

🔧 תחזוקה שוטפת

  • Rolling restarts: עדכון brokers בזה אחר זה
  • Backup: גבה את הקונפיגורציה (לא בהכרח את הדאטה)
  • Clean up: מחק topics ישנים שלא בשימוש
  • Version upgrades: עקוב אחרי גרסאות חדשות

🎯 כללים זהב

  1. תמיד תכנן capacity מראש
  2. נטר את הcluster באופן רציף
  3. בדוק שהרשת מהירה (latency נמוך בין brokers)
  4. השתמש ב-replication factor של לפחות 3
  5. בדיקת load לפני פרודקשן
  6. אל תקח סיכונים עם דאטה קריטי - acks=all
  7. הגדר retention מתאים לצרכים שלך
  8. תעד את הconfiguration והprocess

🐍 Python Producer - פשוט

שליחת messages ל-Kafka עם kafka-python

from kafka import KafkaProducer import json from datetime import datetime # Create producer producer = KafkaProducer( bootstrap_servers=['localhost:9092'], value_serializer=lambda v: json.dumps(v).encode('utf-8') ) # Send message message = { 'user_id': 123, 'event': 'page_view', 'page': '/products', 'timestamp': datetime.now().isoformat() } future = producer.send('user-events', value=message) # Wait for confirmation record_metadata = future.get(timeout=10) print(f"Message sent to topic: {record_metadata.topic}") print(f"Partition: {record_metadata.partition}") print(f"Offset: {record_metadata.offset}") # Close producer producer.close()

🐍 Python Producer - עם Key

שליחת messages עם key (מבטיח partition קבוע)

from kafka import KafkaProducer import json producer = KafkaProducer( bootstrap_servers=['localhost:9092'], key_serializer=lambda k: k.encode('utf-8'), value_serializer=lambda v: json.dumps(v).encode('utf-8'), # Performance optimizations compression_type='lz4', batch_size=16384, linger_ms=10, acks='all', # Wait for all replicas retries=3 ) # Send with key (same key = same partition) user_id = 'user123' for i in range(5): message = { 'user_id': user_id, 'action': f'Action {i}', 'timestamp': datetime.now().isoformat() } producer.send( topic='user-events', key=user_id, # Key determines partition value=message ) producer.flush() # Wait for all messages producer.close()

🐍 Python Consumer

קריאת messages מ-Kafka

from kafka import KafkaConsumer import json # Create consumer consumer = KafkaConsumer( 'user-events', # Topic name bootstrap_servers=['localhost:9092'], auto_offset_reset='earliest', # Start from beginning enable_auto_commit=True, group_id='my-consumer-group', value_deserializer=lambda x: json.loads(x.decode('utf-8')) ) print("Listening for messages...") # Consume messages try: for message in consumer: print(f"\n--- New Message ---") print(f"Topic: {message.topic}") print(f"Partition: {message.partition}") print(f"Offset: {message.offset}") print(f"Key: {message.key}") print(f"Value: {message.value}") # Process the message event = message.value print(f"User {event['user_id']} did: {event['event']}") except KeyboardInterrupt: print("\nShutting down...") finally: consumer.close()

🐍 Consumer - Manual Commit

שליטה מלאה על commit (at-least-once delivery)

from kafka import KafkaConsumer consumer = KafkaConsumer( 'user-events', bootstrap_servers=['localhost:9092'], group_id='my-group', enable_auto_commit=False, # Manual commit auto_offset_reset='earliest' ) for message in consumer: try: # Process message print(f"Processing: {message.value}") # Your business logic here process_event(message.value) # Commit only after successful processing consumer.commit() print(f"✅ Committed offset: {message.offset}") except Exception as e: print(f"❌ Error processing message: {e}") # Don't commit - will reprocess on next run continue

from kafka import KafkaProducer import json from datetime import datetime import time # Create producer producer = KafkaProducer( bootstrap_servers=['localhost:9092'], key_serializer=lambda k: k.encode('utf-8'), value_serializer=lambda v: json.dumps(v).encode('utf-8'), acks='all' ) # Generate and send 10 events events = [ ('user1', 'login'), ('user2', 'login'), ('user1', 'view_product'), ('user3', 'login'), ('user1', 'add_to_cart'), ('user2', 'view_product'), ('user3', 'view_product'), ('user1', 'checkout'), ('user2', 'logout'), ('user3', 'logout') ] for user_id, event_type in events: message = { 'user_id': user_id, 'event_type': event_type, 'timestamp': datetime.now().isoformat() } future = producer.send( topic='user-events', key=user_id, value=message ) # Wait for confirmation metadata = future.get(timeout=10) print(f"✅ Sent: {event_type} for {user_id} → partition {metadata.partition}") time.sleep(0.5) # Small delay producer.close() print("\n🎉 All messages sent!")
תרגיל 5 Python Consumer - עיבוד Events

מטרה:

כתוב consumer שמעבד events ומחשב סטטיסטיקות.

דרישות:

  • קרא messages מהtopic שיצרת בתרגיל 4
  • ספור כמה events מכל סוג (login, logout, etc.)
  • הדפס סיכום כל 5 messages
  • השתמש ב-manual commit
from kafka import KafkaConsumer import json from collections import defaultdict # Create consumer consumer = KafkaConsumer( 'user-events', bootstrap_servers=['localhost:9092'], group_id='stats-group', auto_offset_reset='earliest', enable_auto_commit=False, value_deserializer=lambda x: json.loads(x.decode('utf-8')) ) # Statistics event_counts = defaultdict(int) message_count = 0 print("📊 Processing events...\n") try: for message in consumer: event = message.value event_type = event['event_type'] user_id = event['user_id'] # Count events event_counts[event_type] += 1 message_count += 1 print(f"✅ {user_id}: {event_type}") # Print stats every 5 messages if message_count % 5 == 0: print("\n--- Statistics ---") for event, count in event_counts.items(): print(f"{event}: {count}") print(f"Total: {message_count}\n") # Commit after processing consumer.commit() except KeyboardInterrupt: print("\n\n📊 Final Statistics:") for event, count in event_counts.items(): print(f"{event}: {count}") print(f"\nTotal processed: {message_count}") finally: consumer.close()

🎯 Best Practices - סיכום

Production Checklist:

  • ☐ Cluster: לפחות 3 brokers
  • ☐ Replication: RF >= 3
  • ☐ Partitions: מספר מחושב לפי throughput
  • ☐ Producer acks: all (לדאטה קריטי)
  • ☐ Consumer offset: manual commit
  • ☐ Monitoring: consumer lag, under-replicated
  • ☐ Security: SSL/SASL + ACLs
  • ☐ Retention: מוגדר לפי צורך
  • ☐ Compression: lz4/snappy enabled
  • ☐ Backup: ZooKeeper snapshots
