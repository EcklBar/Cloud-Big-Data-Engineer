מבוא ל-Git

מה זה בקרת גרסאות?

  • אוכסנת ומנהלת כל גרסה של הקבצים והקוד שלך
  • שיתוף פעולה על קוד מקור וניהול גרסאות
  • חזרה לגרסאות קודמות כאשר מתגלים באגים
  • VSC = מערכת בקרת גרסאות

אודות Git

  • נוצר על ידי לינוס טורוואלדס ב-2005 (עבור הליבה של לינוקס)
  • מחליף כלי VSC ישנים יותר (CVS, Subversion, BitKeeper, Mercurial)
  • מתמקד בהפצת קוד מקור (כל אחד יש לו עותק)
  • מיישם אחסון קבצים יעיל
  • מבטיח שלמות הקבצים

יתרונות Git

  • פיתוח מבוזר (עבודה על העותק המקומי שלך)
  • ביצועים גבוהים
  • קלות שימוש
  • תמיכה חזקה בענפים וניהול היסטוריה
  • כלים נרחבים לפעולות מתקדמות

אתגרי Git

  • מודל ניהול קוד חדש
  • עקומת למידה (שינוי חשיבה ופקודות)
  • פעולות הרסניות כאשר נעשה שימוש לא נכון
  • תמיכה מוגבלת בקבצים בינאריים
🚀 שירותי VCS

  • בקרת גישה, מעקב בעיות, תיעוד, אוטומציה
  • שירות VCS מוביל לפרויקטים קוד-פתוח וחברות
  • נרכש על ידי מיקרוסופט ב-2018

Commit ו-Push
  • קבל עותק של כל המאגר מהשרת ("clone")
  • כל התוכן וכל ההיסטוריה
  • עשה שינויים מקומית (commits וענפים)
  • דחף שינויים חזרה לשרת, הידוע כמאגר מרוחק

סוגי אובייקטים ב-Git

  • Blob: התוכן הגולמי של קובץ - ללא מטא-דטא של הקובץ
  • Tree: רמה אחת של מידע תיקיות (שמות קבצים, blobs, תת-תיקיות)
  • Commit: הגרסה של העץ + מטא-דטא (מחבר, תאריכים, הודעות לוג וכו')
  • Tag: תווית המשויכת ל-commit, למשל מספר גרסה

אחסון דלתא

  • כל קובץ נשמר כאובייקט שלם ונפרד כשהוא מוצג
  • שינויים לקובץ קיים נשמרים כדלתאות
  • Git משתמש במנגנון חכם ויעיל לאחסון שינויים
  • זה מביא לחיסכון משמעותי במקום וניהול יעיל של ההיסטוריה
git init
- אתחל מאגר Git חדש
git clone <repository_url>
- העתק מאגר Git קיים
git add <file>
- הכן קבצים (הוסף לאינדקס)
git commit -m "הודעת commit"
- רשום שינויים מוכנים
git status
- בדוק את מצב תיקיית העבודה

תיקיית עבודה

  • צור, ערוך ומחק קבצים בפרויקט שלך
  • תחשוב על זה כ"ארגז חול" שלך שבו אתה עובד על הקוד והנתונים

אזור הכנה (אינדקס)

  • רשימת בדיקה של שינויים שאתה רוצה ל-commit
  • הכן ובחר שינויים ספציפיים לכללה ב-commit הבא
  • באמצעות git add, אתה מעביר שינויים מתיקיית העבודה לאזור ההכנה

מאגר

  • המאגר מאחסן את ההיסטוריה המלאה של הפרויקט שלך
  • כאשר אתה עושה git commit, השינויים באזור ההכנה נרשמים כתמונת מצב חדשה במאגר
  • ההיסטוריה המלאה של הפרויקט שלך נשמרת כאן

מצבי קבצים

  • לא במעקב: קובץ חדש שמוצג לראשונה
  • לא שונה: קובץ קיים ללא שינויים מקומיים
  • שונה: קובץ קיים עם שינויים מקומיים
  • מוכן: קבצים עם שינויים מקומיים המסומנים ל-commit הבא
  • Committed: שינויים נשמרו בבטחה במסד הנתונים המקומי

ענפי Git

🌳 אסטרטגיית ענפים
  • התייחס לקוד המקור כעץ של ענפים
  • כל ענף הוא קבוצה ליניארית של commits
  • השתמש בענף צדדי כדי לעבוד על גרסה פרטית של הקוד
  • מזג את הענף הצדדי חזרה לענף הראשי
  • שתף פעולה עם מפתחים אחרים
git checkout branch_name
- עבור לענף
git checkout -B branch_name
- צור ועבור לענף חדש
git branch -D testing
- מחק ענף
git log --oneline --decorate --graph --all
- הצג כל הענפים
git show branch_name
- הצג ראש הענף

מיזוג ענפים

  • ברגע שמוכן, ענף צדדי נמזג לענף הראשי
  • עבור לענף הראשי: git checkout main
  • מזג את הענף הצדדי: git merge side_branch

היסטוריה מתפצלת

  • שיתוף פעולה באמצעות ענפים מרובים עלול להוביל להיסטוריה מתפצלת
  • מפתח #1 מתחיל ענף חדש ומשנה קובץ
  • מפתח #2 מוסיף commit חדש לענף הראשי וגם משנה את אותו קובץ
  • מפתח #1 לא יכול למזג כי השינויים מתנגשים

קונפליקטי מיזוג

⚡ התמודדות עם קונפליקטים
  • כאשר מופיע קונפליקט מיזוג, המיזוג מושהה כדי לתת לך לפתור אותו
  • השתמש ב-git status כדי לראות אילו קבצים יש בהם קונפליקטים
  • קבצים ללא קונפליקטים מוכנים
  • קבצים עם קונפליקטים מופיעים כ"Both modified"
דוגמת קונפליקט:
<<<<<<< HEAD
טקסט חדש בענף הראשי
=======
טקסט מהענף הצדדי
>>>>>>> my_side_branch

פתרון קונפליקטים באופן ידני

  • פתח את הקובץ שיש בו קונפליקטים
  • מחק/שנה שורות לא רלוונטיות
  • הכן את הקובץ (git add)
  • כדי לסיים את המיזוג כשכל הקונפליקטים נפתרו: git merge --continue

פתרון בסגנון

  • צפייה ופתרון קונפליקטי מיזוג ב-IDE מודרני (למשל VSCode)
  • שימוש בעורך מיזוג הוא הפרקטיקה הטובה יותר
  • ראה את השינויים בבירור רב יותר
  • בחר בקלות את הגרסה המועדפת
  • שלב קוד משני הענפים
  • ממשק אינטואיטיבי וידידותי יותר למשתמש
🔧 דוגמת VSCode למיזוג
  1. פתח את הקובץ בקונפליקט
  2. לחץ על "Resolve in Merge Editor"
  3. השווה את השינויים ופתור את הקונפליקט
  4. בחר "Accept Incoming" לבחירת שינויים מהענף הצדדי
  5. בחר "Accept Current" לבחירת שינויים מהענף הראשי
  6. עדכן באופן ידני את התוצאה אם נדרש
  7. ברגע שמוכן, לחץ "Complete Merge" כדי להכין את הקובץ
  8. הרץ git merge --continue או השתמש ב-VSCode לסיום

פקודות Git חיוניות

ניהול מאגר

git init
- צור מאגר מקומי חדש
git clone
- הורד עותק של מאגר מרוחק
git status
- הצג את המצב (אינדקס) של תיקיית העבודה
git remote -v
- רשימת כל המאגרים המרוחקים
git remote add origin <url>
- הוסף מאגר מרוחק חדש

הכנה ו-Committing

git add
- הכן קבצים (הוסף לאינדקס)
git add .
- הכן את כל השינויים בתיקייה הנוכחית
git reset
- בטל הכנת קבצים (הסר מהאינדקס)
git restore --staged <file>
- הסר קובץ מ־staging (זהה ל reset")
git reset --hard
- אפס את תיקיית העבודה ל-commit האחרון
git commit
- צור commit חדש (קבוצת שינויים) במאגר המקומי
git commit -m "message"
- Commit עם הודעה בשורה אחת

ענפים ואיחוד

git branch
- רשימת כל הענפים
git branch <name>
- צור ענף חדש
git branch -d <name>
- מחק ענף מקומי (ימנע מחיקה אם יש שינויים שלא מוזגו)
git branch -D <name>
- מחק ענף מקומי בכפייה (גם אם יש שינויים שלא מוזגו)
git checkout <branch>
- עבור לענף אחר
git checkout -b <name>
- צור ועבור לענף חדש (ייכשל אם הענף כבר קיים)
git checkout -B <name>
- צור או דרוס ענף קיים והעבר אותו ל־HEAD הנוכחי
git merge <branch>
- אחד ענף לענף הנוכחי

פעולות מרוחקות

git fetch
- הבאת מידע (קומיטים, ברנצ'ים) מהריפוזיטורי המרוחק
git pull
- משיכה ועדכון לקומיט האחרון של הברנץ' הנוכחי
git push
- העלאת הקומיטים שלך לריפוזיטורי המרוחק
git push origin <branch>
- העלאת ברנץ' ספציפי למרוחק
git push -u origin <branch>
- העלאה והגדרת ברנץ' upstream

היסטוריה ומידע

git log
- הצגת היסטוריית הריפוזיטורי המקומי (קומיטים)
git log --oneline
- היסטוריית קומיטים קומפקטית
git show <commit>
- הצגת פרטי קומיט ספציפי
git diff
- השוואת גרסאות (שינויים מקומיים, שינויים בשלב, קומיטים שונים וכו')
git blame <file>
- הצגת מי שינה כל שורה בקובץ
git tag <tag-name>
- צור תג (tag) מקומי ל־commit הנוכחי, כדי לסמן גרסה חשובה בהיסטוריה

ביטול שינויים

git checkout <file>
- שחזור קובץ למצב הקומיט האחרון
git revert <commit>
- יצירת קומיט חדש שמבטל שינויים
git reset HEAD <file>
- הוצאת קובץ ספציפי מהשלב
git reset --soft HEAD~1
- ביטול קומיט אחרון, שמירת שינויים בשלב
git reset --hard HEAD~1
- ביטול קומיט אחרון ומחיקת שינויים

תרגילי תרגול

🎯 תרגיל #1: מבוא ל-Git

השלם את המדריך הבא מתוך הספר הרשמי של git:

יסודות Git - קבלת מאגר Git →
🎯 תרגיל מקומי - חלק #1
  1. אתחל מאגר חדש
  2. צור קובץ בשם "name.txt" עם השם שלך
  3. הכן את הקובץ
  4. צור commit חדש
git init

echo "השם שלך" > name.txt

git add name.txt

git commit -m "הוסף קובץ שם"
🎯 תרגיל מקומי - חלק #2
  1. צור ענף חדש בשם "side_branch"
  2. שנה את התוכן של name.txt לשם של האדם שליד אותך
  3. הכן ו-commit את השינויים
  4. צפה בעבודה שלך באמצעות: git log, git status, git show
git checkout -b side_branch

echo "שם אחר" > name.txt

git add name.txt

git commit -m "עדכן שם"
🎯 תרגיל מקומי - חלק #3
  1. עבור לענף הראשי
  2. מזג את "side_branch" לענף הראשי
  3. פתור את הקונפליקט כך ששני השמות יופיעו ב-"name.txt"
  4. ברגע שהקונפליקט נפתר, השלם את המיזוג
  5. צפה בעץ ה-git המלא באמצעות git log
git checkout main

git merge side_branch

# פתור קונפליקטים ב-name.txt
git add name.txt

git merge --continue
