The Elastic Stack

Welcome to your interactive guide. The Elastic Stack (formerly ELK) is a set of free and open tools for data ingestion, enrichment, storage, analysis, and visualization[cite: 60].

Elasticsearch

The heart of the stack. A distributed, RESTful search and analytics engine based on Apache Lucene. It handles textual, numerical, geospatial, structured, and unstructured data[cite: 59].

Logstash

The processing pipeline. It ingests data from various sources, transforms it (ETL), and sends it to a "stash" like Elasticsearch[cite: 85].

Kibana

The window into your data. A visualization dashboard that lets you visualize data stored in Elasticsearch (charts, maps, logs)[cite: 74].

Beats

Lightweight data shippers. They sit on your servers and ship data (like logs or metrics) to Logstash or directly to Elasticsearch[cite: 98].

Why use it?

The core challenge is searching for a single phrase in billions of text files or logs[cite: 147]. Traditional relational databases (RDBMS) aren't optimized for full-text search. Elasticsearch uses an Inverted Index to solve this[cite: 218].

Elasticsearch Architecture

Elasticsearch operates in a distributed clustered environment. Understanding the topology is key to performance and reliability.

The Cluster & Nodes

  • Cluster: A collection of nodes that communicate to hold your data[cite: 337].
  • Node: A single server instance[cite: 338]. Nodes have different roles:
Node Type Responsibility
Master Controls the cluster state. Requires a minimum of 3 for high availability[cite: 356].
Data Holds indexed data and performs CRUD operations. Can be tiered into Hot (new data) and Warm (older data)[cite: 358].
Ingest Runs pipelines to transform data before indexing[cite: 360].
Coordinating Routes requests and handles the "reduce" phase of search. All nodes act as coordinating nodes by default[cite: 361].

Shards & Replicas

To handle large datasets (e.g., 1TB), indices are split into pieces called Shards[cite: 474].

Key Formula: A shard is a self-contained Lucene index.
Recommendation: If > 5M documents, use: int(expected_docs / 5M + 1)[cite: 821].
  • Primary Shard: The original horizontal partition of the index[cite: 476].
  • Replica Shard: A copy of the primary shard. Increases read performance and provides failover[cite: 508].

The Inverted Index

This is the "magic" behind the speed. Instead of storing text directly, Elastic stores a mapping of Terms → Documents.

Example:

Doc 1: "The bright blue butterfly"

Doc 2: "Under blue sky"


Inverted Index Structure:
  • blue → found in [Doc 1, Doc 2]
  • butterfly → found in [Doc 1]
  • sky → found in [Doc 2]

This allows the engine to find documents containing "blue" instantly without scanning every row[cite: 313].

Analysis

Text fields are analyzed when indexed. An analyzer has three parts[cite: 941]:

  1. Character Filter: Cleans raw text (e.g., removes HTML tags).
  2. Tokenizer: Splits text into tokens (e.g., splitting on whitespace).
  3. Token Filter: Modifies tokens (e.g., Lowercase filter, Stop word filter)[cite: 962].

Ingestion: Logstash & Beats

Beats vs. Logstash

The "L" and "B" in the stack.

  • Beats: Lightweight agents written in Go. Focused on shipping specific data (e.g., Filebeat for logs). Best for "Extract"[cite: 98, 101].
  • Logstash: Server-side processing pipeline. Can aggregate data from multiple sources, transform complex data (Grok), and output to multiple destinations. Best for "Transform & Load"[cite: 101].
