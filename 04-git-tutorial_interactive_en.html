מבוא ל-Git

מה זה בקרת גרסאות?

  • מאחסנת ומנהלת כל גרסה של הקבצים והקוד שלך
  • שיתוף פעולה על קוד מקור וניהול גרסאות
  • חזרה לגרסאות קודמות כאשר מתגלים באגים
  • VSC = מערכת בקרת גרסאות

אודות Git

  • נוצר על ידי לינוס טורוואלדס ב-2005 (עבור הליבה של לינוקס)
  • מחליף כלי VSC ישנים יותר (CVS, Subversion)
  • מתמקד בהפצת קוד מקור (כל אחד יש לו עותק)
  • מיישם אחסון קבצים יעיל

יתרונות Git

  • פיתוח מבוזר (עבודה על העותק המקומי שלך)
  • ביצועים גבוהים
  • קלות שימוש
  • תמיכה חזקה בענפים וניהול היסטוריה

אתגרי Git

  • מודל ניהול קוד חדש
  • עקומת למידה (שינוי חשיבה ופקודות)
  • פעולות הרסניות כאשר נעשה שימוש לא נכון
  • תמיכה מוגבלת בקבצים בינאריים גדולים

Commit ו-Push
  • קבל עותק של כל המאגר מהשרת ("clone")
  • כל התוכן וכל ההיסטוריה
  • עשה שינויים מקומית (commits וענפים)
  • דחף שינויים חזרה לשרת (Remote)
git init
- אתחל מאגר Git חדש
git clone <repository_url>
- העתק מאגר Git קיים
git add <file>
- הכן קבצים (הוסף לאינדקס)
git commit -m "הודעת commit"
- רשום שינויים מוכנים
git status
- בדוק את מצב תיקיית העבודה

תיקיית עבודה

  • צור, ערוך ומחק קבצים בפרויקט שלך
  • תחשוב על זה כ"ארגז חול" שלך שבו אתה עובד

אזור הכנה (אינדקס)

  • רשימת בדיקה של שינויים שאתה רוצה ל-commit
  • הכן ובחר שינויים ספציפיים באמצעות git add

מאגר

  • המאגר מאחסן את ההיסטוריה המלאה של הפרויקט שלך
  • פקודת git commit שומרת תמונת מצב חדשה

מצבי קבצים

  • Untracked: קובץ חדש שאינו במעקב
  • Modified: קובץ קיים עם שינויים
  • Staged: קבצים מוכנים ל-commit
  • Committed: נשמר בבטחה במאגר

ענפי Git

🌳 אסטרטגיית ענפים
  • כל ענף הוא קבוצה ליניארית של commits
  • השתמש בענף צדדי כדי לעבוד על גרסה פרטית
  • מזג את הענף הצדדי חזרה לענף הראשי
git checkout -b branch_name
- צור ועבור לענף חדש
git checkout main
- עבור לענף הראשי
git branch
- רשימת כל הענפים
git merge branch_name
- מזג ענף לתוך הנוכחי

קונפליקטי מיזוג

⚡ התמודדות עם קונפליקטים
  • מתרחש כששני אנשים משנים את אותה שורה בקובץ
  • Git עוצר את המיזוג ומבקש עזרה ידנית
  • קבצים עם קונפליקטים מסומנים כ-"Both modified"
דוגמת קונפליקט:
<<<<<<< HEAD
טקסט חדש בענף הראשי
=======
טקסט מהענף הצדדי
>>>>>>> my_side_branch

תרגילים מעשיים (Hands-on)

🎯 תרגיל 1: ניהול קבצי דאטה רגישים (.gitignore)

מטרה: לוודא שקבצי CSV כבדים וקובצי סיסמאות לא עולים למאגר בטעות.

  1. צור תיקייה חדשה בשם data-project ואתחל אותה: git init
  2. צור קובץ דאטה מדומה: echo "id,val" > huge_data.csv
  3. צור קובץ קונפיגורציה עם סיסמה: echo "password=123" > secrets.env
  4. צור קובץ .gitignore והוסף לו את השורות: 
    *.csv
*.env
  5. נסה להוסיף את כל הקבצים: git add .
  6. בדוק סטטוס: git status (ודא שרק .gitignore מופיע, והקבצים האחרים לא!)
🎯 תרגיל 2: עבודה עם ענפי פיצ'ר (Feature Branch)

מטרה: לדמות פיתוח ETL חדש מבלי לשבור את הקוד הקיים.

  1. צור קובץ סקריפט ראשי: main_pipeline.py ובצע לו Commit ראשון.
  2. צור ענף חדש לתיקון באג: git checkout -b fix-etl-bug
  3. ערוך את הקובץ, שמור ובצע git add . ו-git commit.
  4. חזור לענף הראשי: git checkout main (השינוי ייעלם מהקובץ).
  5. מזג את התיקון: git merge fix-etl-bug.
🎯 תרגיל 3: פתרון קונפליקט בקובץ דרישות

מטרה: מצב נפוץ שבו שני מפתחים מעדכנים גרסאות ספריות במקביל.

  1. צור קובץ requirements.txt עם התוכן: pandas==1.0.0. בצע commit.
  2. צור ענף update-pandas, שנה בקובץ ל-pandas==1.1.0 ובצע commit.
  3. חזור ל-main. שנה באותו קובץ ל-pandas==1.2.0 (גרסה אחרת!) ובצע commit.
  4. נסה למזג: git merge update-pandas.
  5. Git יצעק שיש קונפליקט. פתח את הקובץ, בחר את הגרסה הרצויה (1.2.0), מחק את הסימונים (<<<<, ====, >>>>), ושמור.
  6. סיים את המיזוג: git add . ואז git commit.
