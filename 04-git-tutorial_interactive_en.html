מבוא ל-Git

מהו ניהול גרסאות (Version Control)?

  • מאחסן ומנהל כל גרסה ושינוי של הקבצים והקוד
  • מאפשר שיתוף פעולה על קוד המקור וניהול גרסאות שחרור
  • מאפשר חזרה (Roll back) לגרסאות קודמות כשמתגלים באגים
  • VSC = מערכת לניהול גרסאות (Version Control System)

אודות Git

  • נוצר על ידי לינוס טורבאלדס ב-2005 (עבור ליבת ה-Linux)
  • מחליף כלים ישנים יותר (CVS, Subversion, Mercurial)
  • מתמקד בהפצת קוד מקור (לכל אחד יש עותק מלא)
  • מיישם אחסון קבצים יעיל במיוחד
  • מבטיח את שלמות הקבצים (Integrity)

יתרונות Git

  • פיתוח מבוזר (עבודה על עותק מקומי אצלך במחשב)
  • ביצועים גבוהים
  • קלות שימוש (לאחר עקומת למידה)
  • תמיכה חזקה בענפים (Branching) וניהול היסטוריה
  • כלים נרחבים לפעולות מתקדמות

אתגרים ב-Git

  • מודל חדש לניהול קוד (דורש שינוי תפיסה)
  • עקומת למידה (הבנת הקונספט ופקודות הטרמינל)
  • פעולות הרסניות אפשריות בשימוש לא נכון
  • תמיכה מוגבלת בקבצים בינאריים גדולים

Commit & Push
  • קבל עותק של המאגר המלא מהשרת ("clone")
  • כל התוכן ו-כל ההיסטוריה אצלך במחשב
  • בצע שינויים מקומית (commits ו-branches)
  • דחוף שינויים חזרה (Push) לשרת המרוחק (Remote)

סוגי אובייקטים ב-Git

  • Blob: התוכן הגולמי של הקובץ – ללא מטא-דאטה
  • Tree: רמה אחת של מידע על תיקייה (שמות קבצים, blobs, תתי-תיקיות)
  • Commit: הגרסה של ה-Tree + מטא-דאטה (מחבר, תאריכים, הודעות יומן וכו')
  • Tag: תווית המשויכת ל-commit מסוים, למשל, מספר גרסה (v1.0)

אחסון דלתא (Delta Storage)

  • כל קובץ נשמר כאובייקט שלם ונפרד כשהוא נוצר לראשונה
  • שינויים לקובץ קיים נשמרים כ"דלתא" (ההבדל בלבד)
  • Git משתמש במנגנון חכם ויעיל לאחסון שינויים
  • התוצאה היא חיסכון משמעותי במקום וניהול היסטוריה יעיל
git init
- אתחול מאגר Git חדש
git clone <repository_url>
- העתקת מאגר Git קיים
git add <file>
- הוספת קבצים ל-Stage (מוכנות ל-Commit)
git commit -m "Commit message"
- שמירת השינויים שנמצאים ב-Stage
git status
- בדיקת סטטוס תיקיית העבודה

תיקיית עבודה (Working Directory)

  • יצירה, עריכה ומחיקה של קבצים בפרויקט שלך
  • חשוב על זה כ"ארגז החול" שבו אתה עובד על הקוד והנתונים

אזור ההערכות (Staging Area / Index)

  • רשימת בדיקה (Checklist) של שינויים שאתה רוצה להכניס ל-Commit הבא
  • הכנה ובחירה של שינויים ספציפיים להיכלל בגרסה הבאה
  • באמצעות git add, אתה מעביר שינויים מתיקיית העבודה ל-Staging

מאגר (Repository)

  • המאגר מאחסן את ההיסטוריה המלאה של הפרויקט
  • כשאתה עושה git commit, השינויים ב-Staging נרשמים כ"תמונת מצב" (Snapshot) חדשה במאגר
  • כל ההיסטוריה של הפרויקט שמורה כאן
