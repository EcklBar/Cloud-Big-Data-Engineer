מבוא ל-Git

מהו ניהול גרסאות (Version Control)?

  • מאחסן ומנהל כל גרסה של הקבצים והקוד
  • מאפשר שיתוף פעולה על קוד המקור וניהול שחרורים (releases)
  • מאפשר חזרה לגרסאות קודמות כאשר מתגלים באגים
  • VSC = מערכת לניהול גרסאות

אודות Git

  • נוצר על ידי לינוס טורבאלדס ב-2005 (עבור ליבת הלינוקס)
  • מחליף כלי ניהול גרסאות ישנים יותר (CVS, Subversion, BitKeeper, Mercurial)
  • מתמקד בהפצת קוד מקור (לכל אחד יש עותק)
  • מיישם אחסון קבצים יעיל
  • מבטיח את שלמות הקבצים (Integrity)

יתרונות Git

  • פיתוח מבוזר (עבודה על העותק המקומי שלך)
  • ביצועים גבוהים
  • קלות שימוש
  • תמיכה חזקה בניהול ענפים (branching) והיסטוריה
  • כלים נרחבים לפעולות מתקדמות

אתגרים ב-Git

  • מודל חדש לניהול קוד
  • עקומת למידה (שינוי חשיבה ופקודות)
  • פעולות הרסניות בשימוש לא נכון
  • תמיכה מוגבלת בקבצים בינאריים
🚀 שירותי VCS

GitHub

  • שירות אירוח מבוסס רשת לניהול גרסאות באמצעות Git
  • בקרת גישה, מעקב אחרי תקלות, תיעוד, אוטומציה
  • שירות ה-VCS המוביל לקוד פתוח וארגונים
  • נרכש על ידי מיקרוסופט ב-2018

סוגי אובייקטים ב-Git

  • Blob: התוכן הגולמי של הקובץ – ללא מטא-דאטה
  • Tree: רמה אחת של מידע תיקייה (שמות קבצים, blobs, תתי-תיקיות)
  • Commit: הגרסה של ה-tree + מטא-דאטה (מחבר, תאריכים, הודעות יומן וכו')
  • Tag: תווית המשויכת ל-commit, למשל, מספר גרסה

אחסון דלתא (Delta Storage)

  • כל קובץ נשמר כאובייקט נפרד ושלם כשהוא מוצג לראשונה
  • שינויים לקובץ קיים נשמרים כ"דלתות" (הפרשים)
  • Git משתמש במנגנון חכם ויעיל לאחסון שינויים
  • התוצאה היא חיסכון משמעותי במקום וניהול היסטוריה יעיל

- שמירת השינויים שנמצאים ב-Stage
git status

תיקיית עבודה (Working Directory)

  • יצירה, עריכה ומחיקה של קבצים בפרויקט שלך
  • חשוב על זה כ"ארגז החול" שבו אתה עובד על הקוד והנתונים שלך

אזור ה-Staging (אינדקס)

  • רשימת בדיקה של שינויים שאתה רוצה לבצע להם commit
  • הכנה ובחירה של שינויים ספציפיים שייכללו ב-commit הבא
  • באמצעות git add, אתה מעביר שינויים מתיקיית העבודה לאזור ה-Staging

מאגר (Repository)

  • המאגר מאחסן את ההיסטוריה המלאה של הפרויקט שלך
  • כשאתה מבצע git commit, השינויים באזור ה-Staging נרשמים כ"תמונת מצב" (snapshot) חדשה במאגר
  • ההיסטוריה המלאה של הפרויקט שלך נשמרת כאן

מצבי קובץ

  • Untracked: קובץ חדש שמופיע לראשונה
  • Unmodified: קובץ קיים ללא שינויים מקומיים
  • Modified: קובץ קיים עם שינויים מקומיים
  • Staged: קבצים עם שינויים מקומיים שסומנו ל-commit הבא
  • Committed: שינויים שנשמרו בבטחה במסד הנתונים המקומי

ענפים (Branches) ב-Git

🌳 אסטרטגיית ענפים
  • התייחס לקוד המקור כעץ של ענפים
  • כל ענף הוא סט ליניארי של commits
  • השתמש בענף צדדי כדי לעבוד על גרסה פרטית של הקוד
  • מזג את הענף הצדדי חזרה לענף הראשי
  • שתף פעולה עם מפתחים אחרים
git checkout branch_name
- עבור לענף
git checkout -B branch_name
- צור ועבור לענף חדש
git branch -D testing
- מחק ענף
git log --oneline --decorate --graph --all
- הצג את כל הענפים
git show branch_name
- הצג את ראש הענף (head)

מיזוג ענפים (Merge Branches)

  • ברגע שמוכן, ענף צדדי ממוזג לתוך הענף הראשי
  • עבור לענף הראשי: git checkout main
  • מזג את הענף הצדדי: git merge side_branch

היסטוריה מתפצלת (Divergent History)

  • שיתוף פעולה באמצעות ענפים מרובים עלול להוביל להיסטוריה מתפצלת
  • מפתח מס' 1 מתחיל ענף חדש ומשנה קובץ
  • מפתח מס' 2 מוסיף commit חדש לענף הראשי וגם משנה את אותו הקובץ
  • מפתח מס' 1 לא יכול למזג מכיוון שהשינויים מתנגשים

התנגשויות מיזוג (Merge Conflicts)

⚡ התמודדות עם התנגשויות
  • כאשר נוצרת התנגשות מיזוג, המיזוג נעצר כדי לאפשר לך לפתור אותה
  • השתמש ב-git status כדי לראות באילו קבצים יש התנגשויות
  • קבצים ללא התנגשויות עוברים ל-stage
  • קבצים עם התנגשויות מופיעים כ-"Both modified"
דוגמה להתנגשות:
<<<<<<< HEAD
טקסט חדש ב-main
=======
טקסט מענף צדדי
>>>>>>> my_side_branch

פתרון התנגשויות באופן ידני

  • פתח את הקובץ שיש בו התנגשויות
  • מחק/שנה שורות לא רלוונטיות
  • העבר את הקובץ ל-stage (פקודת git add)
  • לסיום המיזוג לאחר שכל ההתנגשויות נפתרו: git merge --continue

פתרון בסטייל

  • שימוש ב-IDE מודרני לצפייה ופתרון התנגשויות מיזוג (למשל, VSCode)
  • שימוש בעורך מיזוג (merge editor) הוא הפרקטיקה המומלצת
  • ראה את השינויים בצורה ברורה יותר
  • בחר בקלות את הגרסה המועדפת
  • שלב קוד משני הענפים
  • ממשק אינטואיטיבי וידידותי יותר למשתמש
🔧 דוגמת מיזוג ב-VSCode
  1. פתח את הקובץ עם ההתנגשות
  2. לחץ על "Resolve in Merge Editor"
  3. השווה את השינויים ופתור את ההתנגשות
  4. בחר "Accept Incoming" כדי לבחור שינויים מהענף הצדדי
  5. בחר "Accept Current" כדי לבחור שינויים מהענף הראשי
  6. עדכן ידנית את התוצאה במידת הצורך
  7. ברגע שמוכן, לחץ על "Complete Merge" כדי להעביר את הקובץ ל-stage
  8. הרץ git merge --continue או השתמש ב-VSCode לסיום

פקודות Git חיוניות

ניהול מאגר (Repository Management)

git init
- צור מאגר מקומי חדש
git clone
- הורד עותק של מאגר מרוחק
git status
- הצג את הסטטוס של תיקיית העבודה
git remote -v
- הצג את כל המאגרים המרוחקים כולל ה-URL שלהם.
git remote
- הצג את כל המאגרים המרוחקים
git remote add origin <url>
- הוסף מאגר מרוחק חדש
git remote add new_name <path>
- הוסף שם חדש עבור המאגר המרוחק

Staging & Committing

git add <file>
- העבר קובץ ספציפי ל-stage
git add .
- העבר את כל השינויים בתיקייה הנוכחית ל-stage
git restore --staged <file>
- הסר קובץ מ-stage (זהה ל-reset עבור הקובץ)
git restore --staged
- הסר את כל הקבצים מ-stage
git reset
- בטל העברה ל-stage
git reset --hard
- אפס את תיקיית העבודה ל-commit האחרון
git commit
- צור commit (סט שינויים) חדש במאגר המקומי
git commit -m "message"
- Commit עם הודעה בשורה אחת

Branching & Merging

git branch
- הצג את כל הענפים
git branch <name>
- צור ענף חדש
git branch -d <name>
- מחק ענף מקומי (מונע מחיקה אם יש שינויים שלא מוזגו)
git branch -D <name>
- מחק ענף מקומי בכוח (גם אם יש שינויים שלא מוזגו)
git branch -m old_name new_name
- שנה שם של ענף
git checkout <branch>
- עבור לענף אחר
git checkout -b <name>
- צור ועבור לענף חדש (נכשל אם הענף קיים)
git checkout -B <name>
- צור או אפס ענף קיים ל-HEAD הנוכחי
git merge <branch>
- מזג ענף לתוך הענף הנוכחי
git merge <branch> --ff-only
- מזג רק אם אפשרי fast-forward
git merge --no-ff <branch>
- תמיד צור קומיט מיזוג, גם אם אפשרי fast-forward
git merge --squash <branch>
- שלב את כל הקומיטים מהענף לקומיט אחד ללא יצירת קומיט מיזוג
git merge -s octopus <branch1> <branch2> ...
- מזג מספר ענפים לענף הנוכחי עם קומיט מיזוג יחיד

Cherry-pick

git cherry-pick <commit-hash>
- החל commit מענף אחר
git cherry-pick <hash1> <hash2>
- החל מספר קומיטים
git cherry-pick --continue
- המשך לאחר פתרון התנגשויות
git cherry-pick --abort
- בטל תהליך cherry-pick

היסטוריה ומידע (History & Information)

git log
- הצג את היסטוריית הקומיטים במאגר המקומי
git log --oneline
- היסטוריית קומיטים מקוצרת
git log --oneline --graph --all
- היסטוריית קומיטים מקוצרת עם גרף ויזואלי של כל הענפים
git log --since="YYYY-MM-DD" --until="YYYY-MM-DD"
- הצג היסטוריית קומיטים בטווח תאריכים
git show <commit>
- הצג פרטים של commit ספציפי
git diff
- השווה שינויים (מקומיים, staged, או בין קומיטים)
git diff --staged
- השווה קבצים ב-stage ל-commit האחרון
git blame <file>
- הצג מי שינה כל שורה בקובץ
git tag <tag-name>
- צור תג מקומי לקומיט הנוכחי לסימון גרסה חשובה

ביטול שינויים (Undoing Changes)

git checkout <file>
- שחזר קובץ למצבו האחרון ב-commit
git revert <commit>
- צור commit חדש שמבטל שינויים של commit ספציפי
git revert <commit> --no-edit
- בטל ללא פתיחת עורך טקסט
git revert <commit> -n
- בטל ללא יצירת commit חדש
git reset HEAD <file>
- הוצא קובץ ספציפי מ-stage
git reset --soft HEAD~1
- בטל את ה-commit האחרון אך שמור את השינויים ב-stage
git reset --mixed HEAD~1
- בטל את ה-commit האחרון, שמור שינויים בתיקיית העבודה אך לא ב-stage
git reset --hard HEAD~1
- בטל את ה-commit האחרון והשלך את השינויים

Rebase

git rebase -i master
- Rebase אינטראקטיבי של הענף הנוכחי על גבי master, מאפשר לערוך, לשלב או להסיר קומיטים.
  1. p, pick = השתמש בקומיט כמו שהוא
  2. r, reword = השתמש בקומיט, אך ערוך את הודעת הקומיט
  3. e, edit = השתמש בקומיט, אך עצור לתיקון (amending)
  4. s, squash = שלב את הקומיט לתוך הקודם (שמור את שתי ההודעות)
  5. f, fixup = שלב את הקומיט לתוך הקודם (זרוק את ההודעה של הקומיט הזה)
  6. x, exec = הרץ פקודת shell במהלך ה-rebase
  7. d, drop = הסר את הקומיט לחלוטין

Bisect

git bisect
- התחל חיפוש בינארי למציאת באג
git bisect start
- התחל את סשן ה-bisect
git bisect bad
- סמן את הקומיט הנוכחי כרע (bad)
git bisect good <commit-hash>
- סמן קומיט ידוע כטוב (good)
git bisect run <script_name>
- בצע בדיקה אוטומטית עם סקריפט
git bisect reset
- סיים את ה-bisect וחזור למצב המקורי

Reflog

Worktree

git worktree
- נהל מספר תיקיות עבודה

Git Submodule

git submodule
- נהל תתי-מודולים
git submodule add <repository link | dir> <path>
- הוסף תת-מודול חדש
git submodule list
- הצג תתי-מודולים
git submodule status
- הצג את הסטטוס (commit + מצב סנכרון) של תתי-מודולים
git submodule update --init
- אתחל ומשוך (fetch) תתי-מודולים
git submodule update --init --recursive
- אתחל ועדכן את כל תתי-המודולים המקוננים
git submodule update --init <path_to_submodule>
- אתחל ועדכן תת-מודול ספציפי
git submodule deinit <submodule_name>
- בטל רישום של תת-מודול
git rm <path>
- הסר תת-מודול מהמאגר

תרגילים לתרגול

🎯 תרגיל #1: מבוא ל-Git

השלם את המדריך הבא מתוך הספר הרשמי של git:

יסודות Git - קבלת מאגר Git →
🎯 תרגיל מקומי - חלק #1
  1. אתחל מאגר חדש
  2. צור קובץ בשם "name.txt" עם השם שלך
  3. העבר את הקובץ ל-stage
  4. צור commit חדש
git init

echo "Your Name" > name.txt

git add name.txt

git commit -m "Add name file"
🎯 תרגיל מקומי - חלק #2
  1. צור ענף חדש בשם "side_branch"
  2. שנה את התוכן של name.txt לשם של האדם שלידך
  3. העבר ל-stage ובצע commit לשינויים
  4. צפה בעבודה שלך באמצעות: git log, git status, git show
git checkout -b side_branch

echo "Another Name" > name.txt

git add name.txt

git commit -m "Update name"
🎯 תרגיל מקומי - חלק #3
  1. עבור לענף הראשי (main)
  2. מזג את "side_branch" לתוך הענף הראשי
  3. פתור את ההתנגשות כך ששני השמות יופיעו ב-"name.txt"
  4. לאחר פתרון ההתנגשות, השלם את המיזוג
  5. צפה בעץ ה-git המלא באמצעות git log
git checkout main

git merge side_branch

# Resolve conflicts in name.txt
git add name.txt
