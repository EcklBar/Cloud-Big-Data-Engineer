🎯 מבוא ל-CI/CD

מהו CI/CD?

CI/CD מייצג שני תהליכים מרכזיים בפיתוח תוכנה מודרני:

  • CI - Continuous Integration (אינטגרציה רציפה): תהליך של שילוב שינויי קוד באופן תכוף ואוטומטי
  • CD - Continuous Deployment (פריסה רציפה): תהליך של פריסת שינויים לסביבת הייצור באופן אוטומטי

⚠️ החסרונות של SDLC ההיסטורי

בעיות מרכזיות:

  • תיקוני באגים לוקחים זמן רב - מגיעים רק בגרסאות עתידיות
  • שיפורים מתרחשים בקפיצות - לא בצורה הדרגתית
  • הפקת release ידנית - מופעלת באופן ידני ולא אוטומטי
  • עלות גבוהה - של תיאום, בדיקות והפצה

💡 התוצאה: חוסר גמישות, תגובה איטית לבאגים, וחוויית משתמש לא אופטימלית

📈 התפתחות התוכנה

🔀 מורכבות התוכנה

  • מערכות הפכו מתוחכמות ומורכבות יותר
  • מעבר ל-Microservices - הרבה חלקים נעים חלקית-עצמאיים
  • קודי בסיס (codebase) הפכו מורכבים
  • ניהול גרסאות, תלויות ותאימות הפך קריטי

📊 Git והמהפכה המבוזרת

  • Git הציג דרך סקלבילית וזריזה לפתח תוכנה
  • מודל ה-branching המבוזר מאפשר פיתוח סימולטני
  • מיזוג קוד ל-mainline הפך לאירוע תכוף
  • האתגר: ניהול ותיאום החלקים
  • קוד ה-mainline חייב להישאר יציב - באג בו מחייב החזרה וא restart תהליך הבדיקה

🐳 Containers & Microservices

Containers מספקים דרכים חדשות להפוך את ה-SDLC לחזק:

  • סביבות פיתוח ובדיקה מאוחדות - כולם עובדים באותה סביבה
  • הפקת חבילה (bundling) פשוטה - הכל ארוז ב-container
  • הפחתת חיכוך בפריסה - פחות תלות בסביבת ההרצה
  • Microservices הפכה לפופולרית - פירוק האפליקציה לחלקים עצמאיים רבים עם releases נפרדים אך מתואמים

☸️ Kubernetes (K8s)

מהו Kubernetes?

Kubernetes הוא מערכת לתזמור קונטיינרים (Container Orchestration)

  • חולל מהפכה בסביבות ההרצה המודרניות

🔄 סקירת CI/CD

🔄 Continuous Integration

  • שילוב רציף של חלקי קוד ו-branches שונים
  • בנייה ובדיקה אוטומטית של הקוד
  • זיהוי מוקדם של בעיות
  • שמירה על mainline יציב

🚀 Continuous Deployment

  • גרסאות חדשות ארוזות ונבדקות אוטומטית
  • פרסום תיקונים ושיפורים ברגע שהם מוכנים
  • פריסה לסביבות שונות באוטומציה
  • מעקב וניטור אחר deployments

תהליך CI/CD טיפוסי:

  1. Code - המפתח כותב קוד
  2. Commit - העלאה ל-Git
  3. Build - בנייה אוטומטית
  4. Test - הרצת בדיקות
  5. Package - יצירת artifact (Docker image)
  6. Deploy - פריסה לסביבה
  7. Monitor - ניטור ומעקב

🎯 המטרה: מחזור קצר ככל האפשר מקוד לייצור, עם אוטומציה מקסימלית ואמינות גבוהה

🔧 Jenkins

מהו Jenkins?

  • פלטפורמת אוטומציה לתהליכי CI/CD
  • הוצג ב-2011
  • קוד פתוח (Open-Source)
  • ניתן להתאמה אישית והרחבה רחבה
  • שוק של plugins חינמיים

מאפיינים מרכזיים

⚙️ Job Runner

הגדרת jobs לטסקים שונים

🌳 היררכיה

מבנה jobs היררכי

🔐 הרשאות

ניהול משתמשים וקבוצות

🔄 Triggers

הפעלה אוטומטית (git push, merge)

ארכיטקטורה מבוזרת

  • Master-Slave Architecture
  • אחד או יותר Jenkins masters מנהלים את ה-jobs
  • Jobs רצים על slaves שונים (servers)
  • שיפור ביצועים ומהירות לפרויקטים גדולים
  • סקלביליות של מערכת ה-CI/CD

Jenkinsfile

pipeline { agent any stages { stage('Build') { steps { sh 'make build' } } stage('Test') { steps { sh 'make test' } } stage('Deploy') { steps { sh 'make deploy' } } } }

✅ יתרונות Jenkins

  • קוד פתוח וקהילתי
  • רב-תכליתי
  • ניתן להתאמה אישית רבה
  • ניתן להרחבה
  • חלוץ - לא היה משהו טוב יותר בזמנו

❌ חסרונות Jenkins

  • התקנה ותחזוקה מורכבות
  • מיושן (לא חלק כמו CI/CD מודרני)
  • שפת Groovy (שנאויה)
  • בעיות אבטחה ב-platform וב-plugins
  • over-customizable - כל ארגון ממציא את הגלגל

📝 Infrastructure as Code (IaC)

מהו IaC?

Infrastructure as Code הוא הקונספט של ניהול התשתית דרך קוד

  • מניפסטים מתארים את דרישות ה-provisioning
  • Stack של IaC שומר את המצב הנוכחי של משאבים
  • שינויים במניפסטים מובילים ל-delta של דרישות
  • משאבים מוחלפים אוטומטית כדי לשקף את הדרישות החדשות

טכנולוגיות IaC מובילות

🌍 Terraform

Stateful provisioning של תשתית (cross-)cloud

resource "aws_instance" "web" { ami = "ami-12345678" instance_type = "t2.micro" tags = { Name = "WebServer" } }

⎈ Helm

ניהול (define, install, upgrade) אפליקציות K8s

helm install my-app ./my-chart helm upgrade my-app ./my-chart helm rollback my-app

🔧 Cloud SDKs

AWS CDK, Go CDK - SDKs לענן

  • Templates לשלד הכללי
  • Values/Variables לפרטים ספציפיים

🎬 GitHub Actions

למה GitHub Actions?

  • המקום המוביל לאירוח פרויקטי קוד פתוח
  • מגוון פרויקטים מציע טווח רחב של דוגמאות CI/CD
  • פרויקטים ציבוריים נהנים מ-features מתקדמים בחינם
  • תמיכה נרחבת ב-actions ו-workflows של צד שלישי
  • framework יחסית פשוט וישיר

מושגים מרכזיים

📅 Events

אירועים שמפעילים workflows:

  • Push לענף
  • Pull Request
  • Timer (schedule)
  • ועוד...

📋 Workflows

תהליכים אוטומטיים:

  • מוגדרים בקבצי YAML
  • נמצאים ב-.github/workflows
  • מכילים job אחד או יותר

⚙️ Jobs & Steps

יחידות ביצוע:

  • Job רץ על container ייעודי
  • Steps מבצעים פעולות
  • רצים ברצף או במקביל

מבנה ארכיטקטורה

Workflow ├── Event (trigger) ├── Job 1 │ ├── runs-on: ubuntu-latest │ ├── Step 1: Checkout code │ ├── Step 2: Setup Node │ └── Step 3: Run tests └── Job 2 ├── needs: Job 1 ├── Step 1: Build └── Step 2: Deploy

📋 Workflows ב-GitHub Actions

הגדרת Workflow

  • כל workflow מוגדר בקובץ YAML
  • קבצים נמצאים תחת .github/workflows
  • Workflow חייב להגדיר:
    • איזה events מפעילים אותו
    • Job אחד או יותר

Jobs

  • כל job רץ על container ייעודי בתוך מכונה וירטואלית
  • Job חייב להגדיר:
    • את ה-container image לשימוש
    • Step אחד או יותר לביצוע

Steps

Script Steps

Steps שמריצים script חייבים להגדיר את ה-script

- name: Run tests run: npm test

Action Steps

Steps שמשתמשים ב-actions קיימים חייבים להגדיר:

  • נתיב מלא וגרסה
  • כל הארגומנטים הנדרשים
- uses: actions/checkout@v3 with: ref: main

💡 דוגמה מעשית

Workflow פשוט שרץ על כל push:

name: learn-github-actions on: [push] jobs: check-bats-version: runs-on: ubuntu-latest steps: - uses: actions/checkout@v3 - uses: actions/setup-node@v3 with: node-version: '14' - run: npm install -g bats - run: | echo "testing bats version:" bats -v

הסבר שלב אחר שלב

name: learn-github-actions

שם ה-workflow שיוצג ב-UI

on: [push]

הפעלה על כל push לכל ענף

jobs: check-bats-version

job בודד בשם check-bats-version

runs-on: ubuntu-latest

שימוש ב-container image של Ubuntu

actions/checkout@v3

git checkout של הקוד (ברירת מחדל: main)

actions/setup-node@v3

התקנת Node.js גרסה 14

run: npm install -g bats

הרצת script שמתקין את חבילת bats

run: | (multiline)

הדפסת הודעה והצגת גרסת bats

🔍 צפייה ב-Actions

  1. עבור לטאב Actions ב-repository שלך ב-GitHub
  2. לחץ על workflow בפאנל הצדדי השמאלי
  3. לחץ על הרצת workflow מהרשימה
  4. בחר job מהגרף
  5. בחר step
  6. ראה את ה-console output של ה-step הספציפי

דוגמאות נוספות

CI למהנדסי נתונים - Python

name: Python CI on: push: branches: [ main, develop ] pull_request: branches: [ main ] jobs: test: runs-on: ubuntu-latest steps: - uses: actions/checkout@v3 - name: Set up Python uses: actions/setup-python@v4 with: python-version: '3.9' - name: Install dependencies run: | pip install -r requirements.txt pip install pytest pytest-cov - name: Run tests run: pytest --cov=./ --cov-report=xml - name: Upload coverage uses: codecov/codecov-action@v3

Docker Build & Push

name: Docker Build on: push: branches: [ main ] jobs: build: runs-on: ubuntu-latest steps: - uses: actions/checkout@v3 - name: Login to DockerHub uses: docker/login-action@v2 with: username: ${{ secrets.DOCKERHUB_USERNAME }} password: ${{ secrets.DOCKERHUB_TOKEN }} - name: Build and push uses: docker/build-push-action@v4 with: context: . push: true tags: myusername/myapp:latest
