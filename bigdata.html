🐳 מהו Docker?

Docker הוא המימוש הפופולרי ביותר של טכנולוגיית Containers

הגדרה

  • פלטפורמה למפתחים ומנהלי מערכות
  • בנייה, שיתוף והרצה של אפליקציות בצורה ניידת
  • מנגנון בידוד (Isolation)
  • גמיש, קל משקל, מאובטח וסקלבילי
  • תהליך מפואר (OS Process + Namespaces)

🔬 הטכנולוגיה מאחורי Docker

🔄 Processes
קונטיינרים הם תהליכי מערכת הפעלה רגילים (OS Processes)
🔐 Namespaces
סביבות מבודדות (PID, Mount, Network, UTS, IPC, User). כל קונטיינר רואה רק את התהליכים והמשאבים שלו
⚙️ Control Groups (cgroups)
הגבלת משאבים ושליטה (RAM, CPU, Disk I/O). מונעים מקונטיינר אחד לצרוך את כל משאבי המערכת
📁 Overlay FileSystem
שימוש בשכבות filesystem לבניית סביבת הקונטיינר. כל שכבה מוסיפה או משנה שכבות קודמות

💡 למה זה חשוב?

הבנת הטכנולוגיה הבסיסית עוזרת לפתור בעיות, לייעל ביצועים ולהבין את המגבלות של Docker.

📚 הגדרות Docker

Docker Image
תבנית לקריאה בלבד (read-only) המכילה את כל הקבצים, הספריות והתלויות הנדרשות להרצת אפליקציה בקונטיינר Docker.
Docker Container
מופע פועל (running instance) של image. כלומר, חבילה הניתנת להרצה הכוללת את הקוד, הספריות והתלויות הנדרשות להרצת האפליקציה.
Dockerfile
סקריפט המכיל הוראות ליצירת image. קובץ טקסט עם פקודות המגדירות כיצד לבנות את ה-image (מעין מתכון).
Docker Registry
מאגר לאחסון, שיתוף וניהול Docker images. כולל Docker Hub, Amazon ECR, Azure Container Registry ו-Google Cloud Container Registry.
Docker Engine
אפליקציה לניהול Docker containers. כוללת שרת (Docker daemon), כלי שורת פקודה (Docker client) ו-API לאינטראקציה עם ה-daemon.
Docker Client
כלי שורת פקודה לאינטראקציה עם Docker Engine לניהול images וקונטיינרים.
Docker Daemon
תהליך רקע (background process) המנהל Docker images וקונטיינרים בהתאם לפקודות שנשלחות מה-Docker client.
Docker Compose
כלי להגדרה והרצה של אפליקציות Docker מרובות קונטיינרים באמצעות קובץ YAML לתצורת השירותים.

📝 Dockerfile - מושגים ופקודות

מהו Dockerfile?

Dockerfile הוא קובץ טקסט המכיל הוראות לבניית Docker image

זהו למעשה "מתכון" שמגדיר מה יהיה בתוך ה-image שלנו

פקודות Dockerfile עיקריות

FROM - בסיס ה-Image
FROM ubuntu:22.04

מגדיר את ה-image הבסיסי שעליו נבנה

ARG - משתנה זמן בנייה
ARG VERSION=1.0

מגדיר משתנה שזמין רק בזמן הבנייה

ENV - משתנה סביבה
ENV APP_HOME=/app

מגדיר משתנה סביבה זמין בזמן בנייה והרצה

WORKDIR - תיקיית עבודה
WORKDIR /app

מגדיר את תיקיית העבודה בתוך ה-container

RUN - הרצת פקודה
RUN apt-get update && apt-get install -y python3

מריץ פקודת shell בזמן הבנייה

COPY - העתקת קבצים
COPY app.py /app/

מעתיק קבצים מהמחשב המקומי ל-image

ADD - הוספת קבצים מתקדמת
ADD archive.tar.gz /app/

כמו COPY אבל תומך בחילוץ ארכיונים והורדה מ-URL

EXPOSE - חשיפת פורט
EXPOSE 8080

מודיע שה-container מאזין על פורט מסוים (רמז בלבד!)

VOLUME - יצירת נקודת עיגון
VOLUME ["/data"]

יוצר volume אנונימי לשמירת נתונים

CMD - פקודת ברירת מחדל
CMD ["python3", "app.py"]

הפקודה שתרוץ כשה-container יתחיל

ENTRYPOINT - נקודת כניסה
ENTRYPOINT ["python3", "app.py"]

כמו CMD אבל לא ניתן לדרוס

USER - שינוי משתמש
USER appuser

מחליף למשתמש אחר (עדיף לא root)

דוגמת Dockerfile מלאה

FROM python:3.9-slim # משתני סביבה ENV PYTHONUNBUFFERED=1 \ APP_HOME=/app # תיקיית עבודה WORKDIR ${APP_HOME} # התקנת תלויות COPY requirements.txt . RUN pip install --no-cache-dir -r requirements.txt # העתקת הקוד COPY . . # משתמש לא-root RUN useradd -m appuser && chown -R appuser:appuser ${APP_HOME} USER appuser # חשיפת פורט EXPOSE 8000 # פקודת הפעלה CMD ["python", "app.py"]

🔨 בניית Docker Images

פקודת docker build

בנה image מהתיקייה הנוכחית
docker build .
בנה image עם שם ותג
docker build -t myapp:v1.0 .
בנה מקובץ Dockerfile ספציפי
docker build -f Dockerfile.prod -t myapp:prod .
בנה ללא cache
docker build --no-cache -t myapp:latest .
העבר ARG לזמן בנייה
docker build --build-arg VERSION=2.0 -t myapp:v2 .
בנה עם target ספציפי (multi-stage)
docker build --target production -t myapp:prod .

דגלים חשובים

  • -t, --tag - שם ותג ל-image
  • -f, --file - נתיב ל-Dockerfile
  • --no-cache - בנה מחדש ללא שימוש ב-cache
  • --build-arg - העבר משתני ARG
  • --target - בחר שלב ב-multi-stage build

🗂️ מערכת Layers של Docker

Docker בנוי משכבות (layers) של filesystem

  • כל הוראה ב-Dockerfile יוצרת שכבה חדשה
  • שכבות הן read-only
  • רק השכבה העליונה של הקונטיינר היא writable
  • שכבות משותפות בין images חוסכות מקום

Overlay FileSystem

Layer 2: COPY requirements.txt

⚠️ טיפ חשוב: כל הוראה RUN יוצרת שכבה חדשה. לכן מומלץ לשלב פקודות:

❌ לא טוב: RUN apt-get update RUN apt-get install -y python3 RUN apt-get clean ✅ טוב יותר: RUN apt-get update && \ apt-get install -y python3 && \ apt-get clean && \ rm -rf /var/lib/apt/lists/*

🎭 CMD vs ENTRYPOINT

CMD

  • ניתן לדרוס בזמן docker run
  • מספק ברירת מחדל לפקודה
  • נועד לפקודות שעשויות להשתנות
FROM ubuntu CMD ["echo", "Hello"] # להרצה: docker run myimage # פלט: Hello docker run myimage echo "Bye" # פלט: Bye (CMD דרוס!)

ENTRYPOINT

  • לא ניתן לדרוס בקלות
  • מגדיר את התפקיד העיקרי של הקונטיינר
  • נועד לקונטיינרים עם מטרה קבועה
FROM ubuntu ENTRYPOINT ["echo"] CMD ["Hello"] # להרצה: docker run myimage # פלט: Hello docker run myimage "Bye" # פלט: Bye (CMD דרוס, ENTRYPOINT נשאר!)

שילוב CMD + ENTRYPOINT

💡 שימוש מומלץ: השתמש ב-ENTRYPOINT לפקודה הקבועה וב-CMD לפרמטרים של ברירת מחדל

FROM python:3.9 ENTRYPOINT ["python"] CMD ["app.py"] # docker run myimage → מריץ python app.py # docker run myimage test.py → מריץ python test.py

תחביר Shell vs Exec

Shell Form
CMD echo "Hello World"

מופעל דרך /bin/sh -c

Exec Form (מומלץ)
CMD ["echo", "Hello World"]

מופעל ישירות, ללא shell

📄 COPY vs ADD

COPY

  • פשוט וברור
  • מעתיק קבצים/תיקיות מקומיים בלבד
  • מומלץ לשימוש רגיל
  • תמיכה ב-wildcards
# העתק קובץ בודד COPY app.py /app/ # העתק תיקייה COPY src/ /app/src/ # wildcard COPY *.py /app/

ADD

  • מתקדם יותר
  • תומך בכל מה ש-COPY תומך
  • מחלץ ארכיונים אוטומטית
  • יכול להוריד מ-URLs
# חילוץ אוטומטי ADD archive.tar.gz /app/ # הורדה מ-URL ADD https://example.com/file.zip /tmp/ # כמו COPY ADD app.py /app/

מתי להשתמש במה?

✅ השתמש ב-COPY:

  • ברוב המקרים (ברירת המחדל)
  • כשאתה רק מעתיק קבצים מקומיים
  • כשאתה רוצה שהכוונה תהיה ברורה

⚠️ השתמש ב-ADD רק כאשר:

  • אתה צריך לחלץ ארכיון tar/zip
  • אתה צריך להוריד קובץ מ-URL (לא מומלץ בפרודקשן!)

💡 Best Practice: השתמש ב-COPY אלא אם כן אתה צריך באופן ספציפי את היכולות המתקדמות של ADD

🎯 Signal Handling & PID 1 Problem

מהי בעיית PID 1?

⚠️ בעיה קריטית: התהליך הראשון בקונטיינר (PID 1) מקבל טיפול מיוחד מה-kernel של Linux. אם הוא לא מטפל נכון ב-signals, הקונטיינר לא יכבה בצורה נכונה!

Shell Form vs Exec Form

❌ Shell Form (בעיה)

CMD python app.py # או ENTRYPOINT python app.py

מה קורה:

  • Docker מריץ: /bin/sh -c "python app.py"
  • /bin/sh הוא PID 1 (לא Python!)
  • ה-shell לא מעביר signals לתהליך הבן
  • docker stop ימתין 10 שניות ואז יהרוג בכוח (SIGKILL)
  • ❌ Graceful shutdown לא עובד

✅ Exec Form (נכון)

CMD ["python", "app.py"] # או ENTRYPOINT ["python", "app.py"]

מה קורה:

  • Python רץ ישירות כ-PID 1
  • מקבל SIGTERM ישירות מ-Docker
  • יכול לסגור connections, לשמור מצב, וכו'
  • ✅ Graceful shutdown עובד!

בדיקה: מי PID 1?

בדוק מי PID 1 בקונטיינר שלך
docker exec [container] ps aux

השורה הראשונה תראה מי PID 1

פתרון למצבים מיוחדים: tini

💡 כאשר אתה צריך shell script או entrypoint מורכב:

השתמש ב-tini - init system זעיר שמטפל נכון ב-signals

FROM python:3.9-slim # התקן tini RUN apt-get update && \ apt-get install -y tini && \ rm -rf /var/lib/apt/lists/* # הגדר tini כ-entrypoint ENTRYPOINT ["/usr/bin/tini", "--"] # עכשיו אפשר להשתמש ב-shell script CMD ["./start.sh"]

דוגמה מעשית: Flask App

❌ לא יעבוד טוב

FROM python:3.9 COPY app.py . CMD python app.py # docker stop ייקח 10 שניות!

✅ עובד מצוין

FROM python:3.9 COPY app.py . CMD ["python", "app.py"] # docker stop יהיה מיידי

🎯 כלל אצבע:

  • תמיד השתמש ב-Exec Form (JSON array) למעט אם אתה באמת צריך shell features (pipes, variables, etc.)
  • אם אתה צריך shell, הוסף tini או dumb-init
  • בדוק ש-docker stop לא לוקח 10+ שניות

📄 .dockerignore - חובה!

מהו .dockerignore?

🔀 ARG vs ENV - מתי להשתמש במה?

ARG - זמן בנייה בלבד

ARG PYTHON_VERSION=3.9 FROM python:${PYTHON_VERSION} ARG BUILD_DATE RUN echo "Built on ${BUILD_DATE}"
מאפיינים:
  • ✅ זמין רק בזמן docker build
  • ❌ לא זמין בקונטיינר רץ
  • ✅ ניתן לדרוס: --build-arg
  • ❌ לא מופיע ב-docker inspect
שימושים:
  • גרסאות base images
  • build flags
  • paths זמניים

ENV - בנייה + runtime

ENV PORT=8000 ENV NODE_ENV=production ENV DATABASE_URL=postgres://...
מאפיינים:
  • ✅ זמין בזמן build
  • ✅ זמין בקונטיינר רץ
  • ✅ ניתן לדרוס: docker run -e
  • ✅ מופיע ב-docker inspect
שימושים:
  • הגדרות אפליקציה
  • paths קבועים
  • feature flags

שילוב ARG + ENV

# קבל ערך בזמן build ARG APP_VERSION=1.0.0 # שמור אותו כ-ENV לזמן runtime ENV VERSION=${APP_VERSION} # Build: docker build --build-arg APP_VERSION=2.0.0 -t myapp . # Runtime - הגרסה תהיה זמינה: docker run myapp printenv VERSION # Output: 2.0.0

השוואה מעשית

דוגמה: ARG לבחירת base image
ARG PYTHON_VERSION=3.9 FROM python:${PYTHON_VERSION}-slim # Build different versions: # docker build --build-arg PYTHON_VERSION=3.10 -t app:py310 . # docker build --build-arg PYTHON_VERSION=3.11 -t app:py311 .
דוגמה: ENV להגדרות אפליקציה
ENV FLASK_APP=app.py ENV FLASK_ENV=production # Override בזמן run: # docker run -e FLASK_ENV=development myapp

⚠️ אזהרת אבטחה קריטית!

גם ARG וגם ENV נשארים בהיסטוריה של ה-image! אל תשים בהם סודות!

❌ אל תעשה: ARG DB_PASSWORD=secret123 ENV API_KEY=supersecret # כל אחד יכול לראות: docker history myimage

BuildKit Secrets - הפתרון המודרני

✅ השתמש ב-BuildKit secrets עבור מידע רגיש:

# Dockerfile RUN --mount=type=secret,id=db_password \ DB_PASS=$(cat /run/secrets/db_password) \ ./install-with-secret.sh # Build command export DOCKER_BUILDKIT=1 docker build \ --secret id=db_password,src=./secrets/db.txt \ -t myapp . # הסוד לא נשאר ב-image!

💡 יתרונות BuildKit Secrets:

  • ✅ הסוד לא נשמר ב-image layers
  • ✅ לא מופיע ב-docker history
  • ✅ נמחק אחרי הבנייה
  • ✅ לא נגיש מחוץ ל-RUN שמשתמש בו

📦 Docker Registry

מהו Registry?

Docker Registry הוא מערכת אחסון והפצה עבור Docker images

  • Docker Hub - Registry הציבורי של Docker (ברירת מחדל)
  • Private Registries - AWS ECR, Google GCR, Azure ACR
  • Self-hosted - Harbor, Registry

עבודה עם Registry

התחבר ל-Docker Hub
docker login
התחבר ל-registry פרטי
docker login registry.example.com
משוך image
docker pull nginx:latest
תייג image למאגר
docker tag myapp:v1 username/myapp:v1
דחוף image
docker push username/myapp:v1
חפש images
docker search nginx
התנתק
docker logout

תיוג Images

# פורמט כללי [registry]/[username]/[image]:[tag] # דוגמאות: nginx:latest # Docker Hub docker.io/library/nginx:1.21 # Docker Hub מפורש gcr.io/my-project/myapp:v1 # Google Container Registry 123456789.dkr.ecr.us-east-1.amazonaws.com/myapp:latest # AWS ECR

🎼 Docker Compose

  • קובץ YAML אחד מגדיר את כל השירותים
  • הפעלה והורדה בפקודה אחת
  • אידיאלי לסביבות פיתוח

✨ Best Practices ל-Docker

🔒 אל תשתמש ב-root
RUN useradd -m myuser USER myuser
📦 שמור על images קטנים
FROM python:3.9-slim RUN apt-get update && \ apt-get install -y --no-install-recommends gcc && \ pip install --no-cache-dir -r requirements.txt && \ apt-get purge -y gcc && \ apt-get autoremove -y && \ rm -rf /var/lib/apt/lists/*
🎯 השתמש ב-multi-stage builds
FROM node:14 AS builder WORKDIR /app COPY package*.json ./ RUN npm install COPY . . RUN npm run build FROM nginx:alpine COPY --from=builder /app/dist /usr/share/nginx/html
📄 השתמש ב-.dockerignore
# .dockerignore node_modules .git *.md .env .vscode __pycache__ *.pyc
🏷️ תייג images בצורה משמעותית
# לא טוב myapp:latest # טוב myapp:1.2.3 myapp:1.2.3-alpine myapp:v1.2.3-2024-01-15
📊 סדר הוראות לפי תדירות שינוי
FROM python:3.9 WORKDIR /app # משתנה לעיתים רחוקות COPY requirements.txt . RUN pip install -r requirements.txt # משתנה לעיתים קרובות COPY . .
🔐 אל תשמור סודות
# ❌ לא לעשות ENV DB_PASSWORD=secret123 # ✅ עדיף # העבר דרך -e בזמן הרצה או השתמש ב-secrets
💊 הגדר health checks
HEALTHCHECK --interval=30s --timeout=3s \ CMD curl -f http://localhost/ || exit 1
🔍 השתמש ב-LABEL למטאדטה
LABEL version="1.0" \ maintainer="team@example.com" \ description="My Application"
🧹 נקה קבצים זמניים
RUN apt-get update && \ apt-get install -y package && \ rm -rf /var/lib/apt/lists/* /tmp/* /var/tmp/*
⚡ השתמש ב-alpine images
# במקום: FROM python:3.9 # ~900MB # השתמש ב: FROM python:3.9-alpine # ~50MB

דוגמה: Dockerfile אופטימלי

# ============================================ # Stage 1: Build # ============================================ FROM python:3.9-slim AS builder WORKDIR /app # התקן תלויות בנייה RUN apt-get update && \ apt-get install -y --no-install-recommends gcc && \ rm -rf /var/lib/apt/lists/* # העתק והתקן dependencies COPY requirements.txt . RUN pip install --user --no-cache-dir -r requirements.txt # ============================================ # Stage 2: Runtime # ============================================ FROM python:3.9-slim # הגדר משתני סביבה ENV PYTHONUNBUFFERED=1 \ PYTHONDONTWRITEBYTECODE=1 \ PATH=/home/appuser/.local/bin:$PATH WORKDIR /app # צור משתמש לא-privileged RUN useradd -m -u 1000 appuser && \ chown -R appuser:appuser /app # העתק dependencies מה-builder stage COPY --from=builder --chown=appuser:appuser /root/.local /home/appuser/.local # העתק הקוד COPY --chown=appuser:appuser . . # החלף למשתמש USER appuser # Health check HEALTHCHECK --interval=30s --timeout=3s --retries=3 \ CMD python -c "import requests; requests.get('http://localhost:8000/health')" # חשוף פורט EXPOSE 8000 # הגדר metadata LABEL version="1.0.0" \ maintainer="dev@example.com" \ description="Production-ready Python app" # הפעל את האפליקציה CMD ["python", "app.py"]

בעיות נפוצות ופתרונות

❌ "Cannot connect to Docker daemon"
פתרון:
# הפעל את Docker daemon sudo systemctl start docker # בדוק סטטוס sudo systemctl status docker # הוסף משתמש לקבוצה sudo usermod -aG docker $USER
❌ "Port already in use"
פתרון:
# מצא תהליך שתופס את הפורט sudo lsof -i :8080 # או שנה את הפורט docker run -p 8081:80 nginx
❌ "No space left on device"
פתרון:
# נקה הכל docker system prune -a --volumes # בדוק שימוש docker system df
❌ Container מסתיים מיד
פתרון:
# בדוק לוגים docker logs [container] # בדוק exit code docker inspect [container] --format='{{.State.ExitCode}}' # הרץ interactively לדיבאג docker run -it [image] /bin/bash
