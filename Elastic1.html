What is Elasticsearch?

Elasticsearch is a distributed, free and open search and analytics engine for all types of data, including textual, numerical, geospatial, structured, and unstructured.

Why is it so fast?
It is built on top of Apache Lucene and uses a structure called an Inverted Index, which is designed for very fast full-text searches.

Key Features

  • Distributed: Scales horizontally to handle petabytes of data.
  • RESTful API: Accessed via JSON over HTTP.
  • Near Real-Time (NRT): Slight latency (usually 1 second) from the time you index a document until it becomes searchable.

The ELK Stack

"ELK" is the acronym for three open source projects: Elasticsearch, Logstash, and Kibana.

1. Beats (Collection)

Metaphor: Think of Beats as the "Delivery Trucks" picking up packages.

Beats are lightweight data shippers that you install as agents on your servers to capture all sorts of operational data (think of them as "Agents").

They are individual packages written in Go, focused on single types of data (e.g., Filebeat for logs, Metricbeat for metrics).

Source:

2. Logstash (Ingest)

Metaphor: Think of Logstash as the "Sorting & Packaging Center".

Logstash is a server-side data processing pipeline that ingests data from a multitude of sources, transforms it, and then sends it to your "stash" (Elasticsearch).

It can derive structure from unstructured data using grok filters, anonymize sensitive fields, and aggregate data.

Source:

3. Elasticsearch (Store)

Metaphor: Think of Elasticsearch as the "Warehouse" with an ultra-fast filing system.

The heart of the stack. It is a distributed, RESTful search and analytics engine. It stores data as JSON documents and uses an inverted index to allow for near real-time search.

Data is split into shards (horizontal partitions) which are distributed across nodes for scalability and high availability.

Source:

4. Kibana (Visualize)

Metaphor: Think of Kibana as the "Dashboard" in the control room.

Kibana is the visualization layer. It works on top of Elasticsearch to let users visualize data through charts, graphs, maps, and dashboards.

It is the window into the Elastic Stack, allowing you to search, view, and interact with your data stored in Elasticsearch indices.

Source:

Core Terminology

Cluster & Node

A Cluster is a collection of one or more servers (nodes) that together hold your entire data. A Node is a single server that is part of your cluster.

Index

An index is a collection of documents that have somewhat similar characteristics. It is identified by a name (all lowercase) and is similar to a Database in RDBMS.

Document

A document is a basic unit of information that can be indexed. It is expressed in JSON format. Similar to a Row in a table.

Analogy:
Relational DB ⇒ Databases ⇒ Tables ⇒ Rows ⇒ Columns
Elasticsearch ⇒ Indices ⇒ Types (Deprecated) ⇒ Documents ⇒ Fields

Architecture: Sharding & Replication

An index can store a large amount of data that can exceed the hardware limits of a single node. To solve this, Elasticsearch allows you to subdivide your index into multiple pieces called shards.

Shards

  • Allows you to horizontally split/scale your content volume.
  • Allows you to distribute and parallelize operations across shards (performance).

Replicas

To ensure high availability, Elasticsearch allows you to make one or more copies of your index’s shards into replica shards.

  • Provides high availability in case a node/shard fails.
  • Allows you to scale out your search volume/throughput (searches can be executed on replicas in parallel).

The Inverted Index

This is the magic behind the speed. Instead of storing strings directly, Elasticsearch lists every unique word that appears in any document and identifies all of the documents each word occurs in.

Example

Doc 1: "The quick brown fox"
Doc 2: "The quick blue rabbit"

TermDoc IDs
quick[1, 2]
brown[1]
blue[2]
fox[1]

Elasticsearch REST API

Interact with your cluster using simple JSON over HTTP.

GET /{index}/_doc/{id}

Retrieve a JSON document from the index.

GET /customer/_doc/1
PUT /{index}/_doc/{id}

Index a new document (or update) with a specific ID.

PUT /customer/_doc/1 { "name": "John Doe" }
POST /{index}/_search

Run a complex search query using the Query DSL.

POST /customer/_search { "query": { "match": { "name": "John" } } }
DELETE /{index}

Delete an entire index and all its documents.

DELETE /customer

03 Multi-match Search

Search for documents holding the word "guide" examining all fields.

GET /bookdb_index/book/_search { "query": { "multi_match" : { "query" : "guide" } } }
04 Specific Field Search

Search the phrase "action" in the title field.

POST /bookdb_index/book/_search { "query": { "match" : { "title" : "in action" } } }
05 Boosting Relevance

Add a boost of 3 when searching the phrase "search analytics" in all documents.

POST /bookdb_index/book/_search { "query": { "multi_match" : { "query" : "search analytics", "boost" : 3 } } }
06 Boolean Logic

Find books with "Elasticsearch" OR "Solr" in the title, AND authored by "Clinton Gormley", but NOT "Radu Gheorge".

POST /bookdb_index/book/_search { "query": { "bool": { "must": { "bool" : { "should": [ { "match": { "title": "Elasticsearch" }}, { "match": { "title": "Solr" }} ], "must": { "match": { "authors": "clinton gormely" }} } }, "must_not": { "match": {"authors": "radu gheorge" }} } } }
07 Wildcard & Highlight

Find records where author name starts with 't' and highlight the author field.

POST /bookdb_index/book/_search { "query": { "wildcard" : { "authors" : "t*" } }, "highlight": { "fields" : { "authors" : {} } } }
08 Regexp Search

Find authors starting with 't' and ending with 'y'.

POST /bookdb_index/book/_search { "query": { "regexp" : { "authors" : "t[a-z]*y" } } }
09 Phrase Prefix

Find documents with the phrase "search en" in the summary field.

POST /bookdb_index/book/_search { "query": { "match_phrase_prefix" : { "summary": { "query": "search en" } } } }
10 Range Query

Search for books published in 2015 only.

POST /bookdb_index/book/_search { "query": { "range" : { "publish_date": { "gte": "2015-01-01", "lte": "2015-12-31" } } } }
