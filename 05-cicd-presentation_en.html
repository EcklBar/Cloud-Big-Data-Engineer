Historical SDLC

Understanding where we came from helps us appreciate modern practices.

Traditional Software Development Characteristics:

  • Software development managed using central version control systems
  • Every release goes through long cycles (development, test, fix, repeat)
  • Bundling a release is not a frequent event
  • Software released in versioned bundles: packages, disks, etc.
  • Customers actively update their version of the software

The Drawbacks:

  • Bug fixes take long times to arrive (only in following releases)
  • Improvements occur in non-gradual steps
  • Bundling a release is triggered manually

Software Evolution

Significant evolutions in software in recent years have transformed the SDLC.

Key Evolution Factors:

Software Complexity

  • Systems become more sophisticated and complicated
  • Shifting toward microservices – a lot of partially-independent moving parts
  • Code bases become complex
  • Managing versions, dependencies and compatibility becomes crucial

Code Complexity & Git

  • Git introduces a scalable and agile way to develop software
  • The distributed branching model enables simultaneous development
  • Merging code to the mainline becomes a frequent event
  • The challenge becomes managing and coordinating the pieces
  • Mainline code must be stable: a bug in the mainline forces rewinding the code and restarting the testing process

The Modern CI/CD Revolution

R&D Organizations Adapt

  • R&D organization adjust to the changes in software characteristics
  • Increasing the number of individuals and teams involved in a particular software
  • New, agility-driven methodologies (e.g., scrum) gain popularity
    • Focus on value and flexibility
    • Rely on short end-to-end cycles (development, testing, releasing)

Containers & Microservices

  • Containers provide new ways to make the SDLC robust:
    • Consolidated development and testing environments
    • Bundling becomes straight-forward
    • Deployment friction reduced (less dependency on the running environment)
  • Microservices becomes a popular approach: breaking the application into lots of independent parts with distinct but coordinated releases

The Cloud Era

  • The cloud revolution introduced the "as a service" model
    • IaaS, PaaS: Easily provision and manage the infrastructure and platform
  • SaaS
    • Software as a services eliminates versioning and releasing
    • Software being continuously updated and fixed
    • Customers expect proactive updates without taking actions

Jenkins - The Automation Pioneer

🎩

What is Jenkins?

  • An automation platform
  • Introduced in 2011
  • Open-Source
  • Highly customizable and extensible
    • A marketplace of free plugins to extend functionality
    • Almost anything in the platform can be tweaked

Jenkins as a Job Runner

  • Define different jobs for different tasks
  • Structure jobs in hierarchical groups
  • Assign user groups and permissions to jobs
  • Trigger jobs automatically upon events (git push, merge)
  • Track and inspect job runs

Distributed Builds

  • A master-slave architecture
    • One or more Jenkins masters manage the jobs
    • Jobs run on different slaves (servers)
  • Improve performance and speed for large projects/tasks
  • Enable scalability of the CI/CD system

✅ Pros

  • Open-source & community-driven
  • Multi-purpose
  • Highly customizable
  • Pioneer in the field

❌ Cons

  • Complex setup & maintenance
  • Obsolete design
  • Groovy language
  • Security issues

Cloud Deployment & Infrastructure as Code

Kubernetes (K8s)

  • Containers orchestration: run a cluster of containers
  • Released by Google at 2014 as an open source project
  • Revolutionized modern running environment:
    • Every running environment is a K8s cluster
    • Built to scale and support dynamically-provisioned resources
    • Every cloud vendor provides a managed K8s solution off-the-shelf

Cloud Deployment Process

  • Deploying to a modern cloud environment consists of:
    • Provisioning the needed resources (nodes, network, storage, etc.)
    • Preparing updated versions of the K8s manifests
    • Updating the cluster to run the new versions
  • Infrastructure as Code introduced to handle the new infrastructure complexity

Infrastructure as Code (IaC)

  • Infrastructure as code (IaC) is the concept of managing the infrastructure via code
  • Manifests describe the provisioning requirements
  • A stack of IaC saves the state of currently deployed resources
  • Publishing changes to the IaC manifests results in a requirements delta
  • Resources are automatically replaced to reflect the new requirements

GitHub Actions

Why GitHub Actions?

  • The leading hosting for open-source projects
  • The variety of projects offer a wide range of CI/CD examples
  • Public projects enjoy advanced enterprise features for free
  • Extensive support for using 3rd-party actions and workflows
  • A relatively simple and straightforward framework

How GitHub Actions Works

  • Configure a workflow to be triggered when an event occurs in your repository
  • Workflow contains one or more jobs (executed sequentially or simultaneously)
  • Jobs execute steps sequentially, each step either:
    • Runs a script
    • Invokes an action (a reusable extension)

Events that Trigger Workflows

  • Pushing code to a branch
  • Pushing a new tag
  • Opening/updating/closing a pull request
  • Timer (schedule routine workflows)

Example Workflow

name: learn-github-actions on: [push] jobs: check-bats-version: runs-on: ubuntu-latest steps: - uses: actions/checkout@v3 - uses: actions/setup-node@v3 with: node-version: '14' - run: npm install -g bats - run: | echo "testing bats version:" bats -v

Workflow Components

  • Each workflow is defined using a yaml file
  • Place workflow yamls under the .github/workflows directory in your repository
  • A workflow must define:
    • What events trigger the workflow
    • One or more jobs
  • Each job runs on a dedicated container inside a virtual machine
  • A job must define:
    • The container image to use
    • One more or steps to execute
